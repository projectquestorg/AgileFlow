# /deuda

Realice un seguimiento y visualice la deuda técnica en todo el código base. Identifique, categorice y priorice el trabajo de refactorización con puntuación basada en datos.

## Inicio rápido

```bash
/agileflow:debt
```

Ejecuta un análisis rápido y genera un informe de deuda con la configuración predeterminada.

O con opciones personalizadas:

```bash
/agileflow:debt SCAN=full OUTPUT=report THRESHOLD=medium
```

## Parámetros

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `SCAN` | No | `quick` | Scan depth: `quick` (flagged areas only) or `full` (entire codebase) |
| `OUTPUT` | No | `report` | Output type: `report`, `stories`, or `both` |
| `THRESHOLD` | No | `medium` | Minimum severity: `high`, `medium`, or `low` |

## Lo que detecta

El escáner de deuda identifica múltiples tipos de deuda técnica:

### Señales de calidad del código
- TODO/FIXME/HACK comentarios
- Funciones largas (>50 líneas)
- Alta complejidad ciclomática (>10)
- Bloques de código duplicados
- Archivos grandes (>500 líneas)
- Anidamiento profundo (>4 niveles)
- Dios se opone (>10 métodos)
- Cobertura de prueba baja (menos del 70%)

### Deuda basada en historias
Busca historias etiquetadas con `tech-debt: true` o con "refactor" o "limpieza" en el título.

### Deuda de arquitectura
Comprueba las ADR en busca de decisiones obsoletas o consecuencias negativas documentadas.

### Análisis del historial de Git
- Archivos con alta tasa de abandono (modificados con frecuencia)
- Archivos con muchos autores (propiedad poco clara)
- Ramas antiguas aún no fusionadas

### Salida de línea
Analiza la salida de ESLint, TypeScript, Pylint, SonarQube y los informes de cobertura.

## Categorías de deuda

La deuda técnica se organiza en siete categorías:

| Category | Description | Examples |
|----------|-------------|----------|
| **Architecture** | Wrong abstractions, tight coupling, missing patterns | Service coupling, circular dependencies |
| **Code Quality** | Duplication, complexity, inconsistent style | Copy-paste code, high complexity |
| **Documentation** | Missing, outdated, or incomplete docs | Undocumented APIs, stale guides |
| **Testing** | Low coverage, flaky tests, missing E2E | No unit tests, missing integration tests |
| **Security** | Outdated deps, exposed secrets, weak auth | CVE-vulnerable packages, hardcoded tokens |
| **Performance** | N+1 queries, memory leaks, slow ops | Inefficient algorithms, memory bloat |
| **Dependencies** | Outdated packages, unmaintained libraries | Years-old libraries, abandoned projects |

## Fórmula de puntuación

Cada elemento de deuda recibe una puntuación del 0 al 100:

```text
Score = (Severity × Scope × Pain) / 10

Severity: 1-10 (impact level)
Scope:    1-10 (files affected)
Pain:     1-10 (developer frustration)
```

### Niveles de gravedad
- **Low** (1-3): Problema menor, no bloquea nada
- **Medium** (4-6): Problema moderado, ralentiza el desarrollo
- **High** (7-9): Problema importante, bloquea funciones o causa errores
- **Critical** (10): Urgente, causa problemas de producción o bloquea el trabajo.

## Ejemplos

### Escaneo rápido (predeterminado)

```bash
/agileflow:debt
```

Escanea áreas marcadas con TODO/FIXME e historias etiquetadas. Genera informe.

### Escaneo completo de la base de código

```bash
/agileflow:debt SCAN=full
```

Escaneo profundo de toda la base de código, incluido análisis de complejidad, detección de duplicaciones y análisis de cobertura.

### Generar historias para deuda crítica

```bash
/agileflow:debt SCAN=quick OUTPUT=stories THRESHOLD=high
```

Encuentra elementos de deuda críticos y, opcionalmente, crea historias para ellos.

### Genere informes e historias

```bash
/agileflow:debt SCAN=full OUTPUT=both THRESHOLD=medium
```

Crea informes de deuda y genera historias para elementos de deuda de gravedad superior a media.

## Flujo de trabajo

Después de generar un informe de deuda, podrá:

### 1. Revisar el informe
Entender qué deuda existe y dónde está.

### 2. Generar historias (opcional)
```bash
/agileflow:debt OUTPUT=stories
```

Crea historias para artículos de deuda críticos con:
- Criterios de aceptación centrados en el estado "antes/después"
- Análisis de impacto
- Esfuerzo estimado
- Referencias a características afectadas

### 3. Guardar en Documentación
```bash
/agileflow:debt OUTPUT=report
```

Guarda el informe en `docs/08-project/tech-debt-YYYYMMDD.md` y actualiza el README con un enlace.

### 4. Crear deuda épica
Si existen muchos elementos críticos, crea una epopeya:
- EP-XXXX: Reducción de Deuda Técnica
- Historias de los 5 elementos críticos principales
- Realiza un seguimiento del progreso de la reducción de la deuda

## Uso de información de deuda

### Planificar sprints
Reserve entre el 10 y el 20 % de cada sprint para reducir la deuda:

```bash
/agileflow:sprint --include-debt --debt-threshold=high
```

### Retrospectivas
Seguimiento de la reducción de la deuda en retrospectivas:

```bash
/agileflow:retro --analyze-debt
```

### Panel
Cree un panel de deuda que se actualice semanalmente:

```text
docs/08-project/debt-dashboard.md

# Debt Dashboard (Weekly)

Week 52: 42 items (↑ from 39)
Week 51: 39 items (↓ from 43)

Critical: 3 items (target: reduce to 1 this sprint)
```

## Mejores prácticas

1. **Be Honest About Estimates**: Errar en el lado alto
2. **Link to Features**: Priorizar la deuda que bloquea nuevos trabajos
3. **Celebrate Reductions**: Reconocer el pago de la deuda en retrospectivas
4. **Prevent Accumulation**: Aplicar pruebas en CI para código nuevo
5. **Review Regularly**: Ejecute un análisis de deuda mensual o trimestralmente

## Prevención de deuda

Agregue revisión de deuda a su CI/CD:

```yaml
# Run weekly technical debt scan
- cron: '0 0 * * 1'  # Every Monday
  name: Technical debt scan
  run: /agileflow:debt SCAN=full OUTPUT=report
```

## Integración con el Desarrollo

Al implementar funciones, verifique si generan deuda:

```bash
# Before completing story
/agileflow:debt SCAN=quick THRESHOLD=high

# If debt created, track in story
/agileflow:status STORY=US-0042 TECH_DEBT=true DEBT_SCORE=42
```

## Plantilla de historia para la deuda

Al crear historias para artículos de deuda:

```markdown
---
story_id: US-XXXX
title: "Refactor auth service to separate concerns"
type: tech-debt
debt_score: 87
estimate: 3d
---

## Current State (Before)
- auth.ts contains DB queries, business logic, and HTTP handling
- 287 lines, complexity 18
- Hard to test, hard to extend

## Desired State (After)
- Clear separation: AuthService, AuthRepository, AuthController
- Each \<100 lines, complexity \<8
- 80%+ test coverage
- Enables US-0055 (OAuth)

## Acceptance Criteria
- [ ] AuthRepository handles all DB operations
- [ ] AuthService contains only business logic
- [ ] AuthController handles only HTTP/routing
- [ ] Tests cover each layer independently
- [ ] All existing auth tests still pass
```
