# /adr:update

Aktualisieren Sie den Status eines bestehenden ADR oder fügen Sie Ergänzungen hinzu, die Statusänderungen dokumentieren.

## Schnellstart

```bash
/agileflow:adr:update NUMBER=0042 STATUS=accepted REASON="Team approved after review"
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `NUMBER` | Ja | - | Zu aktualisierende ADR-Nummer (4-stellig, z. B. 0042) |
| `STATUS` | Nein | - | Neuer Status: `proposed`, `accepted`, `deprecated`, `superseded` |
| `REASON` | Nein | - | Grund für die Statusänderung |
| `SUPERSEDED_BY` | Nein | - | Bei Ersatz die neue ADR-Nummer (z. B. 0045) |

## Beispiele

### Akzeptieren Sie einen ADR-Vorschlag

```bash
/agileflow:adr:update NUMBER=0042 STATUS=accepted REASON="Team approved after security review"
```

### Eine ADR ablehnen

```bash
/agileflow:adr:update NUMBER=0035 STATUS=deprecated REASON="Context changed, approach is no longer applicable"
```

### ADR als ersetzt markieren

```bash
/agileflow:adr:update NUMBER=0042 STATUS=superseded SUPERSEDED_BY=0045 REASON="New requirements made this approach obsolete"
```

### Veraltete ADR reaktivieren

```bash
/agileflow:adr:update NUMBER=0030 STATUS=accepted REASON="Reconsidered in light of new project constraints"
```

### Interaktiver Modus

```bash
/agileflow:adr:update NUMBER=0042
```

Ohne STATUS oder REASON führt Sie der Befehl durch den Update-Vorgang.

## Statusübergänge

### Gültige Übergänge

| Von | Zu | Gemeinsamer Grund |
|------|-----|---------------|
| `proposed` | `accepted` | Vom Team genehmigte Entscheidung |
| `proposed` | `rejected` | Alternative gewählt, nicht realisierbar |
| `accepted` | `deprecated` | Nicht mehr relevant, Kontext geändert |
| `accepted` | `superseded` | Besserer Ansatz gefunden, ersetzt durch neues ADR |
| `deprecated` | `accepted` | Überarbeitet, immer noch gültig |
| `superseded` | `accepted` | Ersatz hat nicht geklappt |

## Arbeitsablauf

Wenn Sie den Befehl ausführen:

1. **Überprüft**, dass die ADR vorhanden ist
2. **Nach neuem Status fragen** (falls nicht angegeben)
3. **Erfordert einen Grund** für jede Statusänderung
4. **Zeigt eine Vorschau** dessen, was aktualisiert wird
5. **Bitte vor dem Schreiben um Bestätigung**
6. **Hängt Änderung an**, um die Änderung zu dokumentieren
7. **Aktualisiert Frontmatter** mit neuem Status und Zeitstempel

## Änderungsformat

Wenn sich ein ADR-Status ändert, fügt der Befehl einen Änderungsabschnitt hinzu:

```markdown
## Amendment - 2024-12-30

**Status changed**: accepted → superseded
**Reason**: New requirements made this approach obsolete.
**Superseded by**: [ADR-0045: New Approach](adr-0045-new-approach.md)
**Updated**: 2024-12-30
```

Der ursprüngliche ADR-Inhalt bleibt erhalten; ÄnderungenDokumentieren Sie die Änderung.

## Beispiele nach Szenario

### Szenario 1: Genehmigung einer vorgeschlagenen Entscheidung

```bash
# A team member proposed using a new framework
# After review meeting, you approve it

/agileflow:adr:update NUMBER=0047 STATUS=accepted REASON="Team consensus in architecture review meeting 2024-12-30. Framework choice validated against scalability requirements."
```

Ergebnis: ADR-0047 ist jetzt mit dokumentierter Genehmigung als `accepted` markiert.

### Szenario 2: Ersetzen eines alten Ansatzes

```bash
# You've found a better approach to database caching
# The old ADR (0041) should be superseded by new ADR (0050)

/agileflow:adr:update NUMBER=0041 STATUS=superseded SUPERSEDED_BY=0050 REASON="New caching strategy provides 3x better performance with simpler implementation. See ADR-0050 for details."
```

Ergebnis: ADR-0041 markiert als `superseded` mit Link zu ADR-0050.

### Szenario 3: Überdenken einer veralteten Entscheidung

```bash
# Project direction changed, old approach might be relevant again
/agileflow:adr:update NUMBER=0020 STATUS=accepted REASON="Project switched to microservices architecture. Original monolithic approach may apply to specific subdomain. Reactivating for team consideration."
```

Ergebnis: Die Markierung von ADR-0020 als veraltet wurde aufgehoben, jetzt `accepted` zur Diskussion.

### Szenario 4: Kontextänderung dokumentieren

```bash
# Business requirements changed, a decision no longer applies
/agileflow:adr:update NUMBER=0038 STATUS=deprecated REASON="Support for legacy browsers dropped in new product roadmap. This ADR addressed IE11 compatibility which is no longer required."
```

Ergebnis: ADR-0038 wurde als `deprecated` markiert und erklärt, warum es veraltet ist.

## Best Practices

### Wann der ADR-Status aktualisiert werden muss

- **Genehmigen** vorgeschlagene ADRs nach Teamüberprüfung
- **Veraltet**, wenn sich Kontext oder Anforderungen ändern
- **Ersetzen**, wenn ein neueres ADR diesen Ansatz ersetzt
- **Reaktivieren**, wenn sich die Umstände ändern und die alte Entscheidung wieder relevant ist

### Gute Gründe schreiben

**Gute Gründe:**
- „Teamkonsens nach Architekturüberprüfung“
- „Neuer PerformerCE-Anforderungen machen diesen Ansatz ineffizient.“
- „Geschäftsanforderung hat die Einschränkung entfernt, die diese Entscheidung ausgelöst hat“
- „Erfolgreich umgesetzt und in der Produktion validiert“

**Schlechte Gründe:**
- „Nicht mehr benötigt“
- „Es gibt einen besseren Ansatz“
- „Team hat entschieden“

### Ersetzung dokumentieren

Beim Ersetzen eines ADR:
1. Geben Sie bei der neuen ADR-Nummer immer `SUPERSEDED_BY` an
2. Erklären Sie, warum der neue Ansatz besser ist
3. Einzelheiten zur Implementierung finden Sie im neuen ADR
4. Bewahren Sie den Original-ADR für die Historie auf

### Implementierungsarbeit erstellen

Nach der Genehmigung einer ADR:

```bash
# Mark ADR as accepted
/agileflow:adr:update NUMBER=0050 STATUS=accepted REASON="Team approved in review"

# View the ADR to plan implementation
/agileflow:adr:view NUMBER=0050

# Create stories or epic from the ADR
/agileflow:epic TITLE="Implement ADR-0050" ...
```
