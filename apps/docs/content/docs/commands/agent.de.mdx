# /Agent

Integrieren Sie neue Agenten mit Profilen, Systemaufforderungen (Verträgen) und Umfangsgrenzen für eine effektive Teamzusammenarbeit.

## Schnellstart

```bash
/agileflow:agent AGENT_ID=AG-UI ROLE="Frontend Developer" TOOLS="Read,Write,Bash" SCOPE="src/components/,frontend"
```

## Parameter

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `AGENT_ID` | Yes | - | Unique agent identifier (e.g., AG-UI, AG-API) |
| `ROLE` | Yes | - | Agent's role or title |
| `TOOLS` | No | Read,Write,Edit,Bash | Comma-separated available tools |
| `SCOPE` | No | - | Directories and story tags agent can access |

## Beispiele

### Onboard-Frontend-Entwickler

```bash
/agileflow:agent AGENT_ID=AG-UI ROLE="Frontend Developer" TOOLS="Read,Write,Edit,Bash,Glob" SCOPE="src/components/,src/pages/,frontend,ui"
```

Erstellt ein Agentenprofil mit:
- Rolle und Verantwortlichkeiten
- Verfügbare Tools (Lesen, Schreiben, Bearbeiten, Bash)
- Bereichsgrenzen (nur src/components/ und src/pages/)
- Systemaufforderung (Vertrag) mit strengen Regeln

### Onboard-API-Entwickler

```bash
/agileflow:agent AGENT_ID=AG-API ROLE="Backend API Developer" TOOLS="Read,Write,Edit,Bash,Grep" SCOPE="src/api/,src/services/,backend,api"
```

### Onboard-CI/CD-Spezialist

```bash
/agileflow:agent AGENT_ID=AG-CI ROLE="CI/CD Automation Engineer" TOOLS="Read,Write,Bash,Grep" SCOPE=".github/,scripts/,ci,devops"
```

## Ausgabe

### 1. Agentenprofil

Creates agent profile markdown at `docs/02-practices/prompts/agents/agent-<AGENT_ID>.md`

Beispielinhalt:

```text
---
agent_id: AG-UI
role: Frontend Developer
tools: [Read, Write, Edit, Bash, Glob]
scope:
  directories: [src/components/, src/pages/]
  story_tags: [frontend, ui, ux]
---

# AG-UI: Frontend Developer Agent

## Responsibilities
Build and maintain user interface components with React and Tailwind CSS.

## Available Tools
- Read - Read files and directories
- Write - Create new files
- Edit - Modify existing files
- Bash - Run commands (npm, git)
- Glob - Search for files by pattern

## Scope Boundaries

### Allowed Directories
- `src/components/` - UI components
- `src/pages/` - Page layouts
- `public/` - Static assets

### Restricted Directories
- `src/api/` - Backend code
- `.github/` - CI/CD workflows

### Story Tags
- `frontend` - UI features
- `ui` - Component work
- `ux` - User experience
- `design` - Design implementation

## System Prompt (Contract)

### Scope Restrictions
You can ONLY modify files in:
- `src/components/**`
- `src/pages/**`

You CANNOT modify:
- Backend code in `src/api/`
- CI/CD workflows in `.github/`

### Testing Requirements
Before committing, run:
- `npm test -- src/components/` - Run component tests
- `npm run lint:css` - Check Tailwind usage

### Commit Message Format
Use conventional commits with `ui:` prefix:
- `feat(ui): add new button component`
- `fix(ui): correct responsive breakpoint`
- `style(ui): update component spacing`

### Status Updates
After completing a story:
1. Update `docs/09-agents/status.json` with new status
2. Log to `docs/09-agents/bus/log.jsonl`
3. Include completion time and any blockers

### Code Standards
- TypeScript required (no .js files)
- JSDoc comments on exported components
- Storybook stories for all components
- 80%+ test coverage for components
- Accessibility checks (axe-core)
```

### 2. Agentenliste

Aktualisiert oder erstellt `docs/09-agents/roster.yaml`:

```yaml
agents:
  AG-UI:
    id: AG-UI
    role: Frontend Developer
    tools: [Read, Write, Edit, Bash, Glob]
    scope:
      directories: [src/components/, src/pages/]
      story_tags: [frontend, ui, ux]
  AG-API:
    id: AG-API
    role: Backend API Developer
    tools: [Read, Write, Edit, Bash, Grep]
    scope:
      directories: [src/api/, src/services/]
      story_tags: [backend, api, database]
```

### 3. Systemaufforderung

Gibt eine komprimierte Systemaufforderung zum Kopieren/Einfügen als Terminalsystemaufforderung aus:

```text
You are AG-UI, a Frontend Developer Agent.

ROLE: Build and maintain user interface components.

SCOPE BOUNDARIES (Strict):
- Can modify: src/components/, src/pages/
- Cannot modify: src/api/, .github/
- Story tags: frontend, ui, ux

TESTING:
- Run npm test before committing
- Maintain 80%+ coverage
- Check accessibility with axe-core

COMMITS:
- Format: "feat(ui):" or "fix(ui):"
- Conventional commits required

STATUS:
- Update docs/09-agents/status.json after completing stories
- Log events to docs/09-agents/bus/log.jsonl
```

## Komponenten des Agentenprofils

### Verantwortlichkeiten

Klare Beschreibung dessen, was der Agent tut:
- Spezifische Domäne (UI, API, CI/CD)
- Verwendete Technologien
- Wichtigste Ergebnisse
- Teamrolle

### Verfügbare Tools

Was der Agent nutzen kann:
- **Read** - Dateien lesen und Code analysieren
- **Write** - Erstellen Sie neue Dateien
- **Edit** - Vorhandene Dateien ändern
- **Bash** - Führen Sie Shell-Befehle aus (git, npm, docker)
- **Glob** - Suchen Sie Dateien nach Muster
- **Grep** - Dateiinhalte durchsuchen

### Geltungsbereichsgrenzen

Wo der Agent arbeiten kann:
- **Allowed directories** - Der Dateiagent kann Änderungen vornehmen
- **Restricted directories** - Dateien-Agent KANN NICHT berühren
- **Story tags** – Welche Arbeit wird diesem Agenten zugewiesen?

### Systemaufforderung (Vertrag)

Der Regelagent muss Folgendes befolgen:
- **Scope restrictions** - Genaue Dateipfade
- **Testing requirements** - Wann/wie man testet
- **Commit conventions** - Verzweigungs- und Nachrichtenformat
- **Status updates** - So verfolgen Sie den Fortschritt
- **Code standards** - Qualitätsanforderungen

## Best Practices

### Benennung von Agenten
Verwenden Sie beschreibende IDs, die die Rolle klar angeben:
- ✅ Gut: AG-UI, AG-API, AG-CI, AG-DEVOPS, AG-SECURITY
- ❌ Schlecht: AG-001, AG-A, AG-DEV1

### Umfang definieren

Machen Sie Grenzen klar erkennbar:

**Good** - Spezifische Pfade:
- `src/components/`
- `src/api/routes/`
- `.github/workflows/`

**Bad** - Vage Grenzen:
- „Frontend-Zeug“
- „Backend-Dinge“

### Testanforderungen

Fügen Sie spezifische Testbefehle hinzu:

**Good** - Klare Befehle:
- Laufen `npm test -- src/components/`
- Überprüfen Sie die Abdeckung mit `npm run coverage`
- Fusseln mit `npm run lint:tsx`

**Bad** - Vage Anweisungen:
- „Tests schreiben“
- „Stellen Sie sicher, dass es funktioniert“

### Story-Tag-Mapping

Weisen Sie Storys basierend auf Tags zu:

| Agent | Story Tags |
|-------|------------|
| AG-UI | frontend, ui, ux, design |
| AG-API | backend, api, database |
| AG-CI | ci, devops, automation |

## Arbeitsablauf

1. Sammeln Sie Agentendetails:
   - Eindeutige ID (AG-DOMAIN)
   - Rolle/Titel
   - Verfügbare Werkzeuge
   - Geltungsbereichsgrenzen
   - Story-Tags

2. Erstellen Sie ein Agentenprofil aus einer Vorlage
   - Abschnitt „Verantwortlichkeiten“.
   - Werkzeugliste
   - Umfangsgrenzen (klar und spezifisch)
   - Systemaufforderung (Vertrag mit Regeln)

3. Aktualisieren Sie die Agentenliste
   - Eintrag zu docs/09-agents/roster.yaml hinzufügen
   - Karten-ID → Rolle, Tools, Umfang

4. Systemaufforderung generieren
   - Gekürzte Eingabeaufforderung ausgeben
   - Bereit zum Einfügen als Terminalsystem-Eingabeaufforderung

5. Agent validieren
   - Umfangsgrenzen überprüfen (keine Konflikte)
   - Bestätigen Sie die Testanforderungen
   - Überprüfen Sie die Zuweisungen von Story-Tags

## Integration

Agentenprofile lassen sich integrieren mit:
- **Status tracking** - Roster shows active agents
- **Story assignment** - Verwenden Sie Agenten-ID und Tags
- **Capacity planning** - Sehen Sie, wer verfügbar ist
- **Handoffs** - Referenzagentenprofile in der Kommunikation
- **Scope validation** - Verhindern Sie Änderungen außerhalb des Gültigkeitsbereichs

## Verwenden Sie Wann

- **Onboarding new team members** - Profil und Vertrag erstellen
- **Adding specialized roles** - Agenten für Sicherheit, Leistung und Barrierefreiheit
- **Restructuring team** - Bereichsgrenzen aktualisieren
- **Training agents** - Verwenden Sie die Systemeingabeaufforderung als Referenz
- **Resolving conflicts** - Klarer Geltungsbereich verhindert Überschneidungen
