# /auto

Generieren Sie automatisch User Stories aus Produktartefakten wie PRDs, Modellen, API-Spezifikationen oder Designdokumenten. Spart Zeit beim Aufschlüsseln von Anforderungen in umsetzbare Geschichten.

## Schnellstart

```bash
/agileflow:auto SOURCE=<file-or-url> [EPIC=<EP-ID>] [AUTO_CREATE=no]
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `SOURCE` | Ja | - | Pfad zur Datei oder URL (PRD, Mockup, API-Spezifikation, Flussdiagramm) |
| `EPIC` | Nein | Automatisch generiert | Verknüpfen Sie Geschichten mit vorhandenem Epos (z. B. `EP-0010`) |
| `OWNER` | Nein | Automatisch erkannt | Standardeigentümer für generierte Storys (z. B. `AG-API`) |
| `AUTO_CREATE` | Nein | `no` | `yes` zum sofortigen Erstellen, `no` zur ersten Vorschau |

## Beispiele

### Vorschau der Geschichten von PRD

```bash
/agileflow:auto SOURCE=docs/requirements/auth-prd.md
```

Liest das Dokument mit den Authentifizierungsanforderungen und generiert eine Story-Vorschau, ohne Dateien zu erstellen.

### Aus API-Spezifikation generieren

```bash
/agileflow:auto SOURCE=docs/api/openapi.json OWNER=AG-API
```

ExtrAgiert Endpunkte gemäß der OpenAPI/Swagger-Spezifikation und generiert eine Story pro Endpunkt.

### Aus Figma-Mockup generieren

```bash
/agileflow:auto SOURCE=https://figma.com/file/abc123 AUTO_CREATE=no
```

OCR ist das Mockup-Design zur Identifizierung von Bildschirmen, Formularen und Interaktionen und generiert dann UI-Storys.

### Sofort generieren und erstellen

```bash
/agileflow:auto SOURCE=docs/requirements/payment-prd.md EPIC=EP-0011 AUTO_CREATE=yes
```

Erstellt Storys und erstellt alle Dateien sofort (Vorschauschritt überspringen).

## Unterstützte Quelltypen

### Produktanforderungsdokument (PRD)

**Formate**: `.md`, `.docx`, `.pdf`, `.txt`

**Auszüge**:
- Funktionsbeschreibungen
- Benutzerpersönlichkeiten und Anwendungsfälle
- Erfolgskriterien und -metriken
- Grenzfälle und Fehlerszenarien
- Nichtfunktionale Anforderungen

**Beispiel**:
```markdown
# User Authentication Feature

Users should be able to:
- Register with email and password
- Reset forgotten password via email
- Login with OAuth (Google, GitHub)
- Manage account settings

Requirements:
- Email verification required
- Password minimum 8 characters
- Session expires after 24 hours
```

**Generierte Geschichten**: 4–6 Geschichten zu Registrierung, Anmeldung, Passwort-Zurücksetzung, OAuth und Sitzungsverwaltung.

### API-Dokumentation

**Formate**: OpenAPI/Swagger JSON, API-Dokumente `.md`

**Auszüge**:
- Endpunkte (HTTP-Methode + Pfad)
- Anforderungs-/Antwortschemata
- Authentischtionsanforderungen
- Fehlercodes und Randfälle
- Ratenbegrenzung und Quoten

**Beispiel OpenAPI**:
```yaml
paths:
  /api/users:
    post:
      summary: Create user
      requestBody:
        schema:
          properties:
            name: string
            email: string
```

**Generierte Stories**: Eine pro Endpunkt mit Akzeptanzkriterien aus dem Schema.

### UI-Modelle und -Designs

**Formate**: Figma-URL, `.png`/`.jpg` (mit OCR), Designspezifikation `.md`

**Auszüge**:
- Bildschirmlayouts und Komponenten
- Benutzerinteraktionen und -flüsse
- Formularfelder und Validierung
- Navigation und Links
- Systemnutzung entwerfen

**Beispiel**: Screenshot des Anmeldeformulars generiert Storys für Formularkomponente, Validierung, Fehlermeldungen und Link zum Zurücksetzen des Passworts.

### Benutzerflussdiagramme

**Formate**: Mermaid `.mmd`, PlantUML, Flussdiagramme (`.png` mit OCR)

**Auszüge**:
- Schritte in der Benutzerreise
- Entscheidungspunkte und Zweige
- Alternative Wege
- Erfolgs- und Misserfolgsszenarien

**Beispiel**: Das Authentifizierungsflussdiagramm generiert Storys für jeden Schritt.

## Story-Generierungsprozess

### Schritt 1: Quelle analysieren
- Dokument identifizierenTyp
- Extrahieren Sie Anforderungen und Funktionen
- Gruppenbezogene Anforderungen
- Technische Spezifikationen analysieren

### Schritt 2: Funktionen gruppieren
- Verwandte Anforderungen werden zu einer Geschichte
- Vermeiden Sie die Vervielfältigung von Geschichten
- Stellen Sie sicher, dass die Geschichten unabhängig sind
- Überprüfen Sie die INVEST-Kriterien

### Schritt 3: Geschichten generieren
- Erstellen Sie einen Titel aus der Funktionsbeschreibung
- Schreiben Sie Akzeptanzkriterien im Format „Gegeben/Wann/Dann“.
- Aufwand abschätzen (0,5d, 1d, 1,5d, 2d)
- Weisen Sie den entsprechenden Eigentümer zu
- Abhängigkeiten identifizieren

### Schritt 4: Vorschau anzeigen
- Alle generierten Geschichten anzeigen
- Schätzungen und Eigentümer anzeigen
- Zeigen Sie die Anzahl der Geschichten und den Gesamtaufwand an
- Bitten Sie vor der Erstellung um Genehmigung

### Schritt 5: Dateien erstellen (falls genehmigt)
- Epos erstellen (falls nicht angegeben)
- Erstellen Sie Story-Dateien mit Vorlagen
- Test-Stubs erstellen
- Status.json aktualisieren
- An bus/log.jsonl anhängen

## Regeln für die Story-Generierung (INVEST)

Generierte Geschichten folgen den INVEST-Kriterien:

| Kriterium | Beschreibung |
|-----------|-------------|
| **Unabhängig** | Kann ohne weitere Geschichten abgeschlossen werden |
| **Verhandelbar** | Details können nach der Erstellung angepasst werden |
| **Wertvoll** | Bietet Mehrwert für den Benutzer oder für das Team |
| **Schätzbar** | Kann dimensioniert werden (0,5d-2d) |
| **Klein** | In einem Sprint abschließbar |
| **Testbar** | Verfügt über klare Akzeptanzkriterien |

Geschichten >2d werden in kleinere unterteilt.

## Schätzungsrichtlinien

| Schätzung | Wann zu verwenden |
|----------|------------|
| `0.5d` | Einfaches CRUD, Formularfeld, Basiskomponente |
| `1d` | Standardfunktion, moderate Komplexität |
| `1.5d` | Komplexe Logik, mehrere Validierungen, Integration |
| `2d` | Signifikanter Refaktor oder Hauptkomponente |
| `>2d` | Die Geschichte wird in kleinere Teile unterteilt |

## Beispiel: PRD zu Stories

### PRD eingeben

```markdown
# User Authentication Feature

## Overview
Allow users to register, log in, and manage accounts.

## Requirements
- Users register with email and password
- Email verification required
- Password reset via email
- Login with OAuth (Google, GitHub)
- Session management and expiry
```

### Vorschau der generierten Geschichten

```text
Story Generation Preview

Source: docs/requirements/auth-prd.md
Epic: EP-0010 (User Authentication) - CREATED
Stories: 5

1. US-0050: User registration (1.5d, AG-API)
   "Users can register with email and password"

2. US-0051: Email verification (1d, AG-API)
   "Verify email before allowing login"

3. US-0052: Password reset flow (1d, AG-API)
   "Users can reset forgotten password"

4. US-0053: OAuth login - Google (1.5d, AG-API)
   "Users can log in with Google account"

5. US-0054: OAuth login - GitHub (1.5d, AG-API)
   "Users can log in with GitHub account"

Total Estimate: 6.5 days
Recommended Owner: AG-API

Create these stories? (YES/NO)
```

## Generierung von Akzeptanzkriterien

Geschichten werden im Format „Gegeben/Wann/Dann“ generiert:

```markdown
## Acceptance Criteria

- Given a user on registration page
  When they enter valid email and 8+ character password
  Then account is created and verification email sent

- Given invalid password (< 8 characters)
  When they try to submit
  Then form shows "Password must be at least 8 characters"

- Given user opens verification email
  When they click verification link
  Then email is marked verified and login works
```

## Ausgabedateien

Der Befehl erstellt:

1. **Epische Datei** (`docs/05-epics/EP-XXXX.md`)
   -Epischer Titel und Ziel
   - Referenz zum Quelldokument
   - Liste mit Kindergeschichten

2. **Story-Dateien** (`docs/06-stories/EP-XXXX/US-XXXX.md`)
   - Generierter Inhalt aus der Quelle
   - Akzeptanzkriterien
   - Technische Hinweise
   - Teststrategie

3. **Test-Stubs** (`docs/07-testing/test-cases/US-XXXX.md`)
   - Teststruktur auf AC ausgerichtet
   - Platzhalter für die Testdurchführung

4. **Statusverfolgung** (`docs/09-agents/status.json`)
   - Alle Geschichten mit Status hinzugefügt: `ready`

5. **Kommunikation** (`docs/09-agents/bus/log.jsonl`)
   - Ereignis zur Story-Generierung protokolliert

## Arbeitsablauf

1. **Quelldokument vorbereiten** – Stellen Sie sicher, dass PRD/Spezifikation vollständig ist
2. **Vorschau generieren** – Führen Sie `/auto` mit `AUTO_CREATE=no` aus
3. **Überprüfen Sie generierte Geschichten** – Überprüfen Sie Richtigkeit und Vollständigkeit
4. **Bei Bedarf anpassen** – Vorschau bearbeiten oder mit anderen Parametern neu generieren
5. **Storys erstellen** – Mit `AUTO_CREATE=yes` ausführen oder Vorschau genehmigen
6. **Storys validieren** – Verwenden Sie `/story-validate` für generierte Storys
7. **Sprint planen** – Beziehen Sie validierte Storys ein`/sprint`

## Häufige Szenarien

### Szenario: API hat 15 Endpunkte

**Problem**: Zu viele Storys (verstößt gegen die 0,5d-2d-Schätzungsregel)

**Lösung**: Der Befehl gruppiert automatisch verwandte Endpunkte:
- Endpunkte für dieselbe Ressource (GET, POST, PATCH, DELETE) → Eine Geschichte
- Endpunkte auflisten und detailliert → Eine Geschichte
- Authentifizierungsendpunkte → Eine Geschichte

**Ergebnis**: 15 Endpunkte → 5-6 Geschichten

### Szenario: PRD hat unklare Anforderungen

**Problem**: Generierte Geschichten wirken vage

**Lösung**:
1. PRD zuerst mit den Stakeholdern klären
2. Führen Sie `/auto` mit der aktualisierten Quelle erneut aus
3. Oder bearbeiten Sie generierte Geschichten manuell mit `/story`

### Szenario: Generierte Storys entsprechen nicht der Teamgeschwindigkeit

**Problem**: Schätzungen erscheinen zu optimistisch/pessimistisch

**Lösung**:
1. Überprüfen Sie Geschichten und passen Sie Schätzungen an
2. Berücksichtigen Sie die historische Geschwindigkeit des Teams
3. Führen Sie `/velocity` aus, um die Teamkapazität zu ermitteln
4. Teilen Sie große Geschichten in kleinere auf

## Qualitätscheckliste

Vor der Genehmigung des generierten SpeichersThemen:

- [ ] Alle Geschichten haben klare Akzeptanzkriterien
- [ ] Geschichten sind unabhängig (keine versteckten Abhängigkeiten)
- [ ] Schätzungen sind realistisch (Bereich 0,5-2 Tage)
- [ ] Akzeptanzkriterien verwenden das Format „Gegeben/Wann/Dann“.
- [ ] Technische Hinweise sind korrekt
- [ ] Geschichten decken Hauptszenarien ab (Happy Path + Randfälle)
- [ ] Es werden keine Geschichten dupliziert
- [ ] Das epische Ziel ist klar und spezifisch
- [ ] Besitzerzuweisungen sind angemessen

## Integration mit anderen Befehlen

- **Nach der Generierung**: Verwenden Sie `/story-validate`, um die Qualität sicherzustellen
- **Vor dem Sprint**: Verwenden Sie `/sprint`, um Geschichten auszuwählen
- **Für Abhängigkeiten**: Verwenden Sie `/deps`, wenn Storys Beziehungen haben
- **Während der Entwicklung**: Verwenden Sie `/babysit` für Implementierungshilfe
- **Zum Testen**: Verwenden Sie `/verify`, um Tests auszuführen

## Best Practices

- **Beginnen Sie mit den Anforderungen** - Investieren Sie Zeit in klare PRD oder Spezifikationen
- **Generierte Geschichten überprüfen** - Akzeptieren Sie die KI-Generierung nicht blind
- **Validierung vor dem Sprint** - Verwenden Sie `/story-validate` für alle gEnergetische Geschichten
- **Gruppenbezogene Anforderungen** - Fokussierteres Epos = bessere Geschichten
- **Seien Sie in der Quelle genau** - Vage Anforderungen → vage Geschichten
- **Abhängigkeiten berücksichtigen** – Manuelle Überprüfung auf versteckte Abhängigkeiten
- **Konservativ schätzen** - Puffer für Unbekannte hinzufügen

## Einschränkungen und Überlegungen

- **KI kann falsch interpretieren** - Überprüfen Sie generierte Geschichten auf Richtigkeit
- **Der Kontext ist wichtig** - Den generierten Geschichten mangelt es möglicherweise an Domänenkenntnissen
- **Abhängigkeiten sind manuell** – Storyübergreifende Abhängigkeiten manuell hinzufügen
- **Schätzungen variieren je nach Team** - Passen Sie die Geschwindigkeit basierend auf Ihrer Geschwindigkeit an
- **Technische Details** - Möglicherweise sind manuelle Ergänzungen zum Architekturkontext erforderlich
