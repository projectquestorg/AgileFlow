# /atribuir

Atribua ou reatribua uma história a um proprietário com atualização de status opcional. Atualiza os metadados da história e transmite a tarefa por meio do barramento de mensagens.

## Início rápido

```bash
/agileflow:assign STORY=US-0001 NEW_OWNER=AG-UI
```

## Parâmetros

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `STORY` | Yes | - | Story ID to assign (e.g., US-0001, US-0042) |
| `NEW_OWNER` | Yes | - | Agent/owner ID (e.g., AG-UI, AG-API, AG-CI) |
| `NEW_STATUS` | No | - | Update status: `ready`, `in-progress`, `blocked`, `in-review`, `done` |
| `NOTE` | No | - | Optional context about the assignment |

## Exemplos

### Atribuir história ao proprietário

```bash
/agileflow:assign STORY=US-0042 NEW_OWNER=AG-UI
```

Atribua a história à equipe de UI sem alterar o status.

### Atribuir e começar a trabalhar

```bash
/agileflow:assign STORY=US-0042 NEW_OWNER=AG-UI NEW_STATUS=in-progress NOTE="Starting login form implementation"
```

Atribua uma história e marque imediatamente como em andamento com o contexto.

### Reatribuir para proprietário diferente

```bash
/agileflow:assign STORY=US-0038 NEW_OWNER=AG-API NEW_STATUS=ready NOTE="Re-prioritized for OAuth implementation"
```

Mova a história de um agente para outro.

### Atribuir de não atribuído

```bash
/agileflow:assign STORY=US-0050 NEW_OWNER=AG-CI NEW_STATUS=ready NOTE="Ready for CI/CD integration work"
```

Dê trabalho a uma história não atribuída.

### Reatribuir devido ao bloqueador

```bash
/agileflow:assign STORY=US-0055 NEW_OWNER=AG-DEVOPS NEW_STATUS=blocked NOTE="Blocked on infrastructure provisioning. Needs DevOps attention"
```

Reatribuir quando forem necessários conhecimentos diferentes.

## O que é atualizado

O comando atualiza dois arquivos:

### 1. Arquivo de história

Updates story frontmatter (in `docs/06-stories/**/<STORY>.md`):

```yaml
---
owner: AG-UI
status: in-progress
updated: 2025-12-24T14:30:00Z
---
```

### 2. Arquivo de rastreamento de status

Atualizações `docs/09-agents/status.json` com atribuição e mudança de status.

### 3. Barramento de mensagens

Acrescenta um evento "assign" a `docs/09-agents/bus/log.jsonl`:

```json
{"ts":"2025-12-24T14:30:00Z","from":"claude","type":"assign","story":"US-0042","owner":"AG-UI","status":"in-progress","text":"Starting login form implementation"}
```

## Valores de status

Ao usar `NEW_STATUS`, escolha entre:

| Status | Meaning | When to Use |
|--------|---------|-------------|
| `ready` | Prepared, waiting to start | After creation or when unblocked |
| `in-progress` | Currently being worked on | When owner begins work |
| `blocked` | Waiting on external dependency | When blocker identified |
| `in-review` | Complete, awaiting approval | Code review or QA |
| `done` | Finished and verified | After all checks pass |

## Exemplo de fluxo de atribuição

Fluxo de trabalho típico por meio de estados de atribuição:

**1. Story created** (HISTÓRIA=US-0042, PROPRIETÁRIO=não atribuído, STATUS=pronto)

```bash
/agileflow:assign STORY=US-0042 NEW_OWNER=AG-UI NEW_STATUS=ready
```

**2. Owner starts work**

```bash
/agileflow:assign STORY=US-0042 NEW_OWNER=AG-UI NEW_STATUS=in-progress
```

**3. Code complete, ready for review**

```bash
/agileflow:status STORY=US-0042 STATUS=in-review PR=https://github.com/...
```

**4. Review complete**

```bash
/agileflow:status STORY=US-0042 STATUS=done
```

**5. If blocked during work**

```bash
/agileflow:status STORY=US-0042 STATUS=blocked SUMMARY="Blocked on API response format"
```

**6. Unblocked, resume work**

```bash
/agileflow:status STORY=US-0042 STATUS=in-progress SUMMARY="API clarified, resuming development"
```

## Saída

O comando exibe:

1. **Diff Preview** - Mostra alterações a serem feitas
2. **Confirmation** - Solicitação SIM/NÃO para confirmar
3. **Success Message** - Confirma a tarefa concluída

Exemplo de saída:

```text
✅ Story Assigned
Story:     US-0042
Owner:     unassigned → AG-UI
Status:    ready (unchanged)
Updated:   2025-12-24T14:30:00Z

Message broadcast: Assigned US-0042 to AG-UI
```

## Segurança JSON

O comando usa manipulação segura de JSON:

- **Never uses echo/cat** - Sempre usa a ferramenta jq ou Edit
- **Validates after update** - Verifica se o JSON é válido antes de terminar
- **Backs up on error** - Pode restaurar a partir do backup se ocorrer corrupção

## Melhores Práticas

1. **Assign immediately** - Não deixe histórias sem atribuir
2. **Include context** - Use NOTA para explicar por que a tarefa aconteceu
3. **Match expertise** - Atribuir ao agente com habilidades relevantes
4. **Balance load** - Distribuir o trabalho de forma justa entre os agentes
5. **Update status** - Considere atualizar o status ao atribuir
6. **Communicate** - A equipe deve entender por que a reatribuição aconteceu

## Cenários Comuns

### Balanceamento de carga

Distribua o trabalho quando um agente estiver sobrecarregado:

```bash
/agileflow:assign STORY=US-0055 NEW_OWNER=AG-DEVOPS NOTE="Load balancing: AG-API at WIP limit"
```

### Treinamento cruzado

Atribua a um agente menos experiente com orientação:

```bash
/agileflow:assign STORY=US-0060 NEW_OWNER=AG-CI NEW_STATUS=ready NOTE="Cross-training opportunity for CI/CD work. Pair with AG-API"
```

### Transferência de experiência

Reatribuir ao especialista quando for necessária experiência:

```bash
/agileflow:assign STORY=US-0065 NEW_OWNER=AG-SECURITY NEW_STATUS=in-progress NOTE="Security vulnerability fix - needs security expertise"
```

### Recuperação do bloqueador

Retomar o trabalho após desbloquear:

```bash
/agileflow:assign STORY=US-0070 NEW_OWNER=AG-UI NEW_STATUS=in-progress NOTE="Unblocked: API endpoint now available, resuming frontend work"
```
