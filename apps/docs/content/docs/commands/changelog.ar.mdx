# / إنشاء سجل التغيير

يمكنك إنشاء سجل التغيير تلقائيًا من سجل الالتزام الخاص بك وسحب الطلبات باستخدام الالتزامات التقليدية والاحتفاظ بتنسيق سجل التغيير.

## بداية سريعة

__لا تترجم_0__

## الغرض

يقوم هذا الأمر بإنشاء سجلات التغيير الاحترافية عن طريق:
- اكتشاف الإصدار من أحدث علامات git تلقائيًا
- جمع الالتزامات والعلاقات العامة المدمجة منذ الإصدار الأخير
- تصنيف التغييرات (المضافة/المتغيرة/الثابتة/المحذوفة/الأمان)
- تسليط الضوء على التغييرات العاجلة
- اقتراح الإصدارات الدلالية بناء على التغييرات
- تحديث `CHANGELOG.md` بالتنسيق المناسب
- إنشاء علامات git وإصدارات GitHub بشكل اختياري

## أمثلة

### الكشف التلقائي عن كل شيء

__لا تترجم_1__

يستخدم أحدث علامة عند بدء الإصدار، ويكتشف الإصدار التالي، وينشئ سجل التغيير.

### تحديد الإصدارات الدقيقة

__لا تترجم_2__

ينشئ قسم سجل التغيير للإصدار 2.5.0 من الالتزامات منذ الإصدار 2.4.0.

### استخدم تنسيق جيثب

__لا تترجم_3__

يُنشئ سجل التغيير بتنسيق ملاحظات إصدار GitHub.

### تغييرات الالتزام التلقائي

__لا تترجم_4__

يقوم بتحديث CHANGELOG.md ويتم تنفيذه تلقائيًا بعد التأكيد.

## كيف يعمل

يتبع الأمر سير عمل ذكي:

### 1. الكشف عن الإصدار

إذا لم يتم توفير `VERSION`، فسيتم اكتشافه من git:
__لا تترجم_5__

ثم يقترح الإصدار التالي بناءً على التغييرات:
- **الرئيسي (2.0.0)**: تم اكتشاف تغييرات جذرية
- **ثانوي (1.X.0)**: ميزات جديدة (الفذ :) فقط
- **التصحيح (1.1.X)**: إصلاحات الأخطاء (الإصلاح :) فقط

### 2. الالتزام بالتحصيل

يجمع يرتكب الخطيئةهـ `SINCE` النقطة:
__لا تترجم_6__

يقوم أيضًا بجلب العلاقات العامة المدمجة من GitHub:
__لا تترجم_7__

### 3. التصنيف

يتم التصنيف تلقائيًا باستخدام الالتزامات التقليدية:

| البادئة | الفئة | مثال |
|--------|---------|---------|
| __غير مترجم_33__ | **أضيفت** | ميزة X جديدة للمستخدمين |
| __غير مترجم_34__ | **ثابت** | خطأ حيث تسبب E في F |
| __غير مترجم_35__ | **تم التغيير** | تحسين الأداء بنسبة 40% |
| __غير مترجم_36__ | **تم التغيير** | طبقة API المعاد هيكلتها |
| __غير مترجم_37__ | **الأمن** | تم تصحيح ثغرة XSS |
| `feat!:` أو `BREAKING CHANGE:` | **تم التغيير** (مع تحذير) | ⚠️ **عاجل** واجهة برمجة التطبيقات المعاد تصميمها |

### 4. إنشاء سجل التغيير

إنشاء إدخال بتنسيق Keep a Changelog:

__لا تترجم_8__

### 5. مراجعة المستخدم والموافقة عليه

يعرض معاينة الفرق ويطلب التأكيد قبل تحديث CHANGELOG.md.

### 6. الإجراءات الاختيارية

في حالة الموافقة:
- التحديثات `CHANGELOG.md`
- إنشاء التزام git بشكل اختياري
- يمكن إنشاء علامة git
- يمكن أن تخلقإصدار جيثب

## تنسيق سجل التغيير

يستخدم الأمر الاحتفاظ بمعيار Changelog مع هذه الأقسام:

### [غير منشورة]
لم يتم إصدار تغييرات التطوير بعد (تبقى دائمًا في الأعلى)

### [الإصدار] - التاريخ
الأقسام الخاصة بكل إصدار:

- **أضيفت** - وظائف جديدة
- **تم التغيير** - تغييرات على الوظائف الموجودة
- **مهملة** - وظيفة ستتم إزالتها قريبًا
- **تمت الإزالة** - تمت إزالة الوظيفة
- **ثابت** - إصلاحات الشوائب
- **الأمان** - إصلاحات الثغرات الأمنية

## كسر التغييرات

تم الكشف عن التغييرات العاجلة وإبرازها بشكل بارز:

**تم الكشف بواسطة**:
- الالتزام بـ `!` بعد النوع: `feat!: redesign API`
- الالتزام بالتذييل `BREAKING CHANGE:`
- العلاقات العامة تحمل علامة `breaking-change`

**معروض باسم**:
__لا تترجم_9__

## ملف الإخراج

يقوم الأمر بتحديث `CHANGELOG.md` بهذه البنية:

__لا تترجم_10__

## سير العمل

يبدو سير العمل الكامل كما يلي:

1. **إظهار نتائج الاكتشاف**
   __لا تترجم_11__

2. **إظهار معاينة سجل التغيير**
   __لا تترجم_12__

3. **اطلب التأكيد**
   __لا تترجم_13__

4. **تطبيق التغييرات**
   - التحديثات CHANGELOG.md
   - الالتزام اختياريًا: `git commit -m "chore(release): update CHANGELOG for v2.5.0"`
   - يقوم بشكل اختياري بإنشاء العلامة: `git tag -a v2.5.0`
   - إنشاء إصدار GitHub بشكل اختياري

### عملية الإصدار

3. **علامة الدفع** لتشغيل خط أنابيب القرص المضغوط
4. ** أنشئ إصدار GitHub ** مع مقتطف من سجل التغيير
5. **الإعلان عن الإصدار** (المدونة، وسائل التواصل الاجتماعي، النشرة الإخبارية)
