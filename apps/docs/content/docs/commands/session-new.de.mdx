# /session:new

Erstellen Sie eine neue parallele Claude Code-Sitzung mit automatischer Git-Worktree-Einrichtung. Dadurch können Sie ohne Konflikte an mehreren Aufgaben gleichzeitig arbeiten.

## Schnellstart

```bash
/agileflow:session:new
```

## Wie es funktioniert

Wenn Sie diesen Befehl ausführen, werden Ihnen Optionen angezeigt:

1. **Sitzung N automatisch erstellen** – Erstellt automatisch eine nummerierte Sitzung (z. B. Sitzung 2, Sitzung 3)
2. **Diese Sitzung benennen** – Geben Sie einen Spitznamen wie „auth“ oder „refactor“ ein.
3. **Vorhandenen Zweig verwenden** – An einen Zweig anhängen, den Sie bereits haben

Der Befehl dann:
1. Erstellt einen neuen Git-Arbeitsbaum in einem parallelen Verzeichnis
2. Registriert die Sitzung in der Sitzungsregistrierung
3. Zeigt Ihnen den Befehl `cd` zum Wechseln zum neuen Arbeitsbereich

## Beispielausgabe

```
Creating Session 2...

✓ Session created successfully

  Session ID: 2
  Path: /home/user/myproject-2
  Branch: session-2

To start working in this session:
  cd /home/user/myproject-2 && claude
```

## Parameter

Dieser Befehl verwendet interaktive Eingabeaufforderungen (AskUserQuestion) anstelle von Befehlszeilenparametern.

| Option | Beschreibung |
|--------|-------------|
| Automatisch erstellen | Verwendet die nächste verfügbare Sitzungs-ID |
| Benannte Sitzung | Erstellt nach MaßSpitzname (z. B. „auth“ → `/home/user/myproject-auth`) |
| Bestehender Zweig | Wird an einen Zweig angehängt, ohne einen neuen zu erstellen |

## Warum parallele Sitzungen verwenden?

### Das Problem

Das Ausführen mehrerer Claude Code-Instanzen im selben Verzeichnis führt zu Folgendem:
- Dateisperrkonflikte (insbesondere unter macOS)
- Es besteht die Gefahr widersprüchlicher Bearbeitungen derselben Dateien
- Probleme mit dem freigegebenen Verzeichnis `~/.claude/`
- Kein Einblick in die Arbeit anderer Sitzungen

### Die Lösung

Git-Arbeitsbäume bieten:
- **Isolierte Verzeichnisse** – Jede Sitzung hat ihren eigenen Ordner
- **Shared .git** – Festplatteneffizient, kein doppeltes Repo
- **Unabhängige Zweige** - Keine Zusammenführungskonflikte während der Arbeit
- **Saubere Trennung** - Sitzungen können nicht aufeinander treten

## Arbeitsablauf

### Starten einer zweiten Aufgabe

```bash
# In your main project
/agileflow:session:new
# Pick "Auto-create Session 2"

# Claude shows:
#   cd /home/user/myproject-2 && claude

# Open new terminal
cd /home/user/myproject-2
claude
```

### Benennen von Sitzungen nach Funktion

```bash
# Create session for auth work
/agileflow:session:new
# Pick "Name this session" → "auth"

# Creates: /home/user/myproject-auth on branch session-auth
```

### Vorhandenen Zweig verwenden

```bash
# You have a feature branch already
git branch
# feature/new-api
# main

/agileflow:session:new
# Pick "Use existing branch" → "feature/new-api"

# Creates worktree attached to that branch
```

## Sitzungsregistrierung

Sitzungen werden in `.agileflow/sessions/registry.json` verfolgt:

```json
{
  "sessions": {
    "1": {
      "path": "/home/user/myproject",
      "branch": "main",
      "is_main": true
    },
    "2": {
      "path": "/home/user/myproject-2",
      "branch": "session-2",
      "nickname": null
    }
  }
}
```

## Best Practices

- **Eine Aufgabe pro Sitzung** - Behalte sSitzungen fokussiert
- **Verwenden Sie Spitznamen für langwierige Arbeiten** - Leichter zu merken als Zahlen
- **Sitzungen beenden, wenn Sie fertig sind** - Verwenden Sie `/session:end` zum Aufräumen
- **Überprüfen Sie den Status vor dem Start** - Verwenden Sie `/session:status`, um zu sehen, was gerade ausgeführt wird

## Verwandte Befehle

- [`/session:status`](/commands/session-status) – Alle Sitzungen anzeigen
- [`/session:resume`](/commands/session-resume) – Wechseln Sie zu einer anderen Sitzung
- [`/session:end`](/commands/session-end) – Beenden und optional bereinigen Sie eine Sitzung
- [`/session:history`](/commands/session-history) – Vergangene Sitzungen anzeigen
