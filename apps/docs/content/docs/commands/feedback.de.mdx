# /Feedback

Sammeln Sie Feedback von Agenten und Menschen zu Storys, Epics und Sprints für eine kontinuierliche Prozessverbesserung.

## Schnellstart

```bash
/agileflow:feedback SCOPE=story STORY=US-0042
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `SCOPE` | Nein | Geschichte | Feedbackumfang: Story, Epos oder Sprint |
| `STORY` | Bedingt | - | Story-ID (erforderlich, wenn SCOPE=Story) |
| `EPIC` | Bedingt | - | Epic-ID (erforderlich, wenn SCOPE=epic) |
| `ANONYMOUS` | Nein | nein | Anonymes Feedback zulassen (ja/nein) |

## Beispiele

### Feedback zum Abschluss der Story

```bash
/agileflow:feedback SCOPE=story STORY=US-0042
```

Nachdem Sie eine Geschichte als erledigt markiert haben, sammeln Sie Folgendes:
- AC-Klarheitsbewertung (1–5)
- Bewertung der Abhängigkeitsauflösung (1–5)
- Schätzgenauigkeitsbewertung (1-5)
- Bewertung der reibungslosen Implementierung (1–5)
- Was gut gelaufen ist
- Was könnte verbessert werden?

### Episches Retrospektiv-Feedback

```bash
/agileflow:feedback SCOPE=epic EPIC=EP-0010
```

Sammle nach Abschluss des epischen Abenteuers:
- Erfolgskennzahlen für epische Ziele
- Was ist bei der Exe gut gelaufen?Schnitt
- Was nicht gut gelaufen ist
- Überraschungen und Erkenntnisse
- Aktionen für das nächste Epos

### Retrospektives Sprint-Feedback

```bash
/agileflow:feedback SCOPE=sprint
```

Sammeln Sie am Ende des Sprints Team-Feedback:
- Weitermachen (weitermachen)
- Stopp (nicht mehr nützlich)
- Start (neue Praktiken)
- Experimente zum Ausprobieren
- Blocker haben diesen Sprint entfernt

### Anonymes Feedback

```bash
/agileflow:feedback SCOPE=story STORY=US-0045 ANONYMOUS=yes
```

Sammelt Feedback ohne Quellenangabe zu sensiblen Themen.

## Ausgabe

### Feedback-Dateien

Speichert Feedback an:
```text
docs/08-project/feedback/<YYYYMMDD>-<story-or-epic-id>.md
```

### Beispiel-Story-Feedback

```markdown
## Story Feedback: US-0042

**Completed by**: AG-API
**Date**: 2025-12-22

### Ratings (1-5)

```

## Feedback-Typen

### 1. Feedback zum Abschluss der Geschichte

Bewertungen für:
- **AC-Klarheit** - Waren die Akzeptanzkriterien klar? (1-5)
- **Abhängigkeiten** – Wurden Blocker behoben? (1-5)
- **Genauigkeit der Schätzung** - War die Schätzung korrekt? (1-5)
- **Implementierung** – Wie reibungslos verlief der Prozess? (1-5)
- **Tests** - Waren die Tests ausreichend? (1-5)
- **Dokumentation** - War die Dokumentation ausreichend? (1-5)

Plus unbefristet:
- Was ist gut gelaufen? (2-3 Kugeln)
- Was könnte verbessert werden? (2-3 Kugeln)
-Blocker angetroffen?
- Erkenntnisse/Einsichten?

### 2. Feedback zur Agentenleistung

Verfolgt die Wirksamkeit:
- Geschichten abgeschlossen
- Geschichten blockiert
- Durchschnittliche Fertigstellungszeit
- Stärken beobachtet
- Bereiche mit Verbesserungspotenzial
- Empfehlungen

### 3. Epische Retrospektive

Nach epischem Abschluss:
- Erfolgskennzahlen vs. Ziele
- Was ist gut gelaufen?
- Was ist nicht gut gelaufen?
- Überraschungen und Erkenntnisse
- Aktionen für das nächste Epos

### 4. Sprint-Retrospektive

Am Ende des Sprints:
- Weitermachen (weitermachen)
- Stopp (nicht mehr nützlich)
- Start (neue Praktiken)
- Experimente zum Ausprobieren
- Blocker entfernt
- Wiederkehrende Probleme

## Metriken verfolgt

Das Feedbacksystem verfolgt:

| Metrisch | Ziel | Zweck |
|--------|--------|---------|
| Bewertung der Story-Klarheit | >4,0 | AC-Qualität |
| Schätzgenauigkeit | innerhalb von 50 % | Planungsverbesserung |
| Blockerfrequenz | weniger als 20 % der Geschichten | Abhängigkeitsmanagement |
| Durchschnittliche Testabdeckung | >85 % | Codequalität |
| Abschlussgeschwindigkeit | Aufwärtstrend | Teamdurchsatz |

## Analyse & Einblicke

### Muster DeSchutz

Erkennt Muster automatisch:
- **Unklarer AC** – Geschichten mit Klarheitswerten unter 3 → Vorlage verbessern
- **Schlechte Schätzungen** - Große Abweichung → Schätzungsleitfaden überarbeiten
- **Häufige Blocker** – Geschichten werden oft blockiert → Abhängigkeitsverfolgung verbessern
- **Geringe Testabdeckung** - Tests unzureichend → Standards früher durchsetzen

### Umsetzbare Verbesserungen

Automatisch generiert:
- **Verbesserungsgeschichten** für wiederkehrende Probleme
- **Prozessempfehlungen** für erkannte Probleme
- **Anerkennung** für Siege und Spitzenleistungen
- **ADRs** für Architekturlernen

### Retrospektive Berichte

Erzeugt umfassende Zusammenfassungen:
- Gesamtstimmung (verbessernd/rückläufig/stabil)
- Top-Siege mit Jubel
- Top-Herausforderungen
- Empfohlene Maßnahmen
- Einblicke und Erkenntnisse des Teams

## Arbeitsablauf

1. Automatische Eingabeaufforderung an Triggerpunkten:
   - Story-Status ändert sich auf „erledigt“
   - Epic erreicht 100 % Abschluss
   - Sprint-Enddatum erreicht
2. Präsentieren Sie das Feedback-Formular mit vorab ausgefülltem Kontext
3. Fragen Sie: „Jetzt Feedback geben? (JA/NEIN/SPÄTER)“"
4. Wenn JA: Sammeln Sie interaktiv Bewertungen und Kommentare
5. Im Feedback-Verzeichnis speichern
6. Analysieren Sie Muster im gesamten Feedback
7. Schlagen Sie Verbesserungsgeschichten für Probleme vor

## Verwenden Sie Wann

- **Fertigstellung der Geschichte** - Unmittelbar nach der Markierung als erledigt
- **Epische Retrospektive** - Nach epischem Abschluss
- **Sprint-Ende** – Team-Learnings sammeln
- **Prozessverbesserung** - Monatliche Überprüfung der Feedbackmuster
- **Leistungsüberprüfung** – Vierteljährliche Agentenbewertungen
- **Teamrückblicke** – Formelle Sprintzeremonien

## Automatische Auslöser

Feedback-Eingabeaufforderungen werden angezeigt:
- Wenn sich der Story-Status auf „abgeschlossen“ ändert
- Wenn das Epos zu 100 % abgeschlossen ist
- Zu konfigurierten Sprint-Enddaten
- Kann jederzeit manuell angefordert werden

## Integration

Feedback fließt ein in:
- **Retrospektiven** – Einblicke in Sprint-Retros
- **Metriken** – Bewertungen im Laufe der Zeit verfolgt
- **Story-Erstellung** – Verbesserungs-Storys automatisch generieren
- **ADRs** – Dokumentieren Sie Erkenntnisse als Architekturentscheidungen
- **Agentenprofile** – Leistungsdatenaktualisierungslisters
