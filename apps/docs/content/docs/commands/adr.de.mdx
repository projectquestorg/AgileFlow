# /adr

Erstellen Sie Architecture Decision Records (ADRs), um wichtige Architektur- und Designentscheidungen in Ihrem Projekt zu dokumentieren.

## Schnellstart

```bash
/agileflow:adr NUMBER=0001 TITLE="Use PostgreSQL for persistence" CONTEXT="Need reliable database" DECISION="PostgreSQL chosen" CONSEQUENCES="Better reliability, steeper learning curve"
```

## Parameter

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `NUMBER` | Yes | - | 4-digit ADR number (e.g., 0001, 0042) |
| `TITLE` | Yes | - | Short title of the architectural decision |
| `CONTEXT` | Yes | - | Why this decision was needed |
| `DECISION` | Yes | - | What was chosen and why |
| `CONSEQUENCES` | Yes | - | Trade-offs, positive and negative impacts |
| `LINKS` | No | - | Optional links to related documentation |

## Beispiele

### Grundlegendes ADR: Datenbankauswahl

```bash
/agileflow:adr NUMBER=0001 TITLE="Use PostgreSQL for persistence" CONTEXT="We need a reliable database with ACID guarantees for user data and transactions" DECISION="PostgreSQL chosen over MongoDB for its strong consistency guarantees and mature ecosystem" CONSEQUENCES="Better data integrity and reliability. Team needs PostgreSQL training. Slightly higher operational complexity than NoSQL alternatives"
```

Dadurch wird eine neue Datei unter erstellt `docs/03-decisions/adr-0001-use-postgresql-for-persistence.md` mit der vollständigen ADR-Struktur.

### ADR mit verwandten Links

```bash
/agileflow:adr NUMBER=0002 TITLE="Adopt TypeScript for all new code" CONTEXT="Type safety issues in JavaScript codebase causing bugs in production" DECISION="Migrate to TypeScript with strict type checking enabled" CONSEQUENCES="Fewer type-related bugs. Initial slower development. Better IDE support and refactoring. Requires team training" LINKS="- Related Story: US-0042\n- Related ADR: ADR-0001"
```

## Ausgabe

Wenn Sie den Befehl ausführen, sehen Sie Folgendes:

1. **Preview**: Zeigt den generierten ADR-Inhalt vollständig an
2. **Confirmation**: Bittet um JA/NEIN um die Genehmigung des ADR
3. **File Creation**: Erstellt die ADR-Datei in `docs/03-decisions/` Verzeichnis
4. **Filename Format**: `adr-<NUMBER>-<slug>.md` (automatisch aus Titel generiert)

Die generierte ADR umfasst:

```markdown
---
number: 0001
title: Use PostgreSQL for persistence
date: 2025-12-24
status: accepted
tags: [database, architecture]
---

# ADR-0001: Use PostgreSQL for persistence

## Context
We need a reliable database...

## Decision
We will use PostgreSQL...

## Consequences
- Better data integrity...
- Team needs training...
```

## Best Practices

### ADR-Nummerierung
- Nummerieren Sie fortlaufend (0001, 0002, 0003, ...)
- Überspringen Sie keine Zahlen und verwenden Sie sie nicht erneut
- Verwenden Sie aus Konsistenzgründen führende Nullen

### Gute ADRs schreiben
- **Context first**: Erklären Sie das Problem oder die Einschränkung vor der Entscheidung
- **Be specific**: „PostgreSQL 15+ mit asynchroner Replikation verwenden“, nicht „Eine Datenbank verwenden“
- **Document trade-offs**: Listen Sie sowohl positive als auch negative Konsequenzen auf
- **Keep concise**: Streben Sie maximal 1-2 Seiten an
- **Reference related work**: Link zu Epen, Geschichten oder anderen ADRs

### Tipps zum Kontextabschnitt
- Welches Problem oder welche Einschränkung hat diese Entscheidung ausgelöst?
- Was sind die Geschäftstreiber?
- Welche technischen Einschränkungen bestehen?
- Was haben wir schon probiert?

### Tipps zum Abschnitt „Entscheidung“.
- Geben Sie deutlich an, was ausgewählt wurde
- Erklären Sie kurz, warum diese Option gewonnen hat
- Erwähnen Sie Alternativen, die in Betracht gezogen wurden
- Geben Sie Versionsnummern oder spezifische Details an

### Tipps zum Abschnitt „Konsequenzen“.
- Listen Sie zunächst die positiven Auswirkungen auf
- Listen Sie dann Kompromisse und Risiken auf
- Seien Sie ehrlich, wenn es um Nachteile geht
- Schlagen Sie Abhilfestrategien vor
