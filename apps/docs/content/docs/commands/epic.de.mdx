# /episch

Erstellen Sie ein neues Epos mit optionalen untergeordneten Storys, um verwandte User Stories rund um ein größeres Feature oder Ziel zu organisieren.

## Schnellstart

```bash
/agileflow:epic EPIC=<EP-ID> TITLE=<text> OWNER=<id> GOAL=<text>
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `EPIC` | Ja | - | Epische Kennung (z. B. EP-0001) |
| `TITLE` | Ja | - | Epischer Titel, der die Funktion oder das Ziel beschreibt |
| `OWNER` | Ja | - | Eigentümername oder Agenten-ID (z. B. AG-UI, AG-API) |
| `GOAL` | Ja | - | Ergebnis oder Metrik, die den Erfolg dieses Epos definiert |
| `STORIES` | Nein | - | Durch Kommas getrennte Liste der Anfangsgeschichten (Format: `US-ID\|title\|owner`) |

## Beispiele

### Grundlegende epische Erstellung

```bash
/agileflow:epic EPIC=EP-0010 TITLE="User Authentication" OWNER=AG-API GOAL="Enable users to securely register and log in"
```

Erstellt eine neue epische Datei unter `docs/05-epics/EP-0010.md` mit den bereitgestellten Details und registriert sie in `docs/09-agents/status.json`.

### Episch mit ersten Geschichten

```bash
/agileflow:epic EPIC=EP-0011 TITLE="Payment Processing" OWNER=AG-API GOAL="Enable payments via Stripe" STORIES="US-0040|Stripe integration|AG-API,US-0041|Payment form|AG-UI,US-0042|Webhook handler|AG-API"
```

Erstellt das Epos und generiert automatisch erste untergeordnete Geschichten unter `docs/06-stories/EP-0011/`. Jeder story wird erstellt mit:
- Status: `ready` (bereit zur Entwicklung)
- Schätzung: `0.5d` (Standard, kann geändert werden)
- Abhängigkeiten: leer (bei Bedarf manuell aktualisiert)

## Ausgabe

Der Befehl erstellt:

1. **Epische Datei** (`docs/05-epics/EP-XXXX.md`)
   - Epische Metadaten (Titel, Besitzer, Status, Ziel)
   - Aktueller Status und Kennzahlen
   - Liste mit Kindergeschichten

2. **Story-Dateien** (eine pro Story, `docs/06-stories/EP-XXXX/US-XXXX-<slug>.md`)
   - Story-Vorlage mit Platzhaltern für Akzeptanzkriterien
   – Mit übergeordnetem Epos verknüpft
   - Bereit für eine detaillierte Definition

3. **Statusverfolgung** (`docs/09-agents/status.json`)
   - Epischer Eintrag mit Besitzer und aktuellem Status
   - Mit Epos verknüpfte Story-Einträge

4. **Agentenkommunikation** (`docs/09-agents/bus/log.jsonl`)
   - Zuweisungsnachrichten, die das erstellte Epos und die zugewiesenen Geschichten anzeigen

## Arbeitsablauf

1. **Planen Sie Ihr Epos** – Definieren Sie das Funktionsziel und die Erfolgsmetriken
2. **Erstellen Sie das Epos** – Führen Sie den Befehl `/epic` mit grundlegenden Details aus
3. **Stories hinzufügen** – Erstellen Sie sie entweder jetzt mit dem STORIES-Parameteroder verwenden Sie `/story`, um sie später einzeln hinzuzufügen
4. **Storys entwickeln** – Verwenden Sie `/story-validate`, um sicherzustellen, dass jede Story vor der Entwicklung fertig ist
5. **Sprint-Plan** – Verwenden Sie `/sprint`, um Geschichten für den bevorstehenden Sprint auszuwählen
6. **Fortschritt verfolgen** – Verwenden Sie `/board`, um den epischen Abschluss zu überwachen

## Integration mit anderen Befehlen

- **Nach epischer Erstellung**: Verwenden Sie `/story`, um weitere Geschichten hinzuzufügen
- **Vor der Sprintplanung**: Verwenden Sie `/sprint`, um Geschichten aus diesem Epos auszuwählen
- **Zur Fortschrittsverfolgung**: Verwenden Sie `/board` oder `/metrics`, um den epischen Abschluss zu sehen
- **Für Abhängigkeiten**: Verwenden Sie `/deps`, um Story-Abhängigkeiten innerhalb von Epic zu visualisieren

## Best Practices

- **Epics im Fokus behalten** - Ein Feature oder Feature-Bereich pro Epic
- **Ziele messbar machen** - Bestimmte Ergebnisse oder Kennzahlen verwenden (nicht „Benutzererfahrung verbessern“)
- **Geschichten schrittweise hinzufügen** - Beginnen Sie mit Kerngeschichten und fügen Sie im Laufe der Zeit verwandte Geschichten hinzu
- **Geeignete Besitzer zuweisen** - Epic-Eigentümern zuordnenäh mit primärem Entwicklungsagenten
- **Dokumentkontext** – Technischen Kontext in der Epic-Datei nach der Erstellung hinzufügen
