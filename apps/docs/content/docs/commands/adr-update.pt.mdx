# /adr:atualização

Atualize o status de uma ADR existente ou adicione alterações que documentem as alterações de status.

## Início rápido

```bash
/agileflow:adr:update NUMBER=0042 STATUS=accepted REASON="Team approved after review"
```

## Parâmetros

| Parâmetro | Obrigatório | Padrão | Descrição |
|-----------|----------|---------|------------|
| `NUMBER` | Sim | - | Número ADR a atualizar (4 dígitos, por exemplo, 0042) |
| `STATUS` | Não | - | Novo status: `proposed`, `accepted`, `deprecated`, `superseded` |
| `REASON` | Não | - | Motivo da mudança de estado |
| `SUPERSEDED_BY` | Não | - | Em caso de substituição, o novo número ADR (por exemplo, 0045) |

## Exemplos

### Aceitar uma proposta de ADR

```bash
/agileflow:adr:update NUMBER=0042 STATUS=accepted REASON="Team approved after security review"
```

### Descontinuar um ADR

```bash
/agileflow:adr:update NUMBER=0035 STATUS=deprecated REASON="Context changed, approach is no longer applicable"
```

Sem STATUS ou REASON, o comando orienta você durante o processo de atualização.

## Transições de status

### Transições válidas

| De | Para | Razão Comum |
|------|-----|---------------|
| `proposed` | `accepted` | Decisão aprovada pela equipe |
| `proposed` | `rejected` | Alternativa escolhida, não viável |
| `accepted` | `deprecated` | Não é mais relevante, o contexto mudou |
| `accepted` | `superseded` | Encontrada uma melhor abordagem, substituída por um novo ADR |
| `deprecated` | `accepted` | Reconsiderado, ainda válido |
| `superseded` | `accepted` | Substituição não deu certo |

## Fluxo de trabalho

Ao executar o comando:

1. **Valida** a existência de ADR
2. **Solicita novo status** (se não for fornecido)
3. **Requer um motivo** para qualquer mudança de status
4. **Mostra uma prévia** do que será atualizado
5. **Peça confirmação** antes de escrever
6. **Anexa alteração** documentando a alteração
7. **Atualiza o frontmatter** com novo status e carimbo de data/hora

## Formato de alteração

Quando um status de ADR muda, o comando anexa uma seção de Emenda:

## Melhores práticas

### Quando atualizar o status do ADR

- **Aprovar** ADRs propostas após análise da equipe
- **Descontinuar** se o contexto ou os requisitos mudarem
- **Substituir** quando um ADR mais recente substituir esta abordagem
- **Reativar** se as circunstâncias mudarem e a decisão antiga for relevante novamente

### Escrevendo bons motivos

**Boas razões:**
- "Consenso da equipe após revisão da arquitetura"
- "Nova performancerequisitos CE tornam esta abordagem ineficiente"
- "O requisito de negócio removeu a restrição que desencadeou esta decisão"
- "Implementado e validado com sucesso em produção"

**Motivos ruins:**
- "Não é mais necessário"
- "Existe uma abordagem melhor"
- "Equipe decidiu"

### Documentando a Supersessão

Ao substituir um ADR:
1. Sempre inclua __NOTRASLATE_36__ com o novo número ADR
2. Explique por que a nova abordagem é melhor
3. Consulte o novo ADR para obter detalhes de implementação
4. Mantenha o ADR original intacto para o histórico
