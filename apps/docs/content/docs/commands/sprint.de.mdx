# /sprint

Erstellen Sie datengesteuerte Sprintpläne mithilfe historischer Geschwindigkeit, Agentenkapazitätsanalyse und intelligenter Story-Auswahl basierend auf Abhängigkeiten und Prioritäten.

## Schnellstart

```bash
/agileflow:sprint [SPRINT=<id>] [DURATION=<days>] [MODE=suggest|commit]
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `SPRINT` | Nein | Automatisch generiert | Sprint-ID (z. B. „Sprint-42“, „2025-W43“) |
| `DURATION` | Nein | `10` | Sprintdauer in Tagen |
| `AGENTS` | Nein | Alle aktiv | Durch Kommas getrennte Agenten-IDs, die geplant werden sollen (z. B. `AG-UI,AG-API`) |
| `MODE` | Nein | `suggest` | `suggest` zur Vorschau, `commit` zum Speichern des Plans in status.json |
| `FOCUS_EPIC` | Nein | - | Konzentrieren Sie den Sprint optional auf ein bestimmtes Epos (z. B. `EP-0010`) |

## Beispiele

### Sprint-Vorschlag anzeigen

```bash
/agileflow:sprint
```

Analysiert den aktuellen Rückstand, die Kapazität und die Geschwindigkeit. Zeigt empfohlene Storys an, ohne Änderungen vorzunehmen.

### Plan Spezifischer Sprint

```bash
/agileflow:sprint SPRINT=Sprint-42 DURATION=10 MODE=suggest
```

Plant einen 10-tägigen Sprint und zeigt eine Vorschau der empfohlenen Storys, Risiken und Kapazitätsanalysen an.

### Sprint-Plan festlegen

```bash
/agileflow:sprint SPRINT=Sprint-42 DURATION=10 MODE=commit
```

Übergibt den Sprintplan an `docs/09-agents/status.json` und erstellt einen Meilenstein in `docs/08-project/milestones.md`.

### Konzentrieren Sie sich auf ein bestimmtes Epos

```bash
/agileflow:sprint FOCUS_EPIC=EP-0010 AGENTS=AG-API MODE=suggest
```

Der Sprint der Pläne konzentriert sich auf das Authentifizierungsepos und priorisiert Storys aus diesem Epos für den AG-API-Agenten.

## Ausgabe

Der Sprintplanbericht enthält:

### Kapazitätsanalyse
- **Historische Geschwindigkeit**: In den letzten 30 Tagen pro Agent abgeschlossene Tage
- **Geplante Kapazität**: Geschätzte Kapazität für diese Sprintdauer
- **Verfügbare Slots**: Pro Agent verfügbare WIP-Positionen
- **Risikofaktoren**: In der Prüfung befindliche Meldungen, die möglicherweise Kapazitäten freigeben

### Rückstandsstatus
- Fertige Geschichten zählen
- Blockierte Storys (können während des Sprints entsperrt werden)
- Geeignete Geschichten (Abhängigkeiten behoben)

### Empfohlenes Sprint-Engagement
Priorisierte Liste der Geschichten, die festgeschrieben werden sollen:
```text
✅ US-0042 (1d) - AG-UI - Epic: EP-0010
   "User login form with validation"
   Dependencies: None ✅

✅ US-0043 (0.5d) - AG-API - Epic: EP-0010
   "POST /auth/login endpoint"
   Risk: Blocks US-0042 → Schedule first ⚠️
```

### Aufgeschobene Geschichten
Geschichten, die ckann abgeholt werden, wenn Kapazität frei wird:
```text
⚠️ US-0055 (1d) - AG-API - Epic: EP-0013
   "GET /user/settings endpoint"
   → Move to next sprint or pick up if capacity frees
```

### Sprintziele
Hochrangige Ziele basierend auf epischer Verteilung und Prioritäten.

### Risiken und Abhängigkeiten
- Agentenübergreifende Koordination erforderlich
– Abhängigkeitsketten, die sequenziert werden müssen
- Hochwirksame Blocker
- Geschichten, die möglicherweise früher abgeschlossen werden

### Empfehlungen
- Vorgeschlagener Handlungsablauf
- Schwerpunktbereiche der Sprintüberwachung
- Tägliche Standup-Themen
- Wann eine Neubewertung und Anpassung erforderlich ist

## Datenquellen

Der Befehl analysiert:

| Quelle | Zweck |
|--------|---------|
| `docs/09-agents/status.json` | Aktueller Story-Status, WIP, Aufgaben |
| `docs/09-agents/bus/log.jsonl` | Historische Geschwindigkeit aus abgeschlossenen Geschichten |
| `docs/08-project/backlog.md` | Prioritätsreihenfolge für die Story-Auswahl |
| `docs/08-project/roadmap.md` | Strategische Prioritäten |
| `docs/08-project/milestones.md` | Fristbeschränkungen |
| `docs/05-epics/*.md` | Epische Ziele und Kontext |
| `docs/06-stories/**/*.md` | Story-Details, AC, Schätzungen |

## Planungsalgorithmus

### Phase 1: Kapazität analysieren
- Zählen Sie laufende und überprüfte Storys pro Agent
-Verfügbare Slots berechnen (maximal 2 pro Agent)
- Beachten Sie, welche in der Überprüfung befindlichen Artikel möglicherweise bald abgeschlossen werden

### Phase 2: Geschwindigkeit berechnen
- Historische Abschlüsse von bus/log.jsonl analysieren (letzte 30 Tage)
- Summieren Sie Story-Schätzungen, um die Gesamtgeschwindigkeit zu erhalten
- Berechnen Sie die Geschwindigkeit pro Agent
- Projektkapazität für die Sprintdauer

### Phase 3: Geschichten auswählen
- Filtern Sie fertige Geschichten (Definition von „Ready met“)
- Abhängigkeiten validieren (alle Deps müssen den Status „erledigt“ haben)
- Priorisieren Sie nach Rückstandsreihenfolge und epischem Fokus
- Wählen Sie, bis die Kapazität erreicht ist
- Balance zwischen Agenten

### Phase 4: Risiken bewerten
- Identifizieren Sie kritische Pfadgeschichten
- Suchen Sie nach hochwirksamen Blockern
- Markieren Sie den agentenübergreifenden Koordinierungsbedarf
- Finden Sie lange Abhängigkeitsketten

### Phase 5: Commit (wenn MODE=Commit)
– Aktualisieren Sie status.json mit Sprint-Metadaten
- Meilenstein in Milestones.md erstellen
- Protokollieren Sie die Sprint-Planungsentscheidung in bus/log.jsonl

## Arbeitsablauf

1. **Storys vorbereiten** – Stellen Sie sicher, dass die Storys bereit sind (verwenden Sie `/story-validate`)
2. **Kapazität analysieren** – Führen Sie __NOTRANSLATE_ aus31__, um die Empfehlung zu sehen
3. **Risiken überprüfen** – Überprüfen Sie agentenübergreifende Abhängigkeiten und Blocker
4. **Bei Bedarf anpassen** – Geschichten hinzufügen/entfernen oder Reihenfolge anpassen
5. **Plan festschreiben** – Führen Sie zum Speichern `/sprint MODE=commit` aus
6. **Sprint starten** – Beginnen Sie mit der Story mit der höchsten Priorität
7. **Fortschritt überwachen** – Verwenden Sie `/board` und `/blockers` während des Sprints

## Auswahlkriterien für Geschichten

Die Geschichten werden in dieser Prioritätsreihenfolge ausgewählt:

1. **Muss den Status „Bereit“ haben** – Definition von „Bereit“ erfüllt
2. **Abhängigkeiten behoben** – Alle Deps haben den Status „erledigt“
3. **Backlog-Priorität** – Reihenfolge angegeben in backlog.md
4. **Epische Ausrichtung** – Fokus auf episch, falls angegeben
5. **Meilensteinfristen** – Dringende Elemente von Milestones.md
6. **Teamkapazität** – Überschreiten Sie nicht die berechnete Geschwindigkeit
7. **Agentengleichgewicht** – Verteilen Sie die Arbeit gleichmäßig

## WIP-Limits

- **Maximal 2 Storys pro Agent in Bearbeitung**
– Die Überprüfung der Geschichten wird möglicherweise bald abgeschlossen und es werden Kapazitäten frei
- Das Paaren bei Storys mit hohem Risiko zählt als 2 Slots

## Allgemeines Nutzungsmusterrns

### Start des Sprints
```bash
/agileflow:sprint MODE=suggest    # See recommendation
/agileflow:sprint MODE=commit     # Commit when ready
/agileflow:assign STORY=US-0042   # Start with first story
```

### Anpassung während des Sprints
```bash
/agileflow:sprint MODE=suggest    # Reassess remaining capacity
/agileflow:assign STORY=US-0055   # Pick up deferred story if space
```

### Ende des Sprints
```bash
/agileflow:velocity              # Update historical velocity
/agileflow:sprint                # Plan next sprint
```

## Best Practices

- **Auf Geschwindigkeit basierender Plan** - Die historische Geschwindigkeit ist ein besserer Prädiktor als Schätzungen
- **Respektieren Sie die WIP-Grenzwerte** - Überschreiten Sie nicht 2 pro Agent (beschränkt den Kontextwechsel)
- **Abhängigkeiten validieren** – Blockierte Storys werden nicht abgeschlossen
- **Mit Bedacht ordnen** - Backend vor Frontend, wenn Abhängigkeiten bestehen
- **Risiken überwachen** - Hochwirksame Blocker benötigen tägliche Aufmerksamkeit
- **Seien Sie konservativ** - Planen Sie eine Kapazität von 70–80 % ein und sparen Sie Puffer für Probleme
- **Mitten im Sprint anpassen** – Wenn Aufgaben früher abgeschlossen werden, fügen Sie verzögerte Storys hinzu

## Integration mit anderen Befehlen

- **Vor dem Sprint**: Verwenden Sie `/deps`, um Abhängigkeitsrisiken zu verstehen
- **Bevor Sie beginnen**: Verwenden Sie `/story-validate` für alle Storys
- **Während des Sprints**: Verwenden Sie `/board` für das tägliche Aufstehen
- **Bei Blockierung**: Verwenden Sie `/blockers`, um aufzutauchen und aufzulösen
- **Am Ende**: Verwenden Sie `/velocity`, um h zu aktualisierenhistorische Daten
- **Nächster Sprint**: Verwenden Sie `/sprint` erneut mit aktualisierter Geschwindigkeit
