# /قصة

قم بإنشاء قصة مستخدم جديدة بمعايير القبول، ونقاط الاختبار، وتتبع الحالة.

## بداية سريعة

__لا تترجم_0__

## المعلمات

| المعلمة | مطلوب | الافتراضي | الوصف |
|-----------|----------|---------|-------------|
| __غير مترجم_5__ | نعم | - | المعرّف الملحمي الأصلي (على سبيل المثال، EP-0001) |
| __غير مترجم_6__ | نعم | - | معرف القصة (على سبيل المثال، US-0001) |
| __غير مترجم_7__ | نعم | - | عنوان القصة الذي يصف الميزة |
| __غير مترجم_8__ | نعم | - | اسم المالك أو معرف الوكيل (على سبيل المثال، AG-UI، AG-API) |
| __غير مترجم_9__ | لا | __غير مترجم_10__ | تقدير الوقت (على سبيل المثال، `0.5d`، `1d`، `2d`) |
| __غير مترجم_14__ | لا | - | معايير القبول بتنسيق معين/متى/ثم |
| __غير مترجم_15__ | لا | - | قائمة مفصولة بفواصل لمعرفات القصص التابعة (على سبيل المثال، `US-0001,US-0002`) |

## أمثلة

### إنشاء القصة الأساسية

__لا تترجم_1__

ينشئ ملف قصة في `docs/06-stories/EP-0010/US-0001-user-registration-form.md` بالتقدير الافتراضي (0.5d) والحالة (جاهز).

### ستورذ مع معايير القبول

__لا تترجم_2__

ينشئ القصة بمعايير قبول محددة توجه التطوير والاختبار.

### قصة مع التبعيات

__لا تترجم_3__

يقوم بإنشاء القصة ووضع علامة عليها على أنها تعتمد على القصص السابقة. يتم تصور هذه التبعيات في الأمر `/deps`.

## الإخراج

يقوم الأمر بإنشاء:

1. **ملف القصة** (`docs/06-stories/EP-XXXX/US-XXXX-<slug>.md`)
   - البيانات الوصفية للقصة (المعرف، الملحمة، المالك، التقدير، الحالة)
   - الوصف ومعايير القبول
   - سياق الهندسة المعمارية (يتم ملؤها أثناء التطوير)
   - استراتيجية الاختبار

2. **قاعدة الاختبار** (`docs/07-testing/test-cases/US-XXXX.md`)
   - هيكل الاختبار متوافق مع معايير القبول
   - العنصر النائب لتنفيذ الاختبار
   - روابط للقصة للسياق

3. **تتبع الحالة** (`docs/09-agents/status.json`)
   - إدخال القصة مع المالك والتقدير والحالة
   - روابط التبعية لقصص أخرى
   - مهمة ملحمية

4. **اتصال الوكيل** (`docs/09-agents/bus/log.jsonl`)
   - رسالة مهمة توضح القصة التي تم إنشاؤها وتعيينها

## العملفمنخفض

1. **إنشاء القصة** - تشغيل `/story` مع الملحمة والمعرف والعنوان والمالك
2. **أضف معايير القبول** - استخدم `/story-validate` لتحسين واختبار AC
3. **التحقق من الاكتمال** - قم بتشغيل `/story-validate STORY=<US-ID>` قبل التطوير
4. **طور القصة** - ينفذ المالك وفقًا لمعايير القبول
5. **تشغيل الاختبارات** - استخدم `/verify` لتشغيل الاختبارات التلقائية
6. **طلب المراجعة** - قم بتحديث الحالة إلى `in-review` عندما تصبح جاهزًا

## تنسيق معايير القبول

استخدم تنسيق معين/متى/ثم للحصول على معايير واضحة وقابلة للاختبار:

__لا تترجم_4__

يجب أن تكون معايير القبول:
- **محدد**: مرجع للحقول والقيم الفعلية
- **قابل للاختبار**: يمكن التحقق منه من خلال الاختبارات الآلية أو اليدوية
- **مستقل**: يختبر كل معيار شيئًا واحدًا
- **كامل**: تغطية المسار السعيد وحالات الحافة الرئيسية

## التكامل مع الأوامر الأخرى

- **بعد إنشاء القصة**: استخدم `/story-validate` لضمان الاستعداد
- **قبل التطوير**: استخدم `/babysit` للحصول على مساعدة تفاعلية في التنفيذ
- **أثناء التطوير**: استخدم `/verify` لإجراء الاختبارات
- **لمراجعة الكود**: استخدم `/review` للاقتراحات المدعومة بالذكاء الاصطناعي
- **لطلبات السحب**: استخدم `/pr` لإنشاء وصف العلاقات العامة تلقائيًا

## أفضل الممارسات

- **قصة واحدة = ميزة واحدة** - حافظ على تركيز القصص وإمكانية تسليمها بشكل مستقل
- **اكتب معايير قبول واضحة** - وهي بمثابة دليل للتطوير والاختبار
- **قدِّر بشكل واقعي** - ضع في اعتبارك التعقيد وخبرة الفريق
- **تبعيات الارتباط بشكل صريح** - يساعد في التخطيط السريع وتقييم المخاطرsment
- **التحقق قبل التطوير** - استخدم `/story-validate` لاكتشاف المشكلات مبكرًا
