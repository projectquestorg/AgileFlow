# /تلقائي

قم بإنشاء قصص المستخدم تلقائيًا من عناصر المنتج مثل PRDs أو النماذج بالحجم الطبيعي أو مواصفات واجهة برمجة التطبيقات (API) أو مستندات التصميم. يوفر الوقت في تقسيم المتطلبات إلى قصص قابلة للتنفيذ.

## بداية سريعة

__لا تترجم_0__

|-----------|----------|---------|-------------|
| __غير مترجم_10__ | نعم | - | المسار إلى الملف أو عنوان URL (PRD، نموذج بالحجم الطبيعي، مواصفات API، مخطط التدفق) |
| __غير مترجم_11__ | لا | تم إنشاؤها تلقائيًا | ربط القصص بالملحمة الموجودة (على سبيل المثال، `EP-0010`) |
| __غير مترجم_13__ | لا | تم الكشف عنه تلقائيًا | المالك الافتراضي للقصص التي تم إنشاؤها (على سبيل المثال، `AG-API`) |
| __غير مترجم_15__ | لا | __غير مترجم_16__ | `yes` للإنشاء فورًا، `no` للمعاينة أولاً |

## أمثلة

### معاينة القصص من PRD

__لا تترجم_1__

يقرأ مستند متطلبات المصادقة وينشئ معاينة القصة دون إنشاء ملفات.

### إنشاء من مواصفات API

__لا تترجم_2__

خارجيتعمل نقاط النهاية من مواصفات OpenAPI/Swagger وتولد قصة واحدة لكل نقطة نهاية.

### إنشاء من نموذج Figma

__لا تترجم_3__

يعد التعرف الضوئي على الحروف (OCR) بمثابة تصميم بالحجم الطبيعي لتحديد الشاشات والنماذج والتفاعلات، ثم يقوم بإنشاء قصص واجهة المستخدم.

### إنشاء وإنشاء على الفور

__لا تترجم_4__

يُنشئ القصص وينشئ جميع الملفات على الفور (تخطي خطوة المعاينة).

## أنواع المصادر المدعومة

### وثيقة متطلبات المنتج (PRD)

**التنسيقات**: `.md`، `.docx`، `.pdf`، `.txt`

**مقتطفات**:
- أوصاف الميزة
- شخصيات المستخدم وحالات الاستخدام
- معايير النجاح والمقاييس
- حالات الحافة وسيناريوهات الخطأ
- المتطلبات غير الوظيفية

**مثال**:
__لا تترجم_5__

**القصص التي تم إنشاؤها**: 4-6 قصص تغطي التسجيل وتسجيل الدخول وإعادة تعيين كلمة المرور وOAuth وإدارة الجلسة.

### وثائق واجهة برمجة التطبيقات

**التنسيقات**: OpenAPI/Swagger JSON، مستندات API `.md`

**مقتطفات**:
- نقاط النهاية (طريقة HTTP + المسار)
- مخططات الطلب/الاستجابة
- أوثينتيكامتطلبات نشوئها
- رموز الخطأ وحالات الحافة
- تحديد المعدل والحصص

**مثال لـ OpenAPI**:
__لا تترجم_6__

**القصص التي تم إنشاؤها**: قصة واحدة لكل نقطة نهاية مع معايير القبول من المخطط.

### نماذج وتصميمات واجهة المستخدم

**التنسيقات**: عنوان URL للشكل، `.png`/`.jpg` (مع التعرف الضوئي على الحروف)، مواصفات التصميم `.md`

**مقتطفات**:
- تخطيطات الشاشة ومكوناتها
- تفاعلات المستخدم والتدفقات
- حقول النموذج والتحقق من صحتها
- الملاحة والروابط
- استخدام نظام التصميم

**مثال**: لقطة شاشة لنموذج تسجيل الدخول تنشئ قصصًا لمكون النموذج والتحقق من الصحة ورسائل الخطأ ورابط إعادة تعيين كلمة المرور.

### مخططات تدفق المستخدم

**التنسيقات**: حورية البحر `.mmd`، PlantUML، مخططات التدفق (`.png` مع التعرف الضوئي على الحروف)

**مقتطفات**:
- خطوات في رحلة المستخدم
- نقاط القرار وفروعه
- مسارات بديلة
- سيناريوهات النجاح والفشل

**مثال**: يُنشئ مخطط تدفق المصادقة قصصًا لكل خطوة.

## عملية إنشاء القصة

### الخطوة 1: تحليل المصدر
- وثيقة التعريفاكتب
- استخراج المتطلبات والميزات
- المتطلبات المتعلقة بالمجموعة
- تحليل المواصفات الفنية

### الخطوة 2: ميزات المجموعة
- المتطلبات ذات الصلة تصبح قصة واحدة
- تجنب ازدواجية القصة
- التأكد من استقلالية القصص
- التحقق من معايير الاستثمار

### الخطوة 3: إنشاء القصص
- إنشاء عنوان من وصف الميزة
- كتابة معايير القبول بتنسيق معين/متى/ثم
- تقدير الجهد (0.5د، 1د، 1.5د، 2د)
- تعيين المالك المناسب
- تحديد التبعيات

### الخطوة 4: عرض المعاينة
- عرض جميع القصص التي تم إنشاؤها
- عرض التقديرات وأصحابها
- إظهار عدد القصص والجهد الإجمالي
- اطلب الموافقة قبل الإنشاء

### الخطوة 5: إنشاء الملفات (إذا تمت الموافقة عليها)
- إنشاء ملحمة (إذا لم يتم توفيرها)
- إنشاء ملفات القصة مع القوالب
- إنشاء بذرة اختبار
- تحديث الحالة.json
- إلحاق بالحافلة/log.jsonl

## قواعد إنشاء القصة (استثمار)

تتبع القصص التي تم إنشاؤها معايير الاستثمار:

| المعيار | الوصف |
|-----------|------------|
| **مستقلة** | يمكن أن تكتمل دون قصص أخرى |
| **قابل للتفاوض** | يمكن تعديل التفاصيل بعد الإنشاء |
| **قيمة** | يسلم قيمة تواجه المستخدم أو الفريق |
| **مقدر** | يمكن الحجم (0.5d-2d) |
| **صغير** | كاملة في سباق واحد |
| **قابل للاختبار** | لديه معايير قبول واضحة |

القصص > 2d مقسمة إلى قصص أصغر.

## ملفات الإخراج

يقوم الأمر بإنشاء:

1. **ملف ملحمي** (`docs/05-epics/EP-XXXX.md`)
   -عنوان ملحمي والهدف
   - مرجع الوثيقة المصدر
   - قائمة قصص الأطفال

2. **ملفات القصة** (`docs/06-stories/EP-XXXX/US-XXXX.md`)
   - المحتوى الذي تم إنشاؤه من المصدر
   - معايير القبول
   - ملاحظات فنية
   - استراتيجية الاختبار

3. **بذرة الاختبار** (`docs/07-testing/test-cases/US-XXXX.md`)
   - هيكل الاختبار متوافق مع التيار المتردد
   - العناصر النائبة لتنفيذ الاختبار

4. **تتبع الحالة** (`docs/09-agents/status.json`)
   - تمت إضافة جميع القصص بالحالة: `ready`

5. **الاتصالات** (`docs/09-agents/bus/log.jsonl`)
   - تم تسجيل حدث إنشاء القصة

## سير العمل

1. **قم بإعداد المستند المصدر** - تأكد من اكتمال PRD/المواصفات
2. **إنشاء معاينة** - تشغيل `/auto` باستخدام `AUTO_CREATE=no`
3. **مراجعة القصص التي تم إنشاؤها** - التحقق من الدقة والاكتمال
4. **اضبط إذا لزم الأمر** - قم بتحرير المعاينة أو إعادة الإنشاء باستخدام معلمات مختلفة
5. **إنشاء قصص** - يمكنك تشغيلها باستخدام `AUTO_CREATE=yes` أو الموافقة على المعاينة
6. **التحقق من صحة القصص** - استخدم `/story-validate` في القصص التي تم إنشاؤها
7. **خطة السباق** - قم بتضمين القصص التي تم التحقق من صحتها__لا تترجم_44__

## السيناريوهات الشائعة

### السيناريو: تحتوي واجهة برمجة التطبيقات (API) على 15 نقطة نهاية

**المشكلة**: عدد كبير جدًا من القصص (ينتهك قاعدة التقدير 0.5d-2d)

**الحل**: يقوم الأمر تلقائيًا بتجميع نقاط النهاية ذات الصلة:
- نقاط النهاية لنفس المورد (GET، POST، PATCH، DELETE) → قصة واحدة
- قائمة وتفصيل نقاط النهاية → قصة واحدة
- نقاط نهاية المصادقة → قصة واحدة

**النتيجة**: 15 نقطة نهاية ← 5-6 قصص

### السيناريو: لدى PRD متطلبات غير واضحة

**المشكلة**: تبدو القصص التي تم إنشاؤها غامضة

**الحل**:
1. قم بتوضيح PRD مع أصحاب المصلحة أولاً
2. أعد تشغيل `/auto` بالمصدر المحدث
3. أو قم بتحرير القصص التي تم إنشاؤها يدويًا باستخدام `/story`

### السيناريو: القصص التي تم إنشاؤها لا تتطابق مع سرعة الفريق

**المشكلة**: تبدو التقديرات متفائلة/متشائمة للغاية

**الحل**:
1. قم بمراجعة القصص وضبط التقديرات
2. النظر في السرعة التاريخية للفريق
3. قم بتشغيل `/velocity` لفهم قدرة الفريق
4. قسم القصص الكبيرة إلى قصص أصغر

## قائمة التحقق من الجودة

قبل الموافقة على المخزن الذي تم إنشاؤههو:

- [ ] جميع القصص لها معايير قبول واضحة
- [ ] القصص مستقلة (لا توجد تبعيات مخفية)
- [ ] التقديرات واقعية (نطاق 0.5d-2d)
- [ ] تستخدم معايير القبول التنسيق المحدد/متى/ثم
- [ ] الملاحظات الفنية دقيقة
- [ ] تغطي القصص السيناريوهات الرئيسية (المسار السعيد + حالات الحافة)
- [ ] لا توجد قصص مكررة
- [ ] الهدف الملحمي واضح ومحدد
- [ ] تعيينات المالك مناسبة

## التكامل مع الأوامر الأخرى

- **بعد الإنشاء**: استخدم `/story-validate` لضمان الجودة
- **قبل السباق**: استخدم `/sprint` لتحديد القصص
- **بالنسبة للتبعيات**: استخدم `/deps` إذا كانت القصص بها علاقات
- **أثناء التطوير**: استخدم `/babysit` للمساعدة في التنفيذ
- **للاختبار**: استخدم `/verify` لإجراء الاختبارات

## أفضل الممارسات

- **ابدأ بالمتطلبات** - استثمر الوقت في توضيح PRD أو المواصفات
- **مراجعة القصص التي تم إنشاؤها** - لا تقبل إنشاء الذكاء الاصطناعي بشكل أعمى
- **التحقق قبل السباق** - استخدم `/story-validate` على جميع أجهزة gقصص ولدت
- **المتطلبات المتعلقة بالمجموعة** - ملحمة أكثر تركيزًا = قصص أفضل
- **كن محددًا في المصدر** - متطلبات غامضة → قصص غامضة
- **ضع في اعتبارك التبعيات** - مراجعة يدوية للتبعيات المخفية
- **تقدير متحفظ** - أضف مخزنًا مؤقتًا للمجهولين

## القيود والاعتبارات

- **قد يسيء الذكاء الاصطناعي تفسيرها** - قم بمراجعة القصص التي تم إنشاؤها للتأكد من دقتها
- **السياق مهم** - قد تفتقر القصص التي تم إنشاؤها إلى المعرفة بالمجال
- **التبعيات يدوية** - أضف التبعيات عبر القصص يدويًا
- **تختلف التقديرات حسب الفريق** - اضبطها بناءً على سرعتك
- **التفاصيل الفنية** - قد تحتاج إلى إضافات يدوية إلى سياق الهندسة المعمارية
