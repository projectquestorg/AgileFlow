# /validate-expertise

Validieren Sie die Expertenwissensdateien der Agenten, um sicherzustellen, dass sie nicht von der Codebasis abweichen und aktuell bleiben.

## Schnellstart

```bash
/agileflow:validate-expertise
```

Validiert alle Expertise-Dateien. Sie können auch eine bestimmte Domäne validieren:

```bash
/agileflow:validate-expertise DOMAIN=database
```

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `DOMAIN` | Nein | (alle) | Spezifische Domänenexpertisedatei validieren (z. B. `database`, `api`) |

## Validierungsprüfungen

Der Befehl führt vier kritische Validierungsprüfungen für jede Expertise-Datei durch:

### 1. Dateipfaddrift
Überprüft, ob alle Dateipfade, auf die in der Expertise-Datei verwiesen wird, noch in der Codebasis vorhanden sind. Erkennt, wenn dokumentierte Dateien verschoben, umbenannt oder gelöscht wurden.

**Beispiel**: Wenn Expertise.yaml auf `src/old/database.ts` verweist, diese Datei jedoch nicht mehr existiert, wird dies als Drift gekennzeichnet.

### 2. Veraltete Erkenntnisse
Kennzeichnet Expertise-Dateien, die seit 30 oder mehr Tagen nicht aktualisiert wurden. Fachwissen ist das nichtRegelmäßige Wartung wird veraltet und weniger zuverlässig.

**Warnung**: Dateien, die seit mehr als 30 Tagen nicht aktualisiert wurden, werden zur Überprüfung und Aktualisierung markiert.

### 3. Dateigrößenprüfung
Warnt, wenn eine Gutachtendatei mehr als 200 Zeilen umfasst. Übergroße Dateien sind schwer zu verwalten und sollten in kleinere, fokussierte Dokumente aufgeteilt werden.

**Empfehlung**: Halten Sie Fachwissensdateien zur besseren Wartbarkeit auf weniger als 200 Zeilen.

### 4. Erforderliche Abschnitte
Überprüft, ob die Datei „expertise.yaml“ alle erforderlichen Abschnitte enthält:
- `domain`: Domain-ID
- `mental_models`: Mindestens ein mentales Modell (Kernkonzepte)
- `learnings`: Lernverlauf (kann zunächst leer sein)
- `key_files`: Wichtige Dateien für diese Domain
- `patterns`: Gemeinsame Muster und Anti-Muster

## Beispiele

### Validieren Sie alle Expertise-Dateien

```bash
/agileflow:validate-expertise
```

Führt eine Validierung für alle Expertendateien im Verzeichnis `.agileflow/experts/` durch.

### Spezifische Domäne validieren

```bash
/agileflow:validate-expertise DOMAIN=database
```

Validiert nur die Datenbankdomänen-Expertisedatei.

###Überprüfen Sie eine andere Domain

```bash
/agileflow:validate-expertise DOMAIN=api
```

Validiert nur die API-Domänen-Expertisedatei.

## Ausgabeformat

Der Befehl erstellt einen detaillierten Validierungsbericht:

```text
======================================
EXPERTISE VALIDATION REPORT
======================================

Validating: database
  [PASS] File exists: expertise.yaml
  [PASS] Required sections present
  [WARN] 2 stale learnings (>30 days)
  [FAIL] DRIFT: src/old/database.ts not found

Validating: api
  [PASS] All checks passed

Validating: frontend
  [WARN] File size: 287 lines (recommend <200)

--------------------------------------
SUMMARY
--------------------------------------
Total domains: 25
Passed: 23
Warnings: 1
Failures: 1

RECOMMENDED ACTIONS:
1. Update database expertise - remove reference to src/old/database.ts
2. Split frontend expertise - reduce from 287 to <200 lines
3. Review stale learnings in database domain
```

## Statusanzeigen

| Indicator | Bedeutung | Aktion |
|-----------|---------|--------|
| `[PASS]` | Prüfung erfolgreich bestanden | Keine Aktion erforderlich |
| `[WARN]` | Unkritisches Problem gefunden | Überprüfen und erwägen Sie eine Korrektur |
| `[FAIL]` | Kritisches Problem gefunden | Muss behoben werden |

## Häufige Probleme und Lösungen

| Problem | Indikator | Lösung |
|-------|-----------|----------|
| Referenzierte Datei existiert nicht | `[FAIL] DRIFT` | Dateipfadreferenz | entfernen oder aktualisieren
| Datei seit mehr als 30 Tagen nicht aktualisiert | `[WARN] stale` | Überprüfen und aktualisieren Sie die Erkenntnisse |
| Gutachtendatei zu groß | `[WARN] File size` | In fokussierte Dateien aufteilen |
| Fehlende erforderliche Abschnitte | `[FAIL] Required sections` | Fehlende Abschnitte zu „expertise.yaml“ hinzufügen |

## Wann ausgeführt werden soll

Führen Sie eine Fachwissensvalidierung durch:
- **Nach Sitzungsbeginn**: Optional als Teil von Systemzustandsprüfungen (kann als Hook konfiguriert werden)
- **Vor Hauptversionen**: Stellen Sie sicher, dass alle Fachkenntnisse aktuell und korrekt sind
- **Nach großen Refaktorierungen**: Überprüfen Sie, ob Dateiverweise weiterhin auf gültige Speicherorte verweisen
- **Periodisch**: Monatlich/vierteljährlich ausführen, um die Aktualität des Fachwissens aufrechtzuerhalten
- **Beim Ändern von Fachkenntnissen**: Validieren Sie die Änderungen vor dem Festschreiben

## Speicherort der Expertise-Datei

Die Gutachtendateien werden gespeichert unter:
```text
.agileflow/experts/{domain}/expertise.yaml
```

Zum Beispiel:
- `.agileflow/experts/database/expertise.yaml`
- `.agileflow/experts/api/expertise.yaml`
- `.agileflow/experts/frontend/expertise.yaml`

## Erfolgskriterien

Eine gesunde Fachwissensdatei sollte:
- ✅ Alle referenzierten Dateien sind in der Codebasis vorhanden
- ✅ Innerhalb der letzten 30 Tage aktualisiert
- ✅ Unter 200 Zeilen
- ✅ Enthält alle erforderlichen Abschnitte (Domäne, mentale_Modelle, Erkenntnisse, Schlüsseldateien, Muster)

## Integration in Ihren Workflow

You can configure expertise validation to run automatically:

```bash
/agileflow:configure --enable=expertise-validation
```

Dadurch wird ein SessionStart-Hook hinzugefügt, der das Fachwissen bei jedem Sitzungsstart validiert und Sie auf Probleme aufmerksam macht.

## Drift Erkennungsdetails

Dateipfaddrift ist das kritischste erkannte Problem. Der Befehl sucht nach Dateiverweisen im Format:
- `src/...` – Quellcodedateien
– `packages/...` – Monorepo-Pakete
- `lib/...` – Bibliotheksdateien

Wenn diese Pfade nicht mehr vorhanden sind, werden sie zur sofortigen Bearbeitung und Entfernung markiert.
