# /garder les enfants

Un mentor interactif qui vous guide tout au long de la mise en œuvre de fonctionnalités allant des exigences en anglais simple aux demandes d'extraction complétées. Babysit combine une collecte de contexte automatisée, des suggestions de tâches intelligentes et une assistance pratique au code.

## Démarrage rapide

```bash
/agileflow:babysit
```

La commande va :
1. Rassembler tout le contexte du projet (histoires, épopées, tests, documentation)
2. Présenter des suggestions de tâches intelligentes basées sur la priorité et l'état de préparation
3. Vous guider pas à pas dans la mise en œuvre
4. Mettez à jour le statut de votre projet pendant que vous travaillez
5. Générez des descriptions de relations publiques lorsque vous êtes prêt

## Principales fonctionnalités

### Suggestions de tâches intelligentes

Quand tu commences `/babysit`, il analyse votre projet et recommande :

- **READY stories** (tous les climatiseurs sont terminés, tests annulés) - Priorité la plus élevée
- **Blocked stories** (avec des chemins de déblocage clairs) - Haute priorité
- **Near-complete epics** (80 %+ d'histoires terminées) - Terminez ce que vous avez commencé
- **Roadmap priorities** - Travail stratégique important
- **Research gaps** - Travail de préparation pour éviter le débogage plus tard

Les tâches sont classées en fonction de ce qu'un vrai développeur prioriserait, et pas seulement par récence ou par ordre.

### Conseils pratiques de mise en œuvre

Pour chaque histoire ou long métrage :

```text
1. Validate story readiness (acceptance criteria, tests)
2. Read architecture context relevant to the task
3. Plan implementation in small steps
4. Apply code changes incrementally with diffs
5. Run tests after each change
6. Update project status as you progress
7. Generate PR description when complete
```

### Apparition d'experts en agents

Pour les travaux complexes, babysit détecte automatiquement le domaine et fait apparaître des spécialistes :

- **Database work** → Expert AG-BASE DE DONNÉES
- **API endpoints** → Expert AG-API
- **UI components** → Expert AG-UI
- **Testing** → Expert AG-TESTING
- **Documentation** → Expert AG-DOCUMENTATION

Les spécialistes gèrent un travail approfondi et ciblé tandis que le baby-sitter coordonne le flux global.

### Suivi des progrès

Tout le travail est suivi avec une liste de tâches :

```text
1. Run context script (loads all project state)
2. Present task suggestions
3. Read task-specific documentation
4. Validate story readiness
5. Plan implementation steps
6. Apply code changes with tests
7. Update project status
8. Generate PR description
9. Mark complete
```

Vous pouvez voir exactement où vous êtes et ce qui reste.

## Usage

### Basique – Laisser le babysitting suggérer du travail

```bash
/agileflow:babysit
```

Babysit analysera votre projet et vous demandera sur quoi vous aimeriez travailler.

### Reprendre le travail existant

```bash
/agileflow:babysit
```

Si vous avez des histoires dans `in_progress` statut, babysit vous proposera de les reprendre.

### Travailler sur une histoire spécifique

Vous pouvez dire à babysit « Je veux travailler sur US-0042 » et cela :
- Charger le fichier d'histoire
- Validez c'est prêt
- Planifier la mise en œuvre
- Vous guider dans sa construction

## Paramètres

Babysit accepte la saisie libre en langage naturel :

| Input | Effect |
|-------|--------|
| `"Add user profile page"` | Create new feature or find matching story |
| `"US-0042"` | Work on specific story |
| `"Continue"` | Resume the last story you were working on |
| `"What should I work on?"` | Get recommendations (default) |

## Mode boucle autonome

Babysit prend en charge l'exécution autonome grâce à **Loop Mode**, qui traite automatiquement les histoires entières d'une épopée.

### Démarrage rapide

```bash
/agileflow:babysit EPIC=EP-0042 MODE=loop MAX=20
```

### Comment ça marche

```text
1. Initialize loop for epic
2. Load first "ready" story
3. Work on implementation
4. When Claude stops → Stop hook runs tests
   - If tests PASS → mark story complete, load next
   - If tests FAIL → show errors, continue fixing
5. Repeat until epic complete or max iterations
```

### Paramètres

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `EPIC` | Yes | - | Epic ID to process (e.g., EP-0042) |
| `MODE` | Yes | - | Must be `loop` for autonomous mode |
| `MAX` | No | 20 | Maximum iterations (safety limit) |

### Contrôle de boucle

```bash
# Check loop status
node scripts/ralph-loop.js --status

# Stop the loop manually
node scripts/ralph-loop.js --stop

# Reset loop state
node scripts/ralph-loop.js --reset
```

### Quand utiliser le mode boucle

**Good for:**
- Traiter une épopée bien définie avec des histoires claires
- Développement piloté par les tests (les tests définissent « terminé »)
- Traitement par lots de plusieurs étages

**Not good for:**
- Travaux exploratoires sans critères d’acceptation clairs
- Histoires nécessitant un examen humain avant de continuer
- Travail multi-domaines complexe nécessitant une coordination

Pour une documentation complète sur le mode boucle, y compris le mode visuel et le mode couverture, consultez le [Guide des fonctionnalités du mode boucle](/features/loop-mode).

### Exemple de séance

```text
/agileflow:babysit EPIC=EP-0042 MODE=loop MAX=20

→ Ralph Loop Initialized
  Epic: EP-0042
  Stories: 5 ready, 7 total
  Starting: US-0015 "Add user authentication"

[Claude implements US-0015...]

→ Stop Hook: Running tests...
  ✓ Tests pass
  ✓ Marked US-0015 complete
  → Moving to US-0016

[Claude implements US-0016...]

→ Stop Hook: Running tests...
  ✗ 2 tests failing
  → Fix errors and continue

[Claude fixes tests...]

→ Stop Hook: Running tests...
  ✓ Tests pass
  ✓ Marked US-0016 complete
  → Moving to US-0017

[...continues until epic complete...]

→ EPIC COMPLETE!
  EP-0042 finished in 12 iterations
  5 stories completed
```

## Présentation du flux de travail

```text
Start → Gather Context → Suggest Tasks → Choose Task
  ↓
Read Story & Architecture → Validate Readiness → Plan Steps
  ↓
Implement Code → Run Tests → Update Status
  ↓
Generate PR → Finish
```

## Définition de prêt

Avant la mise en œuvre, babysit vérifie que les histoires ont :

- ✓ Critères d'acceptation (format Donné/Quand/Alors)
- ✓ Contexte d'architecture avec citations de sources
- ✓ Talons de test dans docs/07-testing/
- ✓ Dépendances résolues et documentées
- ✓ Aperçus précédents d'histoires connexes

S'il en manque, babysit vous proposera de les compléter en premier (les histoires restent « en brouillon » jusqu'à ce qu'elles soient prêtes).

## Chargement du contexte

Lorsque vous démarrez babysit, il exécute automatiquement ce script contextuel :

```bash
node .agileflow/scripts/obtain-context.js babysit
```

Cette unique commande regroupe :
- Statut Git, branche, modifications non validées
- Toutes les histoires et épopées de status.json
- État de la session et sessions actives
- Structure documentaire
- Notes de recherche (les plus récentes en premier)
- Messages récents des agents
- Présence du fichier clé

Un appel de script, un contexte complet, un démarrage rapide.

## Quand utiliser

### Idéal pour

- **Starting implementation** - Besoin de conseils sur les points à travailler
- **Picking up mid-feature** - Reprenez là où vous vous êtes arrêté
- **Epic completion** - Terminer les histoires finales dans un ensemble de fonctionnalités
- **Uncertain requirements** - Questions de recherche avant de coder
- **Multi-step features** - Travail complexe sur plusieurs fichiers
- **Testing validation** - S'assurer que les tests réussissent avant les relations publiques

### Pas pour

- **Quick fixes** (fautes de frappe, one-liners) - Faites-les vous-même
- **Pure research** - Utiliser `/agileflow:research:ask` plutôt
- **Architectural decisions** - Utiliser `/agileflow:adr` plutôt
- **Documentation only** - Utiliser `/agileflow:docs` plutôt

## Style de sortie

Babysit utilise des résultats clairs et exploitables :

- **Headings** pour les grandes sections
- **Short bullets** pour les options et les explications
- **Code blocks** pour les différences et les exemples
- **Questions** pour guider vos décisions (ne vous demande jamais de « taper »)
- **Progress updates** pendant que tu travailles

Chaque réponse se termine par une question sur l'étape suivante afin que vous ayez toujours le contrôle.

## Conseils

### Obtenez de meilleures suggestions

- Conservez le backlog de votre histoire dans status.json
- Rédiger des critères d'acceptation clairs
- Marquer les histoires comme prêtes une fois terminées
- Maintenir les documents d'architecture à jour

### Implémentations réussies

- Laissez le babysitting valider d'abord la préparation de l'histoire
- Lire la section contexte de l'architecture
- Mettre en œuvre par petites étapes (pas toutes en même temps)
- Exécutez des tests après chaque modification
- Demandez de l'aide si vous êtes bloqué

### Orchestration multi-experts

Pour les fonctionnalités complexes, babysit va automatiquement :
- Expert API Spawn pour le travail backend
- Expert en interface utilisateur Spawn pour le travail frontend
- Exécuter les deux en parallèle s'ils sont indépendants
- Coordonner l’intégration finale

Vous n'avez pas besoin de gérer manuellement plusieurs agents : babysit s'en charge.
