# /العنوان:عرض

عرض تفاصيل ADR الكاملة واتخاذ إجراءات مدركة للسياق بناءً على الحالة الحالية لـ ADR.

## بداية سريعة

```bash
/agileflow:adr:view NUMBER=0042
```

## حدود

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `NUMBER` | Yes | - | ADR number (4-digit, e.g., 0042) |

## أمثلة

### عرض ADR محدد

```bash
/agileflow:adr:view NUMBER=0003
```

يعرض المحتوى الكامل لـ ADR-0003 ويقدم إجراءات تراعي السياق.

### الاختيار التفاعلي

```bash
/agileflow:adr:view
```

بدون معلمة NUMBER، يعرض ADRs الأخيرة ويطلب منك تحديد واحد.

## تنسيق الإخراج

تتضمن كل طريقة عرض ADR ما يلي:

### معلومات الرأس

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture
```

### الأقسام

1. **Context**: يشرح سبب الحاجة إلى هذا القرار
   - المشكلة أو القيد الذي أدى إلى اتخاذ القرار
   - محركات الأعمال والمتطلبات الفنية
   - الخيارات البديلة التي تم النظر فيها

2. **Decision**: ما تم اختياره ولماذا
   - بيان واضح للقرار
   - الأساس المنطقي لهذا الاختيار على البدائل
   - إصدارات أو تكوينات محددة إذا كانت ذات صلة

3. **Consequences**:المقايضات وآثارها
   - **Positive**: الفوائد والتحسينات
   - **Negative**: التكاليف والمخاطر ومنحنى التعلم
   - استراتيجيات التخفيف إن أمكن

4. **Related**: روابط للأبحاث والقصص والإعلانات الأخرى
   - الأبحاث التي أدت إلى هذا القرار
   - قصص التنفيذ التي تم إنشاؤها من ADR هذا
   - التفاعلات الدوائية الأخرى التي تحل محلها أو تحل محلها

## الإجراءات المدركة للسياق

تتغير الإجراءات بناءً على حالة ADR:

### بالنسبة للتفاعلات الدوائية المقترحة

عندما ADR لا يزال `proposed`، أنت تستطيع:
- **Accept this ADR**: وضع علامة باسم `accepted` وفي المفعول
- **Request changes**: تعديل الاقتراح
- **Reject this ADR**: وضع علامة باسم `rejected` مع السبب

### بالنسبة إلى ADRs المقبولة (مستحسن)

عندما يكون ADR `accepted`، أنت تستطيع:
- **Create implementation stories**: تحويل القرار إلى عمل قابل للتنفيذ
- **Supersede with new ADR**: استبدل بالقرار المحدث
- **Deprecate this ADR**: ضع علامة على أنها لم تعد قابلة للتطبيق
- **View related research**: انظر الأبحاث التي أدت إلى اتخاذ القرار

### بالنسبة إلى ADRs المهملة

عندما يكون ADR `deprecated`، أنت تستطيع:
- **View deprecation history**: انظر لماذا تم إهماله
- **Reactivate this ADR**: وضع علامة باسم `accepted` مرة أخرى إذا تغير السياق
- **View replacement ADR**: انظر ما الذي حل محل هذا
- **Back to ADR list**: العودة إلى القائمة الكاملة

### بالنسبة للتفاعلات الدوائية البديلة

عندما يكون ADR `superseded`، أنت تستطيع:
- **View replacement ADR**: انظر ADR الذي حل محل هذا
- **View decision evolution**: فهم كيف تطور هذا القرار
- **Back to ADR list**: العودة إلى القائمة الكاملة

## سير العمل النموذجي

### الموافقة على القرار المقترح

```
1. /agileflow:adr:view NUMBER=0045
   (See proposed architecture decision)
2. Select "Accept this ADR"
3. ADR status changes to accepted
4. Select "Create implementation stories"
5. Stories created in docs/06-stories/
```

### تنفيذ القرار المقبول

```
1. /agileflow:adr:view NUMBER=0042
   (See accepted architecture decision)
2. Select "Create implementation stories"
3. Stories created for implementing this decision
4. Work on stories with /agileflow:babysit
```

### تتبع تطور القرار

```
1. /agileflow:adr:view NUMBER=0010
   (Old approach)
2. See "Superseded by ADR-0035"
3. Select "View replacement ADR"
4. See how decision evolved over time
```

### إعادة النظر في القرار الموقوف

```
1. /agileflow:adr:view NUMBER=0015
   (Deprecated approach)
2. Check why it was deprecated
3. If context changed: "Reactivate this ADR"
4. Mark as accepted again
```

## مثال على عرض ADR

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture

---

### Context

We need a reliable database with ACID guarantees for user data and transactions.
The application handles financial transactions requiring strong consistency.
Current SQLite implementation cannot scale to multiple servers.

---

### Decision

We will use PostgreSQL as our primary database because:
- Mature, well-tested ACID compliance
- Excellent tooling and ecosystem
- Team has existing experience
- Supports advanced features (JSONB, full-text search)

---

### Consequences

**Positive:**
- Strong data integrity guarantees
- Better performance at scale
- Rich query capabilities
- Team familiarity reduces learning curve

**Negative:**
- Requires managed database service or self-hosting
- Team needs PostgreSQL-specific training
- Migration effort from SQLite
- Slightly higher operational complexity than NoSQL

---

### Related

- [Research: Database Selection](../10-research/20241225-database-selection.md)
- [US-0050: Database migration story](../06-stories/EP-0005/US-0050.md)
- Supersedes: ADR-0010 (Use SQLite)
```

## أفضل الممارسات

### عند مراجعة ADR

- اقرأ السياق أولاً لفهم المشكلة
- مراجعة العواقب الإيجابية والسلبية
- تحقق من القسم ذو الصلة لأعمال التنفيذ
- انظر إلى تطور القرار (إذا تم استبداله)

### عند اتخاذ قرار بشأن التنفيذ

اطرح هذه الأسئلة:
1. هل هذا ADR في `accepted` حالة؟
2. هل العواقب الإيجابية تتناسب مع احتياجاتنا الحالية؟
3. هل نحن مستعدون للعواقب السلبية؟
4. هل لدينا القدرة على التنفيذ؟

### عندما تتغير حالة ADR

استخدم دائما `/agileflow:adr:update` لتوثيق:
- لماذا يتغير الوضع
- ما الذي أدى إلى التغيير
- إشارة إلى بديل ADR (إذا تم استبداله)
