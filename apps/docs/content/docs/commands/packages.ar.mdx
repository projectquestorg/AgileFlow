## الغرض

يوفر هذا الأمر إدارة التبعية الكاملة عن طريق:
- الكشف عن تبعيات المشروع عبر npm والنقطة والبضائع ومديري الحزم الآخرين
- إنشاء لوحات معلومات تبعية شاملة تحتوي على معلومات الحالة والضعف
- إجراء عمليات التدقيق الأمني وتحديد مكافحة التطرف العنيف
- تخطيط وتطبيق تحديثات التبعية الآمنة
- إنشاء قصص للقضايا الأمنية
- التتبع والإبلاغ عن صحة الصيانة

## الإجراءات

### ACTION=لوحة المعلومات (افتراضية)

يُنشئ تقرير تبعية شامل يُظهر:
- **العدد الإجمالي** - تبعيات الإنتاج والتطوير
- **حالة الضعف** - حرجة، عالية، متوسطة، منخفضة
- **توفر التحديث** - إصدارات التصحيح الرئيسية والثانوية
- **معلومات الترخيص** - لتتبع الامتثال
- **النتيجة الصحية** - صحة صيانة التبعية بشكل عام

**مثال الإخراج**:
__لا تترجم_7__

### الإجراء = التحديث

يخطط ويطبق تحديثات التبعية مع فحوصات السلامة:

1. ** يكتشف الحزم القديمة ** باستخدام مدير الحزم المناسب
2. **التصنيفات حسب النوع** - الأمان، الرئيسي، الثانوي، التصحيح
3. **يظهر خطة التحديث** مع تحذيرات التغيير
4. **يطلب التأكيد** قبل التقديم
5. **إجراء الاختبارات** بعد التحديثاتيس (إذا كان متاحا)
6. **إنشاء علاقات عامة** إذا طلب ذلك
7. **يُنشئ تقريرًا** بكل التغييرات

** مدراء الحزم المدعومة **:
- Node.js: npm، الغزل، pnpm
- بايثون: النقطة، أدوات النقطة، الشعر
- روبي: المجمع (Gemfile)
- انطلق : انطلق
- الصدأ: البضائع
- جافا: مخضرم، جرادل
- .NET: نوجيت

**يتم دائمًا إعطاء الأولوية للتحديثات الأمنية** ويوصى بها على الفور.

** التحديثات الرئيسية ** تنشئ قصصًا فردية للمراجعة (قد تتطلب تغييرات في الكود).

**التحديثات الثانوية/التصحيحية** يمكن تجميعها بشكل آمن.

### الإجراء=التدقيق

تشغيل التدقيق الأمني فقط دون تحديث:

يعرض جميع نقاط الضعف المعروفة مع:
- مستويات الخطورة (حرجة، عالية، متوسطة، منخفضة)
- معرفات وأوصاف مكافحة التطرف العنيف
- إصدارات الحزمة المتأثرة
- الإصلاحات الموصى بها

مثال:
__لا تترجم_8__

## الكشف عن مدير الحزم

يكتشف الأمر ويدعم تلقائيًا:

| مدير | ملفات | كشف |
|---------|-------|-----------|
| نبم | package.json، package-lock.json | مشاريع Node.js |
| غزل | غزل.قفل | Node.js الحديثة |
| بنم | بنبم-لock.yaml | Node.js الحديثة |
| نقطة | المتطلبات.txt، pyproject.toml | بايثون |
| شعر | قفل الشعر | بايثون الحديثة |
| المجمع | Gemfile، Gemfile.lock | روبي / ريلز |
| اذهب إلى الوضع | go.mod، go.sum | اذهب |
| بضائع | Cargo.toml، Cargo.lock | الصدأ |
| مافن | بوم.xml | جافا |
| جرادل | بناء.gradle | جافا |
| نوجيت | *.csproj، packages.config | صافي |

## أفضل الممارسات

1. **إعطاء الأولوية للأمان** - قم دائمًا بتحديث نقاط الضعف الحرجة على الفور

2. **التحديثات الثانوية المجمعة** - تحديثات جماعية غير منقسمة لتحقيق الكفاءة

3. **الاختبار بعد التحديثات** - قم دائمًا بتشغيل مجموعة الاختبار الكاملة بعد التحديثات

4. **مراجعة التغييرات الرئيسية** - اختبر الإصدارات الرئيسية في المرحلة قبل الإنتاج

5. **مراقبة حالات الإيقاف** - التعامل مع الحزم المهملة مبكرًا

6. **الامتثال للترخيص** - التحقق من التراخيص وفقًا لمتطلبات الشركة

7. **حجم الحزمة** - حزمة الشاشةحجم تأثير التبعيات

8. **عمليات التدقيق التلقائية** - قم بإجراء عمليات التدقيق أسبوعيًا، وليس فقط أثناء الإصدارات

## القواعد

- لا تقم مطلقًا بالتحديث التلقائي للحزم الهامة دون الحصول على موافقة
- قم دائمًا بمعاينة التغييرات قبل تطبيقها (الفرق أولاً، نعم/لا)
- تحديثات المجموعة الثانوية/التصحيحية عندما يكون ذلك ممكنًا
- التحذير بشكل بارز من كسر التغييرات
- رابط لسجلات التغيير وأدلة الهجرة
- إنشاء قصص للترقيات الرئيسية
- لا تخطي الاختبارات أبدًا بعد التحديث
- الالتزام دائمًا بملفات القفل (package-lock.json، Cargo.lock، وما إلى ذلك)
