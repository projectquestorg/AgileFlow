# /adr:ver

Vea todos los detalles del ADR y tome acciones contextuales basadas en el estado actual del ADR.

## Inicio rápido

```bash
/agileflow:adr:view NUMBER=0042
```

## Parámetros

| Parámetro | Requerido | Predeterminado | Descripción |
|-----------|----------|---------|-------------|
| `NUMBER` | Sí | - | Número ADR (4 dígitos, por ejemplo, 0042) |

## Ejemplos

### Ver ADR específico

```bash
/agileflow:adr:view NUMBER=0003
```

Muestra el contenido completo de ADR-0003 y ofrece acciones contextuales.

### Selección interactiva

```bash
/agileflow:adr:view
```

Sin el parámetro NÚMERO, muestra los ADR recientes y le pide que seleccione uno.

## Formato de salida

Cada vista de ADR incluye:

### Información del encabezado

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture
```

### Secciones

1. **Contexto**: Explica por qué era necesaria esta decisión.
   - Problema o limitación que desencadenó la decisión.
   - Impulsores comerciales y requisitos técnicos.
   - Opciones alternativas que se consideraron

2. **Decisión**: Qué se eligió y por qué
   - Declaración clara de la decisión.
   - Justificación de esta elección sobre alternativas- Versiones o configuraciones específicas si son relevantes.

3. **Consecuencias**: compensaciones e impactos
   - **Positivo**: Beneficios y mejoras
   - **Negativo**: Costos, riesgos, curva de aprendizaje
   - Estrategias de mitigación si corresponde

4. **Relacionado**: enlaces a investigaciones, historias y otras ADR
   - Investigación que llevó a esta decisión.
   - Historias de implementación creadas a partir de este ADR
   - Otros ADR que este reemplaza o es reemplazado por

## Acciones sensibles al contexto

Las acciones cambian según el estado del ADR:

### Para ADR propuestos

Cuando un ADR todavía es `proposed`, puedes:
- **Aceptar este ADR**: Marcar como `accepted` y en vigor
- **Solicitar cambios**: Modificar la propuesta
- **Rechazar este ADR**: marcar como `rejected` con motivo

### Para ADR aceptados (recomendado)

Cuando un ADR es `accepted`, puedes:
- **Crear historias de implementación**: convertir las decisiones en trabajo práctico
- **Reemplazar con un nuevo ADR**: Reemplazar con una decisión actualizada
- **Desaprobar este ADR**: marcar como ya no aplicableble
- **Ver investigación relacionada**: vea la investigación que condujo a la decisión

### Para ADR obsoletos

Cuando un ADR es `deprecated`, puedes:
- **Ver historial de obsolescencia**: vea por qué quedó obsoleto
- **Reactivar este ADR**: marcar como `accepted` nuevamente si el contexto cambia
- **Ver ADR de reemplazo**: vea lo que reemplazó a este
- **Volver a la lista ADR**: Volver a la lista completa

### Para ADR reemplazados

Cuando un ADR es `superseded`, puedes:
- **Ver ADR de reemplazo**: vea el ADR que reemplazó a este
- **Ver evolución de la decisión**: comprenda cómo evolucionó esta decisión
- **Volver a la lista ADR**: Volver a la lista completa

## Flujos de trabajo típicos

### Aprobar la decisión propuesta

```
1. /agileflow:adr:view NUMBER=0045
   (See proposed architecture decision)
2. Select "Accept this ADR"
3. ADR status changes to accepted
4. Select "Create implementation stories"
5. Stories created in docs/06-stories/
```

### Implementar decisión aceptada

```
1. /agileflow:adr:view NUMBER=0042
   (See accepted architecture decision)
2. Select "Create implementation stories"
3. Stories created for implementing this decision
4. Work on stories with /agileflow:babysit
```

### Seguimiento de la evolución de las decisiones

```
1. /agileflow:adr:view NUMBER=0010
   (Old approach)
2. See "Superseded by ADR-0035"
3. Select "View replacement ADR"
4. See how decision evolved over time
```

### Reconsiderar decisión obsoleta

```
1. /agileflow:adr:view NUMBER=0015
   (Deprecated approach)
2. Check why it was deprecated
3. If context changed: "Reactivate this ADR"
4. Mark as accepted again
```

## Ejemplo de visualización de ADR

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture

---

### Context

We need a reliable database with ACID guarantees for user data and transactions.
The application handles financial transactions requiring strong consistency.
Current SQLite implementation cannot scale to multiple servers.

---

### Decision

We will use PostgreSQL as our primary database because:
- Mature, well-tested ACID compliance
- Excellent tooling and ecosystem
- Team has existing experience
- Supports advanced features (JSONB, full-text search)

---

### Consequences

**Positive:**
- Strong data integrity guarantees
- Better performance at scale
- Rich query capabilities
- Team familiarity reduces learning curve

**Negative:**
- Requires managed database service or self-hosting
- Team needs PostgreSQL-specific training
- Migration effort from SQLite
- Slightly higher operational complexity than NoSQL

---

### Related

- [Research: Database Selection](../10-research/20241225-database-selection.md)
- [US-0050: Database migration story](../06-stories/EP-0005/US-0050.md)
- Supersedes: ADR-0010 (Use SQLite)
```

## Mejores prácticas

### Al revisar un ADR

- Lea primero el contexto para comprender el problema.
- Revisar ambas posturas.Consecuencias negativas y negativas
- Consulte la sección relacionada para conocer el trabajo de implementación.
- Mirar la evolución de la decisión (si es reemplazada)

### Al decidir si implementar

Haga estas preguntas:
1. ¿Este ADR está en estado `accepted`?
2. ¿Las consecuencias positivas coinciden con nuestras necesidades actuales?
3. ¿Estamos preparados para las consecuencias negativas?
4. ¿Tenemos capacidad de implementación?

### Cuando cambia el estado del ADR

Utilice siempre `/agileflow:adr:update` para documentar:
- Por qué el estado está cambiando
- ¿Qué provocó el cambio?
- Referencia al ADR de sustitución (si se reemplaza)
