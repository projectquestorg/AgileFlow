# /generate-changelog

Generieren Sie automatisch ein Änderungsprotokoll aus Ihrem Commit-Verlauf und Pull-Requests mithilfe herkömmlicher Commits und dem Keep a Changelog-Format.

## Schnellstart

```bash
/agileflow:generate-changelog VERSION=2.5.0 SINCE=v2.4.0
```

## Zweck

Dieser Befehl erstellt professionelle Änderungsprotokolle durch:
- Automatische Versionserkennung anhand der neuesten Git-Tags
- Sammeln von Commits und zusammengeführten PRs seit der letzten Veröffentlichung
- Kategorisieren von Änderungen (hinzugefügt/geändert/behoben/entfernt/Sicherheit)
- Hervorhebung bahnbrechender Änderungen
- Vorschlagen semantischer Versionen basierend auf Änderungen
– Aktualisierung von `CHANGELOG.md` mit der richtigen Formatierung
- Optional Erstellung von Git-Tags und GitHub-Releases

## Parameter

| Parameter | Erforderlich | Standard | Beschreibung |
|-----------|----------|---------|-------------|
| `VERSION` | Nein | automatische Erkennung | Versionsnummer (z. B. `2.5.0`) |
| `SINCE` | Nein | letztes Tag | Ausgangspunkt (z. B. `v2.4.0`) |
| `FORMAT` | Nein | `keepachangelog` | Formatstil: `keepachangelog`, `simple`, `github` |
| `AUTO_COMMIT` | Nein | `no` | Änderungsprotokollaktualisierungen automatisch festschreiben: `yes` oder `no` |

## Beispiele

### Alles automatisch erkennen

```bash
/agileflow:generate-changelog
```

Verwendet das neueste Tag als Versionsstart, erkennt die nächste Version und generiert ein Änderungsprotokoll.

### Geben Sie genaue Versionen an

```bash
/agileflow:generate-changelog VERSION=2.5.0 SINCE=v2.4.0
```

Erstellt einen Änderungsprotokollabschnitt für Version 2.5.0 aus Commits seit Version 2.4.0.

### GitHub-Format verwenden

```bash
/agileflow:generate-changelog FORMAT=github
```

Erstellt ein Änderungsprotokoll im GitHub-Versionshinweisformat.

### Änderungen automatisch festschreiben

```bash
/agileflow:generate-changelog AUTO_COMMIT=yes
```

Aktualisiert CHANGELOG.md und führt nach Ihrer Bestätigung automatisch einen Commit durch.

## Wie es funktioniert

Der Befehl folgt einem intelligenten Workflow:

### 1. Versionserkennung

Wenn `VERSION` nicht angegeben wird, erkennt es anhand von Git:
```bash
git describe --tags --abbrev=0  # Gets latest tag
```

Dann schlägt die nächste Version basierend auf den Änderungen vor:
- **Major (2.0.0)**: Breaking Changes erkannt
- **Minor (1.X.0)**: Nur neue Funktionen (feat:).
- **Patch (1.1.X)**: Nur Fehlerbehebungen (fix:).

### 2. Sammlung festschreiben

Sammelt Commits seite `SINCE` Punkt:
```bash
git log <SINCE>..HEAD --oneline --no-merges
```

Ruft außerdem zusammengeführte PRs von GitHub ab:
```bash
gh pr list --state merged --json number,title,mergedAt,labels
```

### 3. Kategorisierung

Kategorisiert automatisch mithilfe herkömmlicher Commits:

| Präfix | Kategorie | Beispiel |
|--------|----------|---------|
| `feat:` | **Hinzugefügt** | Neue Funktion X für Benutzer |
| `fix:` | **Behoben** | Fehler, bei dem E F | verursachte
| `perf:` | **Geändert** | Leistungssteigerung um 40 % |
| `refactor:` | **Geändert** | Umstrukturierte API-Schicht |
| `security:` | **Sicherheit** | Gepatchte XSS-Schwachstelle |
| `feat!:` oder `BREAKING CHANGE:` | **Geändert** (mit Warnung) | ⚠️ **BREAKING** Neu gestaltete API |

### 4. Changelog-Erstellung

Erstellt einen Eintrag im Format „Keep a Changelog“:

```markdown
## [2.5.0] - 2025-12-22

### Added
- OAuth2 authentication support (#123)
- Dark mode toggle (#137)

### Changed
- ⚠️ **BREAKING**: Redesigned API endpoints (#126)
- Improved query performance by 50% (#125)

### Fixed
- Crash when user is null (#124)
- Memory leak in WebSocket handler (#139)

### Security
- Patched XSS vulnerability in comment rendering (#131)
```

### 5. Überprüfung und Genehmigung durch den Benutzer

Zeigt eine Diff-Vorschau an und bittet um Bestätigung, bevor CHANGELOG.md aktualisiert wird.

### 6. Optionale Aktionen

Bei Genehmigung:
- Aktualisierungen `CHANGELOG.md`
– Erstellt optional einen Git-Commit
- Kann Git-Tags erstellen
- Kann erstellenGitHub-Veröffentlichung

## Changelog-Format

Der Befehl verwendet den Keep a Changelog-Standard mit den folgenden Abschnitten:

### [Unveröffentlicht]
Noch nicht veröffentlichte Entwicklungsänderungen (bleibt immer oben)

### [VERSION] – DATUM
Abschnitte für jede Veröffentlichung:

- **Hinzugefügt** – Neue Funktionalität
- **Geändert** – Änderungen an bestehenden Funktionen
- **Veraltet** – Funktionalität, die bald entfernt wird
- **Entfernt** – Funktionalität entfernt
- **Behoben** – Fehlerbehebungen
- **Sicherheit** – Behebung von Sicherheitslücken

## Breaking Changes

Breaking Changes werden erkannt und deutlich hervorgehoben:

**Erkannt von**:
- Commit mit `!` nach Typ: `feat!: redesign API`
- Commit mit der Fußzeile `BREAKING CHANGE:`
- PR gekennzeichnet mit `breaking-change`

**Angezeigt als**:
```markdown
### Changed
- ⚠️ **BREAKING**: Redesigned API endpoints. See migration guide. (#126)
```

## Ausgabedatei

Der Befehl aktualisiert `CHANGELOG.md` mit dieser Struktur:

```markdown
# Changelog

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

## [2.5.0] - 2025-12-22
### Added

## [2.4.0] - 2025-11-15

```

## Arbeitsablauf

Der komplette Workflow sieht so aus:

1. **Erkennungsergebnisse anzeigen**
   ```
   Latest tag: v2.4.0
   Commits since: 15 (3 Added, 5 Fixed, 2 Changed, 1 Security)
   Suggested version: 2.5.0 (MINOR - new features)
   ```

2. **Changelog-Vorschau anzeigen**
   ```
   ## [2.5.0] - 2025-12-22

   ### Added
   - OAuth2 authentication support (#123)
   ...
   ```

3. **Bitte um Bestätigung**
   ```
   Update CHANGELOG.md? (YES/NO)
   Auto-commit? (YES/NO)
   Create tag? (YES/NO)
   ```

4. **Änderungen übernehmen**
   - Aktualisiert CHANGELOG.md
   - Optionales Commit: `git commit -m "chore(release): update CHANGELOG for v2.5.0"`
   - Erstellt optional das Tag: `git tag -a v2.5.0`
   – Erstellt optional eine GitHub-Version

## Integration mit CI/CD

Sie können die Generierung von Änderungsprotokollen in CI automatisieren:

```yaml
name: Release

on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Version number (e.g., 1.2.0)'
        required: true

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Get all tags

      - name: Generate changelog
        run: /agileflow:generate-changelog VERSION=${{ github.event.inputs.version }}

      - name: Commit changelog
        run: |
          git config user.name "github-actions"
          git config user.email "actions@github.com"
          git add CHANGELOG.md
          git commit -m "chore(release): update CHANGELOG"

      - name: Create tag
        run: git tag -a v${{ github.event.inputs.version }} -m "Release v${{ github.event.inputs.version }}"

      - name: Push changes
        run: git push origin main && git push origin v${{ github.event.inputs.version }}

      - name: Create GitHub release
        run: gh release create v${{ github.event.inputs.version }} --generate-notes
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## Best Practices

### Nachrichtenformat festschreiben

Verwenden Sie herkömmliche Commits für die automatische Kategorisierung:

```bash
# Good - auto-categorizes as Added
git commit -m "feat: add OAuth2 authentication"

# Good - auto-categorizes as Fixed
git commit -m "fix: crash when user is null"

# Good - breaking change detection
git commit -m "feat!: redesign API endpoints"

# Better - with scope
git commit -m "feat(auth): add OAuth2 support"
```

### Semantische Versionierung

Befolgen Sie die semantische Versionierung nach Veröffentlichungen:

```text
MAJOR.MINOR.PATCH
  2  .  5  .  0

- MAJOR: Breaking changes
- MINOR: New features (backwards compatible)
- PATCH: Bug fixes only
```

### Freigabeprozess

1. **Änderungsprotokoll erstellen** für die nächste Version
2. **Git-Tag erstellen** mit Version
3. **Push-Tag**, um die CD-Pipeline auszulösen
4. **GitHub-Release erstellen** mit Änderungsprotokollauszug
5. **Veröffentlichung ankündigen** (Blog, soziale Medien, Newsletter)
