# /adr:vue

View full ADR details and take context-aware actions based on the ADR's current status.

## Démarrage rapide

```bash
/agileflow:adr:view NUMBER=0042
```

## Paramètres

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `NUMBER` | Yes | - | ADR number (4-digit, e.g., 0042) |

## Exemples

### Voir l'ADR spécifique

```bash
/agileflow:adr:view NUMBER=0003
```

Affiche le contenu complet de l'ADR-0003 et propose des actions contextuelles.

### Sélection interactive

```bash
/agileflow:adr:view
```

Sans paramètre NUMBER, affiche les ADR récents et vous demande d'en sélectionner un.

## Format de sortie

Chaque vue ADR comprend :

### Informations d'en-tête

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture
```

### Sections

1. **Context**: Explique pourquoi cette décision était nécessaire
   - Problème ou contrainte ayant déclenché la décision
   - Facteurs commerciaux et exigences techniques
   - Options alternatives envisagées

2. **Decision**: Qu'est-ce qui a été choisi et pourquoi
   - Déclaration claire de la décision
   - Justification de ce choix par rapport aux alternatives
   - Versions ou configurations spécifiques le cas échéant

3. **Consequences**: Compromis et impacts
   - **Positive**: Avantages et améliorations
   - **Negative**: Coûts, risques, courbe d’apprentissage
   - Stratégies d'atténuation, le cas échéant

4. **Related**: Liens vers des recherches, des histoires, d'autres ADR
   - Recherche qui a conduit à cette décision
   - Histoires de mise en œuvre créées à partir de cet ADR
   - Autres ADR qui remplacent ou sont remplacés par

## Actions contextuelles

Les actions changent en fonction du statut ADR :

### Pour les ADR proposés

Lorsqu'un ADR est toujours `proposed`, tu peux:
- **Accept this ADR**: Marquer comme `accepted` et en effet
- **Request changes**: Modifier la proposition
- **Reject this ADR**: Marquer comme `rejected` avec raison

### Pour les ADR acceptés (recommandé)

Lorsqu'un ADR est `accepted`, tu peux:
- **Create implementation stories**: Transformez la décision en un travail réalisable
- **Supersede with new ADR**: Remplacer par une décision mise à jour
- **Deprecate this ADR**: Marquer comme n'étant plus applicable
- **View related research**: Voir les recherches qui ont conduit à la décision

### Pour les ADR obsolètes

Lorsqu'un ADR est `deprecated`, tu peux:
- **View deprecation history**: Voyez pourquoi il était obsolète
- **Reactivate this ADR**: Marquer comme `accepted` encore une fois si le contexte change
- **View replacement ADR**: Voyez ce qui a remplacé ceci
- **Back to ADR list**: Retour à la liste complète

### Pour les ADR remplacés

Lorsqu'un ADR est `superseded`, tu peux:
- **View replacement ADR**: Voir l'ADR qui a remplacé celui-ci
- **View decision evolution**: Comprendre comment cette décision a évolué
- **Back to ADR list**: Retour à la liste complète

## Flux de travail typiques

### Approuver la décision proposée

```
1. /agileflow:adr:view NUMBER=0045
   (See proposed architecture decision)
2. Select "Accept this ADR"
3. ADR status changes to accepted
4. Select "Create implementation stories"
5. Stories created in docs/06-stories/
```

### Mettre en œuvre la décision acceptée

```
1. /agileflow:adr:view NUMBER=0042
   (See accepted architecture decision)
2. Select "Create implementation stories"
3. Stories created for implementing this decision
4. Work on stories with /agileflow:babysit
```

### Suivre l’évolution des décisions

```
1. /agileflow:adr:view NUMBER=0010
   (Old approach)
2. See "Superseded by ADR-0035"
3. Select "View replacement ADR"
4. See how decision evolved over time
```

### Reconsidérer la décision obsolète

```
1. /agileflow:adr:view NUMBER=0015
   (Deprecated approach)
2. Check why it was deprecated
3. If context changed: "Reactivate this ADR"
4. Mark as accepted again
```

## Exemple d'affichage ADR

```markdown
## ADR-0042: Use PostgreSQL for Persistence

**Status**: accepted
**Date**: 2024-12-28
**Tags**: database, architecture

---

### Context

We need a reliable database with ACID guarantees for user data and transactions.
The application handles financial transactions requiring strong consistency.
Current SQLite implementation cannot scale to multiple servers.

---

### Decision

We will use PostgreSQL as our primary database because:
- Mature, well-tested ACID compliance
- Excellent tooling and ecosystem
- Team has existing experience
- Supports advanced features (JSONB, full-text search)

---

### Consequences

**Positive:**
- Strong data integrity guarantees
- Better performance at scale
- Rich query capabilities
- Team familiarity reduces learning curve

**Negative:**
- Requires managed database service or self-hosting
- Team needs PostgreSQL-specific training
- Migration effort from SQLite
- Slightly higher operational complexity than NoSQL

---

### Related

- [Research: Database Selection](../10-research/20241225-database-selection.md)
- [US-0050: Database migration story](../06-stories/EP-0005/US-0050.md)
- Supersedes: ADR-0010 (Use SQLite)
```

## Meilleures pratiques

### Lors de l’examen d’un ADR

- Lisez d'abord le contexte pour comprendre le problème
- Examinez les conséquences positives et négatives
- Consultez la section connexe pour le travail de mise en œuvre
- Regarder l'évolution de la décision (si elle est remplacée)

### Au moment de décider de mettre en œuvre

Posez ces questions :
1. Cet ADR est-il en `accepted` statut?
2. Les conséquences positives correspondent-elles à nos besoins actuels ?
3. Sommes-nous prêts à affronter des conséquences négatives ?
4. Avons-nous une capacité de mise en œuvre ?

### Lorsque le statut ADR change

Utilisez toujours `/agileflow:adr:update` documenter :
- Pourquoi le statut change
- Qu'est-ce qui a déclenché le changement
- Référence à l'ADR de remplacement (si remplacé)
