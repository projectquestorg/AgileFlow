# Agente de documentación

El Agente de documentación (AG-DOCUMENTATION) es su especialista en redacción técnica, documentación de API, guías de usuario y en mantener la documentación actualizada con su código base. Este agente garantiza que toda la documentación se mantenga actualizada, clara y útil tanto para desarrolladores nuevos como experimentados.

## Capacidades

- **Documentación de API**: especificaciones de OpenAPI/Swagger, descripciones de puntos finales, ejemplos de solicitud/respuesta y códigos de error
- **Guías de usuario y tutoriales**: guías de introducción, instrucciones paso a paso, mejores prácticas y solución de problemas
- **Guías para desarrolladores**: configuración del proyecto, flujo de trabajo de desarrollo, descripción general de la arquitectura y patrones de extensión.
- **Administración README**: archivos README raíz, específicos del módulo y específicos de funciones
- **Mantenimiento del registro de cambios**: notas de la versión y seguimiento de cambios de cara al usuario
- **Auditorías de documentación**: identifique contenido desactualizado, enlaces rotos y ejemplos faltantes
- **Documentos generados automáticamente**: compatibilidad con OpenAPI Generator, TypeDoc/JSDocy diagramas de arquitectura

## Cuándo utilizar

Utilice este agente cuando necesite:

- Escribir o actualizar documentación API con ejemplos prácticos.
- Crear guías de usuario para nuevas funciones (instrucciones paso a paso)
- Actualizar archivos README para reflejar el estado actual del proyecto
- Escribir guías de incorporación de desarrolladores.
- Crear guías de solución de problemas y preguntas frecuentes para problemas comunes
- Decisiones de arquitectura documental y diseño de sistemas.
- Mantener el registro de cambios con notas de la versión.
- Reparar enlaces de documentación rotos
- Documentación de auditoría para verificar su obsolescencia y precisión.
- Mantenga la documentación sincronizada con los cambios de código.

## Cómo funciona

1. **Carga de contexto**: el agente lee el archivo de experiencia (patrones de documentación, ubicaciones de archivos, convenciones) y analiza la estructura del proyecto (CLAUDE.md, status.json, versiones recientes)

2. **Auditoría de documentación**: el agente identifica lagunas en la documentación, secciones obsoletas, enlaces rotos y ejemplos faltantes escaneando documentos/estructura y comparándolos con el código.

3. **Planificación**: El agente propone mejorar la documentaciónents, sugiere nuevas guías, identifica contenido obsoleto y prioriza el trabajo

4. **Redacción**: el agente crea/actualiza la documentación priorizando la claridad sobre la brevedad, incluye ejemplos prácticos (listos para copiar y pegar) y mantiene un formato coherente.

5. **Revisión de calidad**: el agente verifica que la documentación esté actualizada con el código, que todos los pasos estén explicados, que los ejemplos funcionen y que la navegación entre documentos sea clara.

6. **Coordinación**: el agente actualiza status.json, agrega mensajes de finalización y publica actualizaciones en el bus del agente.

## Ejemplo

```bash
# Via /babysit
/agileflow:babysit
> "We just released v2.17.0. Need to update the changelog and API docs for the new endpoints"
```

El Agente de Documentación:
1. Verifique los nuevos puntos finales de API agregados en v2.17.0
2. Actualice la documentación de la API con ejemplos de solicitud/respuesta
3. Agregue notas de la versión al registro de cambios destacando los cambios que enfrenta el usuario.
4. Actualice README si es necesario
5. Verifique que todos los enlaces funcionen

## Comportamientos clave

- **Prioridad de documentación**: claridad sobre brevedad: explica todo para los nuevos usuarios
- **Incluir siempre ejemplos**: la documentación sin ejemplos es inútil; todos los ejemplos son copy-pasado listo
- **Mantener actualizado**: la documentación debe mantenerse actualizada con el código; Los documentos obsoletos son peores que ningún documento.
- **Incluir solución de problemas**: soluciona problemas comunes que encontrarán los usuarios
- **Cambios importantes en documentos**: fundamental para los usuarios que actualizan versiones
- **Comprobación de enlaces**: corrige proactivamente enlaces de documentación rotos
- **Coherencia**: mantiene un formato, tono y estructura consistentes en todos los documentos.
- **Preservación del contexto**: utiliza compact_context (prioridad: alta) para mantener el enfoque durante conversaciones largas, preservando el seguimiento de la cobertura de la documentación y la validación de enlaces mediante la compactación del contexto.

## Configuración de contexto compacto

El agente de documentación utiliza compact_context de **alta prioridad** para garantizar que la calidad de la documentación se mantenga enfocada:

```yaml
compact_context:
  priority: high
  preserve_rules:
    - "LOAD EXPERTISE FIRST: Always read packages/cli/src/core/experts/documentation/expertise.yaml"
    - "NEVER LET DOCS LAG BEHIND CODE: Stale docs are worse than no docs (misinformation)"
    - "ALWAYS INCLUDE EXAMPLES: Documentation without examples is useless"
    - "KEEP LINKS CURRENT: Run link checker, fix broken references"
    - "DOCUMENT BREAKING CHANGES: Critical for users upgrading versions"
    - "CLARITY OVER BREVITY: Explain everything for new users"
  state_fields:
    - current_story
    - documentation_coverage
    - broken_links_count
    - outdated_sections
```

Esto garantiza que las reglas críticas para la documentación (mantener los documentos actualizados, incluidos ejemplos, la integridad de la documentación) y el estado actual (qué documentos están desactualizados, qué enlaces están rotos) persisten a través de la compactación del contexto.

## Lista de verificación de calidad

Antes de documentar:
- La documentación está actualizada con el código.
- Todas las características nuevas documentadas con ejemplos.
- La documentación API incluye ejemplos prácticos.
- Los enlaces no están rotos.
- El formato es consistente en todos los archivos.
- Los ejemplos funcionan y están listos para copiar y pegar.
- La sección de solución de problemas aborda problemas comunes
- La navegación entre documentos es clara.
- README es preciso y útil
- No queda ninguna información obsoleta

## Patrones comunes

**Documentación como código**
- Toda la documentación versionada con código en git
- La fuente de la verdad es el código base vivo, no documentos separados.
- Los cambios en el código activan actualizaciones de la documentación.

**Estructura de directorio numerada**
- Documentación organizada en documentos/con directorios numerados (00-meta hasta 10-research)
- Estructura consistente en todos los proyectos.

**Mantenimiento de documentación**
- Programe auditorías trimestrales para encontrar información obsoleta
- Después de cada lanzamiento, actualice la documentación.
- Después de cada cambio de API, actualice los documentos de API
- Reparar enlaces rotos de forma proactiva

## Mensajes de coordinación

El Agente de Documentación se comunica con otros agentes mediante mensajes de bus:

```jsonl
{"ts":"2025-10-21T10:00:00Z","from":"AG-DOCUMENTATION","type":"status","text":"API docs updated for v2.17.0"}
{"ts":"2025-10-21T10:05:00Z","from":"AG-DOCUMENTATION","type":"question","text":"UI component API changed - need updated documentation for props"}
```

Cuando otros agentes completan el trabajo:
- AG-API completa el punto final → El agente de documentación actualiza los documentos API
- Componente de lanzamientos AG-UI → Componente de documentos del Agente de documentación proPD y uso
- Se produce la publicación → El agente de documentación actualiza el registro de cambios y las notas de la versión
- Decisión de arquitectura tomada → El agente de documentación actualiza la documentación ADR
