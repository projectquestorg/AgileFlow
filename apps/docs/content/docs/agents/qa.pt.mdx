# Agente de controle de qualidade

O Agente de controle de qualidade (AG-QA) é um especialista em garantia de qualidade que garante que os recursos atendam aos padrões de qualidade antes do lançamento. Este agente se concentra na estratégia de teste, teste de regressão, métricas de qualidade e preparação para lançamento – complementando o Agente de Teste que projeta implementações de teste.

## Capacidades

- **Desenvolvimento de estratégia de teste**: crie planos de teste abrangentes com escopo, metas de cobertura e critérios de sucesso
- **Testes de regressão**: planeje e gerencie conjuntos de testes de regressão para versões
- **Métricas de qualidade**: Defina e rastreie KPIs (cobertura de código, taxa de escape de bugs, densidade de defeitos)
- **Prontidão para liberação**: crie e valide listas de verificação de liberação e procedimentos de aprovação
- **Gerenciamento de casos de teste**: organize e gerencie casos de teste manuais
- **Triagem de bugs**: avalie a gravidade e priorize a resolução de problemas
- **Planejamento UAT**: Coordenar testes de aceitação do usuário com as partes interessadas
- **Portões de qualidade**: defina pontos de verificação de qualidade automatizados no pipeline de CI/CD

## Quando usar

Use o agente de controle de qualidadequando:

- Planejando um novo lançamento e precisando de critérios de qualidade
- Necessidade de definir estratégia de testes com metas de cobertura
- Criação de casos de teste para fluxos de trabalho de usuários críticos
- Configurando testes de regressão para lançamentos
- Triagem de bugs e avaliação de gravidade
- Planejando testes de aceitação do usuário (UAT)
- Necessidade de estabelecer portas de qualidade e critérios de saída
- Criação de listas de verificação de preparação para lançamento

## Como funciona

1. **Carregamento de conhecimento**: o agente lê a experiência e os padrões de qualidade do projeto
2. **Análise de recursos**: o agente analisa a história para necessidades de teste
3. **Estratégia de teste**: o agente projeta um plano de teste com escopo e metas de cobertura
4. **Planejamento de teste**: o agente cria casos de teste e escopo de regressão
5. **Portões de qualidade**: o agente define métricas e limites de qualidade
6. **Prontidão para liberação**: o agente cria procedimentos de aprovação e listas de verificação
7. **Coordenação**: Agente se alinha com as equipes de teste e desenvolvimento
8. **Verificação**: O agente garante que os critérios de qualidade sejam atendidos
9. **Documentação**: Agente atualiza status.jfilho e relatórios de qualidade

## Principais comportamentos

- **Qualidade em primeiro lugar**: Nunca lance sem atender aos critérios de qualidade
- **Teste antecipado**: o teste acontece durante todo o desenvolvimento, não apenas no final
- **Baseado em risco**: áreas de maior risco recebem mais atenção nos testes
- **Direcionado por métricas**: decisões de qualidade apoiadas por dados e KPIs
- **Critérios claros**: Critérios de lançamento documentados e mensuráveis
- **Foco na regressão**: testes de regressão automatizados evitam bugs conhecidos
- **Aprovação do usuário**: o UAT valida as expectativas do usuário antes do lançamento

## Estrutura de Métricas de Qualidade

**Métricas de cobertura**:
Cobertura de código: >80% de unidade, >60% de integração, >30% E2E
- Cobertura de recursos:% de recursos testados
- Cobertura de requisitos: % de requisitos testados

**Métricas de bugs**:
- Bugs encontrados: contagem total
- Taxa de escape de bugs: % de bugs encontrados em produção vs testes (target: menos de 2%)
- Densidade de defeitos: Bugs por 1.000 linhas de código (meta: abaixo de 2,5)

**KPIs de qualidade**:
- Taxa de aprovação no teste: >95%
- Tempo médio para correção: tempo médio desde o relatório até a resolução
- Taxa de regressão: % de regressões em lançamentos

## Lista de verificação de preparação para liberação

**Deve ter antes do lançamento**:
- [] Revisão de código concluída (cobertura de 100%)
- [] Todos os testes automatizados foram aprovados
- [] Bugs críticos resolvidos
- [] Linha de base de desempenho atingida
- [] Acessibilidade verificada (WCAG AA)
- [] Revisão de segurança aprovada
- [ ] Documentação atualizada
- [] aprovação do UAT obtida

**Deve ser feito antes do lançamento**:
- [] Bugs de alta prioridade resolvidos
- [] Desempenho otimizado
- [] Teste de carga concluído
- [] Configuração de monitoramento

**É bom ter antes do lançamento**:
- [] Bugs de prioridade média resolvidos
- [] Benchmarks de desempenho publicados
- [] Comunicação do usuário redigida

## Níveis de gravidade de bugs

| Gravidade | Descrição | Exemplo | Prioridade |
|----------|------------|---------|----------|
| Crítico | Recurso inutilizável, simperda de ta, violação de segurança | Os usuários não podem fazer login | Corrija imediatamente |
| Alto | Recurso significativamente prejudicado | O pagamento falha 50% das vezes | Corrija o mais rápido possível |
| Médio | Recurso funciona com limitações | Email atrasado 2 horas | Agendar futuro próximo |
| Baixo | Problema menor, solução fácil | Texto do botão desalinhado | Pendências |

## Gerenciamento de casos de teste

**Modelo de caso de teste**:
```text
Test Case ID: TC-001
Title: User can log in with valid credentials
Feature: Authentication

Preconditions:
- User has registered account
- Login page accessible

Test Steps:
1. Navigate to login page
2. Enter email address
3. Enter password
4. Click "Sign In"

Expected Result:
- Dashboard loads in <2 seconds
- User data displayed correctly
- Welcome message shown

Status: ☐ Pass ☐ Fail ☐ Blocked
```

## Estratégia de teste de regressão

**O que fazer o teste de regressão**:
- Fluxos de trabalho principais do usuário (login, inscrição, recursos críticos)
- Recursos alterados (afetados por esta versão)
- Recursos relacionados (dependências)
- Caminhos críticos (recursos de alto uso)
- Áreas sensíveis ao desempenho
- Recursos sensíveis à segurança

**Exemplo de escopo de regressão**:
```text
Feature: User Login
├── Happy path: Valid credentials
├── Invalid credentials: Wrong password
├── Inactive account: Suspended user
├── Password reset flow
├── Login rate limiting
├── Session management
└── Related features:
    ├── Signup (account creation)
    ├── Password change
    └── Session timeout
```

## QualidadePortões

**Code Quality Gate** (antes da mesclagem):
- Revisão de código aprovada
- Linting passou
- Verificação de tipo aprovada
- Aprovação em testes unitários (>80% de cobertura)
- Sem vulnerabilidades de segurança

**Integration Quality Gate** (antes da preparação):
- Testes de integração aprovados
- Testes E2E aprovados
- Desempenho dentro da linha de base
- Migrações de banco de dados validadas
- Contratos API validados

**Lançamento do Quality Gate** (antes da produção):
- Teste de regressão aprovado
- Aprovação do UAT obtida
- Bugs críticos resolvidos
- Benchmarks de desempenho atendidos
- Revisão de segurança aprovada

## Princípios de Qualidade

- **A qualidade não é negociável**: Nunca libere abaixo dos padrões de qualidade
- **Meça tudo**: decisões de qualidade apoiadas por métricas e dados
- **Teste antecipado**: mude o teste para o desenvolvimento
- **Abordagem baseada em risco**: teste áreas de alto risco mais detalhadamente
- **Melhoria contínua**: acompanhe as métricas e melhore ao longo do tempo
