# Agente mentor

El agente Mentor (MENTOR) es su especialista en orquestación para flujos de trabajo de implementación de funciones completos. Lo guía desde las solicitudes iniciales de funciones hasta la integración de la investigación, la creación de historias, la coordinación de la implementación y la preparación de relaciones públicas.

## Capacidades

- Orquestación de funciones de un extremo a otro, desde la idea hasta las relaciones públicas fusionadas.
- Integración de la investigación de docs/10-research/
- Creación de historias y épicas cuando sea necesario.
- Coordinación multiagente y gestión de estado.
- Documentación de patrones y mantenimiento de CLAUDE.md.
- Orquestación de comandos de Slack (41 comandos disponibles)
- Definición de validación lista y llenado de brechas.
- Resolución de dependencia de agentes y gestión de WIP.

## Cuándo utilizar

Utilice el agente Mentor cuando:

- Tiene una solicitud de función en inglés sencillo para implementar
- Debes planificar la implementación en múltiples historias/épicas.
- Necesitas investigar una tecnología desconocida antes de comenzar.
- Estás coordinando el trabajo entre agentes especializados (AG-UI, AG-API, AG-CI, AG-DEVOPS)
- Necesitasd para garantizar que se cumpla la Definición de Listo antes de la implementación
- Quiere automatizar la generación y coordinación de relaciones públicas.

## Cómo funciona

1. **Carga de contexto**: el agente lee el archivo de experiencia, status.json, bus/log.jsonl y la hoja de ruta.
2. **Análisis de conocimientos**: el agente identifica los WIP actuales, los bloqueadores, las historias listas y las lagunas de investigación.
3. **Propuesta**: el agente sugiere entre 3 y 7 próximas acciones priorizadas según la hoja de ruta y las dependencias.
4. **Planificación**: el agente crea historias/épicas faltantes y garantiza que se cumpla la definición de Listo
5. **Integración de investigación**: el agente aprovecha o solicita investigación de docs/10-research/
6. **Coordinación**: El agente organiza agentes especializados y actualiza el estado/bus
7. **Documentación**: El agente actualiza CLAUDE.md con patrones recién aprendidos
8. **Finalización**: el agente genera una descripción de PR y se prepara para la fusión

## Ejemplo

```bash
# Via /babysit (recommended)
/agileflow:babysit
> "I need to add payment processing to the platform"
```

El mentor:
1. Verifique si existe una épica de pago; crear si es necesario
2. Dividase en historias (configuracion, integracion de Stripe, manejo de webhooks, etc.)
3. Investigue los patrones de pago (consulte docs/10-research/ o sugiera una nueva investigación)
4. Validar la definición de Listo para cada historia
5. Sugiera qué historia implementar primero
6. Coordinar la implementación con agentes especializados
7. Actualizar el estado a medida que avanza el trabajo

O generar directamente:

```text
Task(
  description: "Implement user authentication",
  prompt: "Add JWT-based auth with login/logout flows, profile management",
  subagent_type: "agileflow-mentor"
)
```

## Comportamientos clave

- **Ejecución autónoma de comandos**: invoca directamente comandos de barra (tablero, estado, análisis de impacto, investigación: preguntar, etc.) sin pedir permiso
- **Gestión del ciclo de vida de la historia**: valida la definición de Listo, marca en progreso, pasa a revisión, actualizaciones para finalizar
- **Cumplimiento del límite de WIP**: Respeta un máximo de 2 pisos por agente en estado en progreso
- **Seguimiento de dependencia**: identifica bloqueadores, coordina el desbloqueo entre agentes
- **Planificación basada en la investigación**: integra investigaciones existentes o solicita nuevas investigaciones a través de /agileflow:research:ask
- **Actualizaciones proactivas de CLAUDE.md**: documenta nuevos patrones/convenciones descubiertos durante la implementación
- **Coordinación entre agentes**: Utiliza bus/log.jsonl para enviar mensajes a otros agentes (AG-UI, AG-API, AG-CI, AG-DEVOPS)
- **Preservación del contexto**: utiliza compact_context (prioridad: crítica) para mantener el enfoque de orquestación durante conversaciones largas, preservando la validación de la definición de listo y el seguimiento de bloqueadores a través de la compactación del contexto.

## Configuración de contexto compacto

El agente Mentor utiliza **prioridad crítica** compact_context para garantizar que nunca se pierda el enfoque de la orquestación:

```yaml
compact_context:
  priority: critical
  preserve_rules:
    - "ALWAYS read expertise.yaml first"
    - "ALWAYS validate Definition of Ready before implementation"
    - "Max 2 stories per agent in-progress (WIP limit)"
    - "Slash commands are autonomous (invoke directly)"
    - "File operations require diff + YES/NO confirmation"
    - "Update status.json + bus/log.jsonl for all state changes"
  state_fields:
    - current_story
    - story_status
    - wip_count
    - blockers
    - next_actions
```

Esto garantiza que las reglas críticas del mentor (definición de validación de listo, límites de WIP, ejecución autónoma de comandos) y el estado de orquestación (historia actual, bloqueadores, próximas acciones priorizadas) siempre se conserven incluso después de una compactación extensa del contexto.

- **Flujo de trabajo**: `packages/cli/src/core/experts/mentor/workflow.md` (Planificar → Construir → Auto-mejorar)

- **Hoja de ruta**: `docs/08-project/roadmap.md` (prioridades y contexto)
- **Investigación**: `docs/10-research/` (notas de investigación técnica)
- **ADR**: `docs/03-decisions/` (decisiones de arquitectura)
- **CLAUDE.md**: Aviso del sistema de IA específico del proyecto

## Pasos del flujo de trabajo

1. **Cargar experiencia**: lea experticia.yaml para cargar el modelo mental
2. **Verificar estado**: lea status.json para conocer el WIP actual, los bloqueadores y las historias listas.
3. **Leer Bus**: consulte bus/log.jsonl para conocer la actividad reciente del agente.
4. **Consulte la hoja de ruta**: revise docs/08-project/roadmap.md para conocer las prioridades
5. **Proponer acciones**: Sugiera entre 3 y 7 acciones siguientes clasificadas por prioridad
6. **Plan**: dividir la función en épicas/historias con criterios de aceptación
7. **Investigación**: integrar investigaciones existentes o sugerir nuevas investigaciones
8. **Asignar**: cree historias con asignaciones de agentes y resguardos de pruebas
9. **Actualizar estado**: marcar historias como listas, actualizar status.json
10. **Bus de mensajes**: Agregar mensajes de coordinación
11. **Implementar**: Guiar la implementación a través de agentes especializados
12. **Documento**: Actualiza CLAUDE.md con nuevos patrones
13. **Generar PR**: Utilice /agileflow:pr-template para la descripción del PR
14. **Automejora**: ejecute self-improve.md una vez completado

## Coordinación de agentes

**MENTOR coordina con**:

- **AG-UI**: Historias de frontend: verifica si hay historias de UI bloqueadas en espera en la API
- **AG-API**: Historias de backend: garantiza que los puntos finales de API estén listos para desbloquear la interfaz de usuario
- **AG-CI**: Infraestructura de prueba: coordina la configuración y verificación de la prueba.
- **AG-DEVOPS**: Dependencias e implementación: coordina migraciones y actualizaciones de paquetes.
- **EPIC-PLANNER**: Descomposición de la historia: delega la planificación épica/de la historia cuando sea necesario
- **INVESTIGACIÓN**: Investigación técnica: coordina la investigación sobre tecnologías desconocidas.
