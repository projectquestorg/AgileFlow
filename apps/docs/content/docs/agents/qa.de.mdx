# QA-Agent

Der QA-Agent (AG-QA) ist ein Qualitätssicherungsspezialist, der vor der Veröffentlichung sicherstellt, dass Funktionen den Qualitätsstandards entsprechen. Dieser Agent konzentriert sich auf Teststrategie, Regressionstests, Qualitätsmetriken und Release-Bereitschaft – und ergänzt den Testing Agent, der Testimplementierungen entwirft.

## Fähigkeiten

- **Teststrategieentwicklung**: Erstellen Sie umfassende Testpläne mit Umfang, Abdeckungszielen und Erfolgskriterien
- **Regressionstests**: Planen und verwalten Sie Regressionstestsuiten für Releases
- **Qualitätsmetriken**: KPIs definieren und verfolgen (Codeabdeckung, Bug-Escape-Rate, Fehlerdichte)
- **Release-Bereitschaft**: Erstellen und validieren Sie Release-Checklisten und Freigabeverfahren
- **Testfallverwaltung**: Manuelle Testfälle organisieren und verwalten
- **Bug Triage**: Bewerten Sie den Schweregrad und priorisieren Sie die Problemlösung
- **UAT-Planung**: Koordinieren Sie Benutzerakzeptanztests mit Stakeholdern
- **Quality Gates**: Definieren Sie automatisierte Qualitätskontrollpunkte in der CI/CD-Pipeline

## Wann zu verwenden

Nutzen Sie das QA-Agentt wenn:

- Planen Sie eine neue Veröffentlichung und benötigen Sie Qualitätskriterien
- Es muss eine Teststrategie mit Abdeckungszielen definiert werden
- Erstellen von Testfällen für kritische Benutzerworkflows
- Einrichten von Regressionstests für Releases
- Fehlersuche und Schweregradbeurteilung
- Planung von Benutzerakzeptanztests (UAT)
- Es müssen Qualitätstore und Ausstiegskriterien festgelegt werden
- Erstellen von Checklisten für die Release-Bereitschaft

## Wie es funktioniert

1. **Wissen laden**: Agent liest Fachwissen und Projektqualitätsstandards
2. **Funktionsanalyse**: Agent überprüft die Story für Testanforderungen
3. **Teststrategie**: Der Agent entwirft einen Testplan mit Umfang und Abdeckungszielen
4. **Testplanung**: Agent erstellt Testfälle und Regressionsumfang
5. **Quality Gates**: Der Agent definiert Qualitätsmetriken und Schwellenwerte
6. **Freigabebereitschaft**: Agent erstellt Freigabeverfahren und Checklisten
7. **Koordination**: Agent stimmt sich mit Test- und Entwicklungsteams ab
8. **Verifizierung**: Der Agent stellt sicher, dass die Qualitätskriterien erfüllt werden
9. **Dokumentation**: Agent aktualisiert status.jSohn- und Qualitätsberichte

## Schlüsselverhalten

- **Quality First**: Veröffentlichen Sie niemals, ohne die Qualitätskriterien zu erfüllen
- **Früh testen**: Tests finden während der gesamten Entwicklung statt, nicht nur am Ende
- **Risikobasiert**: Bereiche mit höherem Risiko erhalten mehr Testaufmerksamkeit
- **Metrikgesteuert**: Qualitätsentscheidungen, die durch Daten und KPIs gestützt werden
- **Klare Kriterien**: Freigabekriterien dokumentiert und messbar
- **Regressionsfokussiert**: Automatisierte Regressionstests verhindern bekannte Fehler
- **Benutzerabmeldung**: UAT validiert die Benutzererwartungen vor der Veröffentlichung

## Qualitätsmetrik-Framework

**Abdeckungsmetriken**:
- Codeabdeckung: >80 % Einheit, >60 % Integration, >30 % E2E
- Funktionsabdeckung: % der getesteten Funktionen
- Anforderungsabdeckung: % der getesteten Anforderungen

**Fehlermetriken**:
- Gefundene Fehler: Gesamtzahl
- Fehlerfluchtrate: % der in der Produktion gefundenen Fehler im Vergleich zum Testen (taZiel: unter 2%)
- Fehlerdichte: Fehler pro 1000 Zeilen Code (Ziel: unter 2,5)

**Qualitäts-KPIs**:
- Testerfolgsquote: >95 %
- Mittlere Fehlerbehebungszeit: Durchschnittliche Zeit vom Bericht bis zur Lösung
- Regressionsrate: % der Regressionen in Veröffentlichungen

## Checkliste für die Release-Bereitschaft

**Muss vor der Veröffentlichung vorhanden sein**:
- [ ] Codeüberprüfung abgeschlossen (100 % Abdeckung)
- [ ] Alle automatisierten Tests bestanden
- [ ] Kritische Fehler behoben
- [ ] Leistungsgrundlinie erreicht
- [ ] Barrierefreiheit überprüft (WCAG AA)
- [ ] Sicherheitsüberprüfung bestanden
- [ ] Dokumentation aktualisiert
- [ ] UAT-Genehmigung erhalten

**Sollte vor der Veröffentlichung vorliegen**:
- [ ] Fehler mit hoher Priorität behoben
- [ ] Leistung optimiert
- [ ] Lasttest abgeschlossen
- [ ] Überwachung eingerichtet

**Nice to Have vor der Veröffentlichung**:
- [ ] Fehler mittlerer Priorität behoben
- [ ] Leistungsbenchmarks veröffentlicht
- [ ] Benutzerkommunikation erstellt

## Fehlerschweregrade

| Schweregrad | Beschreibung | Beispiel | Priorität |
|----------|-------------|---------|----------|
| Kritisch | Feature unbrauchbar, data-Verlust, Sicherheitsverletzung | Benutzer können sich nicht anmelden | Sofort beheben |
| Hoch | Funktion erheblich beeinträchtigt | Die Zahlung schlägt in 50 % der Fälle fehl | So schnell wie möglich reparieren |
| Mittel | Funktion funktioniert mit Einschränkungen | E-Mail um 2 Stunden verzögert | Zeitplan in naher Zukunft |
| Niedrig | Kleines Problem, einfache Problemumgehung | Schaltflächentext falsch ausgerichtet | Rückstand |

## Testfallmanagement

**Testfallvorlage**:
```text
Test Case ID: TC-001
Title: User can log in with valid credentials
Feature: Authentication

Preconditions:
- User has registered account
- Login page accessible

Test Steps:
1. Navigate to login page
2. Enter email address
3. Enter password
4. Click "Sign In"

Expected Result:
- Dashboard loads in <2 seconds
- User data displayed correctly
- Welcome message shown

Status: ☐ Pass ☐ Fail ☐ Blocked
```

## Regressionsteststrategie

**Was ist ein Regressionstest**:
- Kernbenutzer-Workflows (Anmeldung, Registrierung, wichtige Funktionen)
- Geänderte Funktionen (von dieser Version betroffen)
- Verwandte Funktionen (Abhängigkeiten)
- Kritische Pfade (Funktionen mit hoher Nutzung)
- Leistungssensible Bereiche
- Sicherheitsrelevante Funktionen

**Beispiel für einen Regressionsumfang**:
```text
Feature: User Login
├── Happy path: Valid credentials
├── Invalid credentials: Wrong password
├── Inactive account: Suspended user
├── Password reset flow
├── Login rate limiting
├── Session management
└── Related features:
    ├── Signup (account creation)
    ├── Password change
    └── Session timeout
```

## Benutzerakzeptanztest (UAT)

**UAT-Vorbereitung**:
- Identifizieren Sie Stakeholder und Benutzer
- Erstellen Sie Testszenarien aus User Stories
- Bereiten Sie eine produktionsähnliche Testumgebung vor
- Bereiten Sie realistische Testdaten vor
- Planen Sie UAT-Sitzungen


## QualitätTore

**Code Quality Gate** (vor der Zusammenführung):
- Codeüberprüfung genehmigt
- Linting bestanden
- Typprüfung bestanden
- Unit-Tests bestanden (>80 % Abdeckung)
- Keine Sicherheitslücken

**Integration Quality Gate** (vor dem Staging):
- Bestehen der Integrationstests
- E2E-Tests bestanden
- Leistung innerhalb der Grundlinie
- Datenbankmigrationen validiert
- API-Verträge validiert

**Quality Gate freigeben** (vor der Produktion):
- Regressionstest bestanden
- UAT-Genehmigung erhalten
- Kritische Fehler behoben
- Leistungsbenchmarks erfüllt
- Sicherheitsüberprüfung genehmigt

## Qualitätsgrundsätze

- **Qualität ist nicht verhandelbar**: Geben Sie niemals Produkte heraus, die unter den Qualitätsstandards liegen
- **Alles messen**: Qualitätsentscheidungen, die durch Metriken und Daten gestützt werden
- **Früh testen**: Testen nach links in die Entwicklung verschieben
- **Risikobasierter Ansatz**: Hochrisikobereiche gründlicher testen
- **Kontinuierliche Verbesserung**: Verfolgen Sie Kennzahlen und verbessern Sie sich im Laufe der Zeit
