# ADR Writer Agent

Der ADR Writer Agent (ADR-WRITER) ist Ihr Spezialist für die Dokumentation von Architekturentscheidungen, die Aufzeichnung von Kompromissen und die Führung einer historischen Aufzeichnung technischer Entscheidungen. Dieser Agent stellt sicher, dass wichtige Entscheidungen im richtigen Kontext erfasst werden, sodass Teams nicht erneut über dieselben Entscheidungen debattieren.

## Fähigkeiten

- **ADR-Erstellung**: Erstellen Sie Architekturentscheidungsaufzeichnungen, die dokumentieren, warum technische Entscheidungen getroffen wurden
- **Alternativendokumentation**: Notieren Sie 2–5 geprüfte Alternativen mit den jeweiligen Vor- und Nachteilen
- **Kompromissanalyse**: Dokumentieren Sie positive, negative und neutrale Konsequenzen von Entscheidungen
- **Entscheidungsverknüpfung**: Verbinden Sie verwandte ADRs und verweisen Sie auf unterstützende Forschung
- **Statuslebenszyklus**: Verfolgen Sie ADRs über die Zustände „Vorgeschlagen“ → „Akzeptiert“ → „Veraltet/Ersetzt“.
- **Forschungsintegration**: Koordinieren Sie sich mit dem Forschungsagenten zur Unterstützung der technischen Analyse
- **ADR-Indexpflege**: Halten Sie docs/03-decisions/README.md mit allen Entscheidungen auf dem neuesten Stand

## Wann zu verwenden

Benutzen Sie diesAgent, wenn Sie Folgendes dokumentieren müssen:

- **Technologieauswahl**: Framework-Auswahl, Datenbankauswahl, Programmiersprache, Bibliotheksentscheidungen
- **Architekturmuster**: Monolith vs. Microservices, REST vs. GraphQL, ereignisgesteuert vs. Anfrage-Antwort
- **Datenmodellierung**: Schemadesign, Normalisierungsstrategie, Caching-Ansatz
- **Sicherheitsansätze**: Authentifizierungsmechanismus, Verschlüsselungsstrategie, Geheimnisverwaltung
- **Infrastrukturentscheidungen**: Hosting-Plattform, CI/CD-System, Überwachungslösung
- **Entwicklungspraktiken**: Teststrategie, Verzweigungsmodell, Richtlinien für den Codestil
- **Leistungskompromisse**: Wenn Entscheidungen über Geschwindigkeit vs. Speicher oder Durchsatz vs. Latenz getroffen werden

## Wie es funktioniert

1. **Kontextladen**: Der Agent liest die Fachwissensdatei, überprüft aktuelle ADRs und durchsucht docs/10-research/ nach unterstützender Forschung

2. **Verständnis**: Der Agent stellt klar, welche Entscheidung getroffen wird und warum sie jetzt erforderlich ist

3. **Forschungsintegration**: Der Agent prüft, ob zum Entscheidungsthema vorhandene Forschungsergebnisse vorhanden sind.oder fordert bei Bedarf Recherchen über `/agileflow:research:ask` an

4. **Alternativenanalyse**: Agent identifiziert 2–5 Alternativen mit Vor-/Nachteilen (aus Recherche oder Benutzereingaben)

5. **ADR-Entwurf**: Agent erstellt ADR mit:
   - Fortlaufende 4-stellige Nummer (0001, 0002 usw.)
   - Titel in Imperativform („PostgreSQL für Datenpersistenz verwenden“)
   - Kontext, der erklärt, warum jetzt eine Entscheidung erforderlich ist
   - Entscheidungserklärung (was wurde gewählt)
   - Alternativen mit Vor-/Nachteilen
   - Konsequenzen (positiv, negativ, neutral)
   - Status (Vorgeschlagen, Akzeptiert, Veraltet, Ersetzt)
   - Referenzen, die auf Forschungsnotizen und offizielle Dokumente verweisen

6. **Vorschau und Genehmigung**: Der Agent zeigt eine Vorschau für die JA/NEIN-Genehmigung an

7. **Erstellung**: Agent erstellt `docs/03-decisions/adr-<NUMBER>-<slug>.md`

8. **Indizierung**: Agent aktualisiert docs/03-decisions/README.md mit neuem Eintrag

## Beispiel

```bash
# Via /babysit
/agileflow:babysit
> "We need to document why we chose PostgreSQL over MongoDB for our database"
```

Der ADR Writer Agent würde:
1. Überprüfen Sie docs/10-research/ auf vorhandene Datenbankrecherchen
2. Falls fehlend, fordern Sie an: „Lassen Sie mich zuerst die Datenbankoptionen recherchieren.“
3. GWeitere Recherchen zu PostgreSQL, MongoDB und anderen Optionen
4. Befragen Sie Sie zum Kontext (Warum jetzt? Einschränkungen? Teamerfahrung?)
5. Entwurf ADR-0003: Verwenden Sie PostgreSQL für die Datenpersistenz
6. Vorschau mit allen Alternativen und Kompromissen anzeigen
7. Genehmigung einholen (JA/NEIN)
8. Erstellen Sie adr-0003-use-postgresql.md
9. Aktualisieren Sie docs/03-decisions/README.md
10. Ankündigung: „ADR-0003 erstellt und bereit zur Teamüberprüfung“

## Schlüsselverhalten

- **Research First**: Prüft immer, ob bereits Forschungsergebnisse vorhanden sind, bevor ADR verfasst (oder angefordert) wird.
- **Kontext über Implementierung**: Konzentriert sich darauf, warum eine Entscheidung getroffen wurde, nicht auf Implementierungsdetails
- **Historische Aufzeichnung**: Löscht niemals ADRs (wird für den historischen Kontext und die Mustererkennung aufbewahrt)
- **Klare Alternativen**: Dokumentiert mindestens zwei Alternativen mit ehrlichen Vor- und Nachteilen (keine Befürwortung)
- **Fortlaufende Nummerierung**: ADRs fortlaufend nummeriert (0001, 0002, 0003 usw.)
- **Statusverfolgung**: Aktualisiert den ADR-Status, wenn sich Entscheidungen weiterentwickeln (Vorgeschlagen → Angenommen → Ersetzt)
- **Verknüpfung**: Verbindet verwandte ThemenADRs zum Erstellen eines Entscheidungsdiagramms im Laufe der Zeit

## Werkzeuge verfügbar

Dieser Agent hat Zugriff auf:
- **Lesen**: Durchsuchen Sie ADRs und verstehen Sie den Entscheidungsverlauf
- **Schreiben**: Neue ADR-Dateien erstellen
- **Bearbeiten**: Bestehende ADRs aktualisieren (Status ändern, Referenzen hinzufügen)
- **Glob**: Finden Sie ADRs nach Muster
- **Grep**: Entscheidungsinhalte durchsuchen

## ADR-Vorlage

Jeder ADR folgt dieser Struktur:

```markdown
# ADR-<NUMBER>: <Title>

**Date**: YYYY-MM-DD
**Status**: Proposed | Accepted | Deprecated | Superseded
**Deciders**: [Names/roles who made decision]

## Context
[Describe forces at play: technical, business, team, timeline]
[What problem are we solving? What constraints exist?]

## Decision
[State the decision clearly in 1-3 sentences]
[Explain key reasons for this choice]

## Alternatives Considered

### Option 1: [Name]
**Pros**:
- [Benefit]
- [Benefit]

**Cons**:
- [Cost or limitation]
- [Cost or limitation]

**Why rejected**: [Explanation]

### Option 2: [Name]
...

## Consequences

### Positive
- [Benefit we expect]
- [Advantage]

### Negative
- [Cost or limitation]
- [Trade-off]

### Neutral
- [Change that's neither good nor bad]

## References
- [Title](URL) - Description
- [Research Note](docs/10-research/YYYYMMDD-slug.md) - How research informed decision
```

## ADR-Statuslebenszyklus

- **Vorgeschlagen**: Wird geprüft, noch nicht genehmigt (Diskussionsphase)
- **Akzeptiert**: Genehmigt und sollte vom Team befolgt werden
- **Veraltet**: Nicht mehr empfohlen (aber für den Verlauf aufbewahrt)
- **Ersetzt**: Durch neueres ADR ersetzt (Link zum Ersatz, z. B. „Ersetzt ADR-0001“)

## Qualitätscheckliste

Vor der Erstellung von ADR:
- Der Kontext erklärt deutlich, warum JETZT eine Entscheidung erforderlich ist
- Mindestens 2 Alternativen dokumentiert mit Vor- und Nachteilen
- Die Entscheidungserklärung ist klar und spezifisch
- Konsequenzen sind ausgewogen (positiv, negativ, neutral)
- Referenzen für wichtige Ansprüche enthalten
- Status angemessen (Vorschlägezur Überprüfung freigegeben, zur Fertigstellung angenommen)
- Die Nummer ist fortlaufend (überprüfen Sie die neueste ADR-Nummer)
- Forschungsergebnisse werden zitiert, sofern verfügbar

## Entscheidungskategorien

ADRs fallen typischerweise in diese Kategorien:

**Technologieentscheidungen**
- Welches Framework verwendet werden soll
- Welche Datenbank soll ausgewählt werden?
- Welche Programmiersprache für neue Dienste
- Welche Bibliothek für bestimmte Funktionen

**Architekturentscheidungen**
- Monolith vs. Microservices
- Synchrone vs. asynchrone Kommunikation
- Ereignisgesteuert vs. Anfrage-Antwort
- REST vs. GraphQL-API-Stil

**Entscheidungen zum Datendesign**
- Schema-Normalisierungsebene
- Caching-Strategie
- Richtlinien zur Datenaufbewahrung
- Master-Slave- oder Peer-to-Peer-Replikation

**Sicherheit und Infrastruktur**
- Authentifizierungsmechanismus (JWT, OAuth2, sitzungsbasiert)
- Verschlüsselung im Ruhezustand und während der Übertragung
- Secrets-Management-Ansatz
- Bereitstellungs- und Hosting-Plattform

**Entwicklungspraktiken**
- TEntwicklungsstrategie (Einheit, Integration, E2E-Abdeckung)
- Git-Verzweigungsmodell (Git-Flow, Trunk-basiert usw.)
- Codeüberprüfungsprozess
- CI/CD-Automatisierungsebene

## ADRs aktualisieren und verknüpfen

**Als veraltet markieren**:
```markdown
**Status**: Deprecated

This decision is no longer recommended because [reason].
See ADR-0005 for the current approach.
```

**Als abgelöst markieren**:
```markdown
**Status**: Superseded

Superseded by ADR-0005: [New Decision Title]

This decision was replaced because [reason].
```

**Querverweise**:
```markdown
## Related Decisions
- Builds on: ADR-0003 (Database choice)
- Related to: ADR-0002 (API design)
- Supersedes: ADR-0001 (Previous framework decision)
```

## Forschungsintegration

Der ADR Writer Agent immer:
1. Überprüft docs/10-research/ auf vorhandene Forschungsergebnisse zu diesem Thema
2. Wenn Recherche fehlt, bitte um: `/agileflow:research:ask TOPIC=<decision>`
3. Integriert Forschungsergebnisse in den Abschnitt „Alternativen“.
4. Referenzen Forschungsnotiz im ADR-Abschnitt „Referenzen“.
5. Verlinkt ADR zurück zur Forschungsnotiz zur späteren Bezugnahme

Dies gewährleistet:
- Entscheidungen basieren auf einer gründlichen Analyse
- Forschungsergebnisse bleiben erhalten
- Andere Agenten können unterstützende Beweise für Entscheidungen finden
- Konsistenz in der gesamten Entscheidungsdokumentation

## Wiederdebatte verhindern

ADRs verhindern, dass Teams dieselben Entscheidungen erneut diskutieren, indem sie:
- **Kontext erfassen**: Aufzeichnen dessen, was zum Zeitpunkt der Entscheidung bekannt wargemacht
- **Dokumentieren von Einschränkungen**: Warum bestimmte Optionen nicht möglich waren
- **Kompromisse bei der Aufnahme**: Was aufgegeben wurde, um andere Vorteile zu erzielen
- **Verlinkung zur Forschung**: Verweis auf technische Analysen, die eine fundierte Entscheidung getroffen haben
- **Verlauf pflegen**: Zeigt, wie sich Entscheidungen im Laufe der Zeit entwickelt haben

Wenn eine ähnliche Entscheidungsfrage auftritt:
1. Überprüfen Sie docs/03-decisions/ auf entsprechende ADRs
2. Überprüfen Sie den Kontext und die Konsequenzen
3. Entscheiden Sie, ob sich die Situation geändert hat (ersetzen Sie, wenn ja, beziehen Sie sich auf die bestehende, wenn nein)

## Gemeinsame Muster

**ADRs ermöglichen den Wissenstransfer**
- Neues Team für michMitglieder verstehen frühere Überlegungen
- Verhindert die erneute Implementierung fehlgeschlagener Ansätze
- Beschleunigt das Onboarding (ADRs lesen vs. Versuch und Irrtum)

**Entscheidungsdiagramm im Zeitverlauf**
- Entscheidungen bauen aufeinander auf (ADR-0002 baut auf ADR-0001 auf)
- Einige Entscheidungen haben Vorrang vor anderen (ADR-0003 ersetzt ADR-0001)
- Bilden Sie ein zusammenhängendes Netz des Verständnisses

**Evidenzbasierte Entscheidungen**
- Jede ADR wird durch Forschungsnotizen untermauert
- Zitiert offizielle Dokumentation, keine Meinungen
- Beinhaltet Benchmarks und Kompromissanalysen
- Ermöglicht eine sichere Entscheidungsfindung
