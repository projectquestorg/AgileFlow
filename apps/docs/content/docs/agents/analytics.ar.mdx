# متخصص التحليلات

AG-ANALYTICS يتخصص في تحليلات المنتجات وتتبع الأحداث وتحليل سلوك المستخدم ولوحات معلومات المقاييس لتوفير رؤى مستندة إلى البيانات.

## القدرات

- **Event Tracking Schema Design**: اتفاقيات الأسماء والخصائص المطلوبة وتتبع متوافق مع الخصوصية
- **Analytics Dashboards**: مقاييس حقيقية والمقاييس الاشتراك بشفق التحويل
- **User Behavior Analysis**: تحليل الكتل ، تجزئة المستخدمين ، تتبع الاحتفاظ
- **Conversion Funnel Analysis**: تحديد التسريبات وفرص التحسين
- **A/B Testing Infrastructure**: تتبع متغير الاختبار ، التحليل الإحصائي
- **Data Quality Validation**: كشف الشذوذ ، فحوصات اكتمال البيانات
- **Privacy Compliance**: معالجة موافقة GDPR ، سياسات الاحتفاظ بالبيانات

## متى تستخدم

استخدم AG-ANALYTICS عندما:

- تصميم تتبع الأحداث للميزات الجديدة
- إنشاء لوحات معلومات التحليلات لتصور المقاييس
- تحليل سلوك المستخدم وأنماط الاشتراك
- إعداد تتبع قمع التحويل
- إنشاء إطار عمل اختبار A/B
- ضمان امتثال GDPR / CCPA للتتبع
- مراقبة جودة البيانات والشذوذ

## كيف يعمل

1. **Context Loading**: يقرأ ملف الخبرة وحالة التحليلات الحالية
2. **Business Metrics**: يحدد المقاييس المهمة لأهداف الأعمال
3. **Event Schema Design**: ينشئ اتفاقيات الأسماء ويتتبع مخاوف PII
4. **Documentation**: يوثق جميع الأحداث والمخطط ومتطلبات الخصوصية
5. **Implementation**: ينسق مع AG-API و AG-UI لتنفيذ التتبع
6. **Dashboards**: ينشئ لوحات معلومات ومشاهد تحليلية حقيقية
7. **Monitoring**: ينشئ التحقق من جودة البيانات وكشف الشذوذ

## مثال

```bash
# عبر /babysit
/agileflow:babysit
> "I need to track user engagement and identify where users drop off"

# AG-ANALYTICS ستقوم بـ:
# 1. Define business metrics (DAU, MAU, retention)
# 2. Design event schema (no PII, GDPR compliant)
# 3. Create event catalog documentation
# 4. Coordinate with AG-API for backend tracking
# 5. Coordinate with AG-UI for frontend tracking
# 6. Create engagement dashboards
# 7. Set up cohort analysis
```

## السلوكيات الرئيسية

- **No PII Tracking**: لا تتتبع أبدًا كلمات المرور أو رسائل البريد الإلكتروني أو بطاقات الائتمان أو بيانات الصحة
- **Privacy-First**: تأخذ دائمًا في الاعتبار امتثال GDPR / CCPA
- **Schema First**: جميع الأحداث المحددة مع اتفاقيات الأسماء قبل التنفيذ
- **Data Quality**: التحقق من الأحداث ومراقبة الشذوذ
- **User Privacy**: احترم موافقة المستخدم وتمكن الإقالة بسهولة

## الأدوات المتاحة

- Read, Write, Edit, Bash, Glob, Grep
- الوصول إلى Session Harness للتحقق

## تتبع الأحداث

**اتفاقية تسمية الحدث** (صيغة object_action):
- `button_clicked` - نقر المستخدم على الزر
- `form_submitted` - أرسل المستخدم النموذج
- `page_viewed` - عرض المستخدم للصفحة
- `payment_completed` - الدفع بنجاح
- `feature_enabled` - تم تبديل الميزة

استخدم snake_case وليس camelCase.

**مثال مخطط الحدث**:
```json
{
  "event_name": "button_clicked",
  "timestamp": "2025-10-21T10:00:00Z",
  "user_id": "user-123",
  "session_id": "session-456",
  "properties": {
    "button_name": "sign_up",
    "page_url": "/landing"
  },
  "context": {
    "os": "iOS",
    "browser": "Safari",
    "country": "US",
    "app_version": "2.1.0"
  }
}
```

## لوحات معلومات التحليلات

**لوحة معلومات المقاييس الرئيسية**:
- المستخدمون في الوقت الفعلي (مباشر)
- مشاهدات الصفحات (آخر 24 ساعة)
- معدل التحويل (٪)
- معدل الارتداد (٪)
- مدة الجلسة (متوسط)

**مقاييس الاشتراك**:
- المستخدمون النشطون يوميًا (DAU)
- المستخدمون النشطون شهريًا (MAU)
- عودة المستخدمين (٪)
- استخدام الميزة
- الاشتراك بالمحتوى

**قمع التحويل**:
- الخطوة 1: عرض الصفحة الرئيسية
- الخطوة 2: بدء الاشتراك
- الخطوة 3: التحقق من البريد الإلكتروني
- الخطوة 4: استخدام الميزة الأولى
- معدل التحويل الكلي

## الامتثال للخصوصية

**لا تتبع**:
- كلمات المرور أو الرموز المميزة للمصادقة
- أرقام بطاقات الائتمان أو تفاصيل الدفع
- أرقام الضمان الاجتماعي أو الهويات الحكومية
- معلومات الصحة / الطب
- البيانات البيومترية
- أي PII بدون موافقة صريحة

**متطلبات GDPR**:
- الانضمام الصريح (وليس مُحددًا مسبقًا)
- الكشف الواضح عما يتم تتبعه
- خيار الإقالة السهلة
- يمكن للمستخدم طلب الوصول إلى البيانات / الحذف
- احتفاظ البيانات لمدة 90 يومًا (الأحداث الخام)

## اختبار A / B

**إعداد الاختبار**:
```json
{
  "test_id": "checkout_button_color_2025",
  "variant_a": "blue_button",
  "variant_b": "green_button",
  "allocation": "50/50",
  "primary_metric": "checkout_completion_rate",
  "minimum_sample_size": 10000
}
```

**التحليل**:
- هل حجم العينة كافٍ؟
- هل الفرق ذو دلالة إحصائية؟ (p <0.05)
- ما هو حجم التأثير العملي؟
- أي متغير فاز؟

## تقسيم المستخدمين

**القطاعات الشائعة**:
- حسب تاريخ الاشتراك (مستخدمون جدد ، 7 أيام ، 30 يومًا ، 90+ يومًا)
- حسب مستوى الاستخدام (مستخدمون قويون ، عاديون ، سبات)
- حسب اعتماد الميزة
- حسب الجغرافيا
- حسب المستوى الدعم
- حسب مصدر الاستحواذ

## تحليل الكتل

**احتفاظ الكتل** (تتبع حسب تاريخ الاشتراك):
- الأسبوع 0: 100٪ (خط الأساس)
- الأسبوع 1: 65٪ محتفظ به
- الأسبوع 2: 42٪ محتفظ به
- الأسبوع 3: 31٪ محتفظ به
- الأسبوع 4: 24٪ محتفظ به

الاتجاه: هل منحنيات الاحتفاظ تتحسن؟

## تحليل القمع

**تحديد التسريبات**:
1. عرض الصفحة الرئيسية: 50000
2. نموذج الاشتراك مفتوح: 15000 (30٪)
3. النموذج المرسل: 8000 (16٪)
4. البريد الإلكتروني التحقق: 6500 (13٪)
5. أول تسجيل دخول: 5200 (10٪)

أكبر قطرة: 70٪ يفقدون الاهتمام بين الصفحة الرئيسية ونموذج الاشتراك.

## جودة البيانات

**قواعد التحقق**:
- طابع الحدث صحيح (ضمن آخر 30 يومًا)
- اسم الحدث يطابق المخطط
- تنسيق معرّف المستخدم صحيح
- الخصائص المطلوبة موجودة
- لا PII في الخصائص
- تنسيق معرف الجلسة صحيح

**المراقبة من أجل**:
- أحداث مكررة (إلغاء التكرار)
- خصائص مفقودة (فجوات التتبع)
- الطوابع الزمنية غير الصحيحة (انجراف الساعة)
- انتهاكات المخطط
- الشذوذ (ارتفاع مفاجئ أو انخفاض)

## قائمة التحقق من الجودة

قبل وضع علامة على عمل التحليلات كاملة:

- تم تصميم مخطط الحدث والتوثيق
- اتفاقيات تسمية الحدث متسقة
- لا PII في التتبع (التحقق من الخصوصية)
- تم تنفيذ موافقة GDPR
- تم تحديد سياسة الاحتفاظ بالبيانات
- تم إنشاء لوحات المعلومات ومفيدة
- تم تنفيذ قواعد التحقق من جودة البيانات
- تم تكوين كشف الشذوذ
- إطار عمل اختبار A / B جاهز
- التوثيق كامل (فهرس الحدث ، لوحات المعلومات)
- اختبار الاختبار الناجح مع test_status: "passing"
