# وكيل الوصولية

وكيل الوصولية (AG-ACCESSIBILITY) هو متخصص في الوصولية يضمن أن التطبيقات قابلة للاستخدام من قبل الجميع، بما في ذلك الأشخاص ذوي الإعاقات. يقوم هذا الوكيل بإجراء تدقيقات WCAG، والاختبار باستخدام تقنيات المساعدة، والتأكد من اتباع مبادئ التصميم الشامل.

## القدرات

- **WCAG 2.1 Auditing**: التحقق من امتثال المستوى AA و AAA
- **Screen Reader Testing**: الاختبار مع NVDA و JAWS و VoiceOver
- **Keyboard Navigation**: ضمان إمكانية الوصول إلى جميع الوظائف عبر لوحة المفاتيح
- **Color Contrast Verification**: التحقق من نسبة التباين ≥4.5:1
- **Focus Management**: مؤشرات التركيز المرئية وترتيب التبويب المنطقي
- **Accessibility Testing**: الاختبار الآلي واليدوي للوصولية
- **Assistive Technology Support**: الدعم لقارئات الشاشة والتحكم الصوتي وغيرها
- **Inclusive Design**: أنماط التصميم التي تعمل لجميع المستخدمين
- **ARIA Implementation**: HTML الدلالي وعلامات ARIA الصحيحة
- **Accessibility Documentation**: إنشاء أدلة واختبارات الوصولية والمعايير

## متى تستخدم

استخدم وكيل الوصولية عندما:

- تصميم ميزات أو مكونات جديدة
- الحاجة إلى التحقق من امتثال WCAG 2.1
- مشاكل التباين اللوني أو مخاوف الوصولية البصرية
- عدم عمل الملاحة عبر لوحة المفاتيح
- مشاكل التوافق مع قارئ الشاشة
- الحاجة للاختبار باستخدام تقنيات المساعدة
- إنشاء معايير الوصولية
- التخطيط للإصدارات (الاختبار الإلزامي للوصولية)
- إبلاغ المستخدمين ذوي الإعاقات عن مشاكل
- الحاجة لـ ADR بشأن قرارات الوصولية

## كيف يعمل

1. **Knowledge Loading**: يقرأ الوكيل الخبرة ومعايير WCAG
2. **Feature Analysis**: يقوم الوكيل بمراجعة القصة لآثار الوصولية
3. **Automated Audit**: يقوم الوكيل بتشغيل axe و Lighthouse و WAVE
4. **Manual Testing**: يقوم الوكيل باختبار الملاحة عبر لوحة المفاتيح وقارئات الشاشة
5. **Issue Documentation**: يقوم الوكيل بسرد جميع مشاكل الوصولية مع شدتها
6. **Remediation Planning**: يقوم الوكيل بالتنسيق مع فريق التصميم/الواجهة الأمامية
7. **Verification**: يقوم الوكيل بإعادة الاختبار بعد الإصلاح
8. **Documentation**: يقوم الوكيل بتحديث status.json والإبلاغ عن النتائج

## مثال

```bash
# عبر babysit - تدقيق مكون نموذج جديد للوصولية
/agileflow:babysit
> "لقد أنشأنا مكون نموذج جديد. هل هو قابل للوصول؟"

# إخراج الوكيل:
# Accessibility Audit: New Form Component
#
# Issues Found:
# CRITICAL (blocks functionality):
# ├── Form fields have no labels (missing <label> elements)
# ├── Submit button has no accessible name
# └── Error messages not announced to screen readers
#
# MAJOR (significantly impacts usability):
# ├── Color contrast: 3.8:1 (need 4.5:1)
# └── Focus indicators not visible
#
# MINOR (impacts some users):
# ├── Form instructions could be clearer
# └── Missing aria-required on required fields
#
# WCAG Compliance: 25% AA (need 100%)
#
# Remediation Steps:
# 1. Add <label> for each form field
# 2. Add accessible names to buttons (aria-label)
# 3. Add aria-live to error messages
# 4. Increase color contrast to 4.5:1
# 5. Add visible focus indicators (2px outline)
#
# Accessibility Checklist:
# ✅ Test with screen reader (NVDA)
# ✅ Keyboard-only navigation
# ✅ Tab order verification
# ✅ Color contrast validation
# ✅ Focus indicator verification
```

## السلوكيات الرئيسية

- **Inclusive by Default**: تصميم لجميع المستخدمين من البداية
- **Test with Real Users**: استخدام تقنيات المساعدة الفعلية
- **Keyboard First**: جميع الوظائف قابلة للوصول بدون ماوس
- **Color Not Only**: المعلومات ليست ممثلة باللون وحده
- **Clear Focus**: مؤشرات تركيز مرئية على كل عنصر تفاعلي
- **Semantic HTML**: استخدام عناصر HTML الصحيحة للمعنى
- **No Assumptions**: عدم افتراض أن جميع المستخدمين لديهم لوحات مفاتيح أو أجهزة إشارة

## معايير WCAG 2.1

**مستويات WCAG 2.1**:
- **Level A**: أدنى وصولية
- **Level AA**: وصولية معززة (المعيار الموصى به)
- **Level AAA**: الوصولية المثلى (استهدف هذا)

**مبادئ WCAG 2.1**:
1. **Perceivable**: يجب أن تكون المعلومات ومكونات الواجهة قابلة للإدراك
2. **Operable**: يجب أن تكون مكونات الواجهة والملاحة قابلة للتشغيل
3. **Understandable**: يجب أن تكون المعلومات والعمليات مفهومة
4. **Robust**: يجب أن يكون المحتوى قويًا لتقنيات المساعدة

**معايير WCAG 2.1 AA الرئيسية**:
- **Color Contrast**: النص ≥4.5:1، مكونات الواجهة ≥3:1
- **Keyboard Accessible**: جميع الوظائف عبر لوحة المفاتيح
- **No Keyboard Traps**: يمكن الخروج من أي مكون بـ Tab
- **Focus Order**: ترتيب Tab يطابق الترتيب البصري (منطقي)
- **Focus Visible**: مؤشر التركيز مرئي ≥2px outline
- **Touch Target Size**: ≥44x44 CSS pixels
- **Semantic HTML**: استخدام عناصر HTML الصحيحة
- **ARIA Labels**: الأزرار/الروابط لها أسماء قابلة للوصول
- **Form Labels**: جميع حقول النموذج ملصقة بشكل صحيح

## اختبار الوصولية

**الاختبار الآلي** (قم بتشغيل أولاً للحصول على تعليقات سريعة):
- **axe DevTools**: امتداد المتصفح، يحدد المشاكل الشائعة
- **Lighthouse**: تدقيق مدمج في Chrome DevTools
- **WAVE**: أداة WebAIM للتعليقات البصرية
- **Color contrast checkers**: التحقق من نسبة 4.5:1

**الاختبار اليدوي** (ضروري للتقييم الحقيقي):
- **Keyboard-only navigation**: التبويب عبر الصفحة بأكملها (بدون ماوس)
- **Screen reader testing**: NVDA (Windows) و JAWS (Windows) و VoiceOver (macOS)
- **Tab order verification**: التحقق من التدفق المنطقي
- **Focus indicator visibility**: مشاهدة التركيز على كل عنصر تفاعلي
- **Color contrast validation**: التحقق من الأداة والفحص البصري

**دعم قارئ الشاشة**:
- NVDA (Windows، مجاني)
- JAWS (Windows، مدفوع، الأكثر شهرة)
- VoiceOver (macOS/iOS، مدمج)
- TalkBack (Android، مدمج)

## مشاكل الوصولية الشائعة

| المشكلة | المشكلة | الحل |
|-------|---------|-----|
| لا توجد تسميات على الإدخالات | قارئ الشاشة لا يعرف ما هو الحقل | أضف `<label for="id">` |
| أزرار بدون أيقونة | لا يوجد اسم لقارئ الشاشة | أضف `aria-label="name"` |
| التباين اللوني المنخفض | من الصعب القراءة لمستخدمي الرؤية المنخفضة | زيادة التباين إلى 4.5:1 |
| مؤشرات التركيز المفقودة | لا يمكن لمستخدمي لوحة المفاتيح رؤية مكانهم | أضف مخطط التركيز المرئي |
| لا يوجد نص بديل على الصور | لا يمكن لقارئ الشاشة وصف الصورة | أضف `alt="..."` وصفي |
| فخاخ لوحة المفاتيح | لا يمكن للمستخدمين الابتعاد عن الملاحة | إصلاح إدارة التركيز |
| معلومات بـ اللون فقط | يفتقد مستخدمو عمى الألوان المعنى | استخدم الأيقونات والنصوص والأنماط |

## قائمة التحقق من الوصولية

**الوصولية البصرية**:
- [ ] التباين اللوني ≥4.5:1 للنص (WCAG AA)
- [ ] التباين اللوني ≥3:1 لمكونات الواجهة
- [ ] لا توجد معلومات ملونة فقط (استخدم الأيقونات/النصوص)
- [ ] يمكن تغيير حجم النص إلى 200% بدون تمرير
- [ ] مؤشر التركيز مرئي (≥2px outline)
- [ ] أهداف اللمس ≥44x44 CSS pixels
- [ ] تصميم بصري متسق (الأزرار تبدو مثل الأزرار)

**إمكانية الوصول عبر لوحة المفاتيح**:
- [ ] جميع الوظائف قابلة للوصول عبر لوحة المفاتيح
- [ ] لا توجد فخاخ لوحة المفاتيح (Tab يمكنه الخروج من أي مكون)
- [ ] ترتيب Tab منطقي (يطابق الترتيب البصري)
- [ ] Enter/Space ينشط الأزرار
- [ ] أسهم المفاتيح تنقل القوائم/القوائم المنسدلة
- [ ] Escape يغلق النوافذ المنبثقة/القوائم المنسدلة

**إمكانية الوصول إلى قارئ الشاشة**:
- [ ] HTML دلالي (button, link, form labels)
- [ ] جميع الأزرار/الروابط لها اسم قابل للوصول
- [ ] الصور لها نص بديل (أو مخفية إذا كانت زخرفية)
- [ ] حقول النموذج لها تسميات (أو aria-label)
- [ ] أخطاء النموذج مُعلنة لقارئ الشاشة
- [ ] علامات مرجعية موجودة (main, nav, complementary)
- [ ] روابط تخطي لملاحة لوحة المفاتيح
- [ ] مناطق حية لمحتوى ديناميكي

## أنماط التصميم الشامل

**النماذج**:
```html
<!-- جيد: تسمية مرتبطة بالإدخال -->
<label for="email">البريد الإلكتروني</label>
<input id="email" type="email" required aria-required="true">
<span class="error" role="alert">البريد الإلكتروني مطلوب</span>

<!-- سيء: محتوى عنصر نائب فقط، بدون تسمية -->
<input placeholder="Email" type="email">
```

**الأزرار**:
```html
<!-- جيد: اسم قابل للوصول واضح -->
<button aria-label="Close menu">
  <svg aria-hidden="true"><!-- icon --></svg>
</button>

<!-- سيء: أيقونة فقط بدون تسمية -->
<button>
  <svg><!-- icon --></svg>
</button>
```

**الصور**:
```html
<!-- جيد: نص بديل وصفي -->
<img src="profile.jpg" alt="Sarah Chen, product manager">

<!-- سيء: النص البديل مفقود -->
<img src="profile.jpg">

<!-- جيد: صورة زخرفية مخفية -->
<img src="decoration.jpg" alt="" aria-hidden="true">
```

**التباين اللوني**:
```text
جيد: النص الأبيض (#FFF) على الأزرق (#0066CC) = 8.6:1 ✅ AAA
سيء: النص الأبيض (#FFF) على الرمادي (#999) = 3.9:1 ❌ AA (need 4.5:1)
جيد: النص الأبيض (#FFF) على الأسود (#000) = 21:1 ✅ AAA
```

## أولوية التصحيح

| الأولوية | الشدة | أمثلة |
|----------|----------|----------|
| حرجة | تحجب الوظيفة الأساسية | لا يوجد وصول من لوحة المفاتيح للميزات، فقدان تسميات النموذج |
| رئيسية | تؤثر بشكل كبير على سهولة الاستخدام | التباين اللوني المنخفض، مؤشرات التركيز المفقودة |
| بسيطة | تؤثر على بعض المستخدمين | حاشيات ARIA المفقودة، رسائل الخطأ المطولة |

## الحركة والرسوم المتحركة

**اعتبارات الوصولية**:
- [ ] الرسوم المتحركة تحترم `prefers-reduced-motion`
- [ ] لا توجد وميض يسبب نوبات (لا توجد وميضات >3 في الثانية)
- [ ] التشغيل التلقائي معطل بشكل افتراضي
- [ ] يمكن للمستخدمين إيقاف الرسوم المتحركة

## التنسيق

يقوم وكيل الوصولية بالتنسيق مع:

- **AG-DESIGN**: التباين البصري ومؤشرات التركيز والأنماط الشاملة
- **AG-UI**: تنفيذ ARIA و HTML الدلالي وملاحة لوحة المفاتيح
- **AG-TESTING**: أتمتة اختبار الوصولية
- **AG-QA**: الوصولية في معايير الإصدار
- **فريق المنتج**: التحقق من تحسينات الوصولية مع المستخدمين الحقيقيين

## أوامر القطع

- `/agileflow:research:ask TOPIC=...` - البحث عن أفضل الممارسات a11y
- `/agileflow:ai-code-review` - مراجعة الرمز لمشاكل الوصولية
- `/agileflow:adr-new` - توثيق قرارات الوصولية
- `/agileflow:status STORY=... STATUS=...` - تحديث حالة القصة

## مبادئ الوصولية

- **Inclusive by design**: ضع في الاعتبار جميع المستخدمين من البداية
- **Test with real users**: استخدم تقنيات المساعدة الفعلية
- **No shortcuts**: امتثال WCAG إلزامي وليس اختياري
- **Layers of accessibility**: بصري، لوحة المفاتيح، قارئ الشاشة، إدراكي
- **Continuous improvement**: إصلاح المشاكل كما يتم اكتشافها
- **Education**: ساعد الفريق في فهم سبب أهمية الوصولية
