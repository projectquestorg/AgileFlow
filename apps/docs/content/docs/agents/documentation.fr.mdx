# Agent de documentation

L'agent de documentation (AG-DOCUMENTATION) est votre spécialiste pour la rédaction technique, la documentation API, les guides d'utilisation et la mise à jour de la documentation avec votre base de code. Cet agent garantit que toute la documentation reste à jour, claire et utile pour les développeurs nouveaux et expérimentés.

## Capacités

- **Documentation API** : spécifications OpenAPI/Swagger, descriptions des points de terminaison, exemples de requêtes/réponses et codes d'erreur
- **Guides de l'utilisateur et didacticiels** : guides de démarrage, instructions étape par étape, bonnes pratiques et dépannage
- **Guides du développeur** : configuration du projet, flux de travail de développement, présentation de l'architecture et modèles d'extension
- **Gestion README** : fichiers README racine, spécifiques au module et aux fonctionnalités
- **Maintenance du journal des modifications** : notes de version et suivi des modifications destiné aux utilisateurs
- **Audits de documentation** : identifiez le contenu obsolète, les liens rompus et les exemples manquants
- **Documents générés automatiquement** : prise en charge du générateur OpenAPI, TypeDoc/JSDocet diagrammes d'architecture

## Quand l'utiliser

Utilisez cet agent lorsque vous avez besoin de :

- Rédiger ou mettre à jour la documentation de l'API avec des exemples pratiques
- Créer des guides d'utilisation pour les nouvelles fonctionnalités (instructions étape par étape)
- Mettre à jour les fichiers README pour refléter l'état actuel du projet
- Rédiger les guides d'intégration des développeurs
- Créer des guides de dépannage et des FAQ pour les problèmes courants
- Documenter les décisions d'architecture et la conception du système
- Tenir à jour le journal des modifications avec les notes de version
- Correction des liens de documentation brisés
- Documentation d'audit pour l'obsolescence et l'exactitude
- Garder la documentation synchronisée avec les modifications de code

## Comment ça marche

1. **Chargement du contexte** : l'agent lit le fichier d'expertise (modèles de documentation, emplacements des fichiers, conventions) et analyse la structure du projet (CLAUDE.md, status.json, versions récentes)

2. **Audit de documentation** : l'agent identifie les lacunes dans la documentation, les sections obsolètes, les liens rompus et les exemples manquants en analysant les documents/structures et en les comparant au code.

3. **Planification** : l'agent propose une amélioration de la documentationents, suggère de nouveaux guides, identifie le contenu obsolète et priorise le travail

4. **Rédaction** : l'agent crée/met à jour la documentation avec une priorité de clarté plutôt que de brièveté, inclut des exemples de travail (prêts à copier-coller) et maintient un formatage cohérent.

5. **Contrôle qualité** : l'agent vérifie que la documentation est à jour avec le code, que toutes les étapes sont expliquées, que les exemples fonctionnent et que la navigation entre les documents est claire.

6. **Coordination** : l'agent met à jour le fichier status.json, ajoute les messages d'achèvement et publie les mises à jour sur le bus de l'agent.

## Exemple

```bash
# Via /babysit
/agileflow:babysit
> "We just released v2.17.0. Need to update the changelog and API docs for the new endpoints"
```

L'agent de documentation :
1. Vérifiez les nouveaux points de terminaison d'API ajoutés dans la v2.17.0
2. Mettre à jour la documentation de l'API avec des exemples de requêtes/réponses
3. Ajoutez des notes de version au journal des modifications mettant en évidence les modifications apportées aux utilisateurs
4. Mettez à jour le fichier README si nécessaire
5. Vérifiez que tous les liens fonctionnent

## Comportements clés

- **Priorité à la documentation** : clarté plutôt que brièveté – explique tout pour les nouveaux utilisateurs
- **Toujours inclure des exemples** : la documentation sans exemples est inutile ; tous les exemples sont des copiesle goût est prêt
- **Garder à jour** : la documentation doit rester à jour avec le code ; les documents périmés sont pires que pas de documents
- **Inclure le dépannage** : résolvez les problèmes courants que les utilisateurs rencontreront
- **Modifications majeures dans le document** : critique pour les utilisateurs mettant à niveau les versions
- **Vérification des liens** : corrige de manière proactive les liens de documentation rompus
- **Cohérence** : maintient une mise en forme, un ton et une structure cohérents dans tous les documents.
- **Préservation du contexte** : utilise compact_context (priorité : élevée) pour rester concentré pendant les longues conversations, en préservant le suivi de la couverture de la documentation et la validation des liens grâce au compactage du contexte.

## Configuration du contexte compact

L'agent de documentation utilise compact_context **haute priorité** pour garantir que la qualité de la documentation reste au centre :

```yaml
compact_context:
  priority: high
  preserve_rules:
    - "LOAD EXPERTISE FIRST: Always read packages/cli/src/core/experts/documentation/expertise.yaml"
    - "NEVER LET DOCS LAG BEHIND CODE: Stale docs are worse than no docs (misinformation)"
    - "ALWAYS INCLUDE EXAMPLES: Documentation without examples is useless"
    - "KEEP LINKS CURRENT: Run link checker, fix broken references"
    - "DOCUMENT BREAKING CHANGES: Critical for users upgrading versions"
    - "CLARITY OVER BREVITY: Explain everything for new users"
  state_fields:
    - current_story
    - documentation_coverage
    - broken_links_count
    - outdated_sections
```

Cela garantit que les règles critiques pour la documentation (maintenir les documents à jour, y compris les exemples, l'exhaustivité de la documentation) et l'état actuel (quels documents sont obsolètes, quels liens sont rompus) persistent grâce au compactage du contexte.

## Modèles courants

**Documentation en tant que code**
- Toute la documentation versionnée avec du code dans git
- La source de vérité est la base de code vivante, pas des documents séparés
- Modifications apportées aux mises à jour de la documentation des déclencheurs de code

**Structure de répertoire numérotée**
- Documentation organisée en docs/ avec répertoires numérotés (00-méta à 10-recherche)
- Structure cohérente dans tous les projets

**Maintenance des documents**
- Planifier des audits trimestriels pour trouver des informations périmées
- Après chaque version, mettre à jour la documentation
- Après chaque changement d'API, mettez à jour la documentation de l'API
- Réparer les liens rompus de manière proactive

## Messages de coordination

L'agent de documentation communique avec d'autres agents via des messages de bus :

```jsonl
{"ts":"2025-10-21T10:00:00Z","from":"AG-DOCUMENTATION","type":"status","text":"API docs updated for v2.17.0"}
{"ts":"2025-10-21T10:05:00Z","from":"AG-DOCUMENTATION","type":"question","text":"UI component API changed - need updated documentation for props"}
```

Lorsque d'autres agents terminent leur travail :
- AG-API termine le point de terminaison → L'agent de documentation met à jour la documentation de l'API
- Composant de versions AG-UI → Composant de documents de l'agent de documentation props et utilisation
- La publication a lieu → L'agent de documentation met à jour le journal des modifications et les notes de version
- Décision d'architecture prise → L'agent de documentation met à jour la documentation ADR
