# وكيل الأبحاث

وكيل الأبحاث (RESEARCH) هو المتخصص في البحث الفني وإدارة المعرفة وبناء مطالبات بحثية شاملة. يجمع هذا الوكيل المعلومات من مصادر موثوقة، ويجمع النتائج، ويحتفظ بفهرس بحثي قابل للبحث للرجوع إليه في المستقبل.

## القدرات

- **Web Research**: ابحث في الوثائق الرسمية والمصادر الرسمية عن المعلومات الفنية
- **ChatGPT Prompt Building**: قم بإنشاء مطالبات بحثية شاملة تطلب خطط التنفيذ ومقتطفات التعليمات البرمجية ومسودات ADR وتفاصيل القصة والمزيد
- **Research Note Curation**: احفظ نتائج الأبحاث المنظمة في المستندات/10-research/ بتنسيق متسق
- **Research Index Maintenance**: حافظ على تحديث docs/10-research/README.md بأحدث الأبحاث أولاً
- **Stale Research Detection**: حدد البحث الذي مضى عليه أكثر من 90 يومًا والذي قد يحتاج إلى التحديث
- **Research Gap Identification**: قم بوضع علامة على الأبحاث المفقودة للقرارات الفنية والتفاعلات العكسية
- **Agent Coordination**: قم بإخطار الوكلاء الطالبين عند اكتمال البحث عبر رسائل الناقل

## متى تستخدم

استخدم هذا الوكيل عندما تحتاج إلى:

- ابحث عن موضوع تقني أو اختيار تكنولوجي
- فهم المفاضلات بين الأساليب المختلفة
- قم ببناء موجه ChatGPT شامل لإجراء تحليل أعمق
- توثيق النتائج في شكل منظم للرجوع إليها في المستقبل
- تحديث نتائج الأبحاث القديمة (> 90 يومًا)
- ابحث عن الأبحاث الموجودة بالفعل قبل اتخاذ القرارات
- دعم إنشاء ADR من خلال البحث في الخلفية التقنية
- احصل على إرشادات التنفيذ لنهج الميزة

## كيف يعمل

### سير عمل أبحاث الويب

1. **Understanding**: يوضح الوكيل سؤال البحث والقيود
2. **Search**: يقوم الوكيل بالبحث في الوثائق الرسمية والمصادر الرسمية (WebSearch، WebFetch)
3. **Gathering**: يقوم الوكيل بجمع النتائج الرئيسية - الأساليب والمقايضات وأفضل الممارسات والمعايير
4. **Synthesis**: يقوم الوكيل بتنظيم النتائج في تنسيق منظم
5. **Saving**: الوكيل يحفظ البحث في `docs/10-research/<YYYYMMDD>-<slug>.md`
6. **Indexing**: يقوم الوكيل بتحديث docs/10-research/README.md مع الإدخال في الجدول الزمني
7. **Coordination**: يقوم الوكيل بنشر رسالة الناقل لإعلام الوكيل الطالب

### سير العمل الفوري لأبحاث ChatGPT

1. **Context Loading**: يقرأ الوكيل CLAUDE.md، وcontext.md، وstatus.json، والإعلانات ذات الصلة، والأبحاث الحالية لتجنب الازدواجية
2. **Understanding**: يوضح الوكيل موضوع البحث والأسئلة المحددة التي تحتاج إلى إجابات
3. **Prompt Building**: يقوم الوكيل بإنشاء طلب سريع شامل:
   - تل؛ ملخص الدكتور
   - خطة التنفيذ خطوة بخطوة مع مسارات الملفات
   - الحد الأدنى من مقتطفات التعليمات البرمجية القابلة للتشغيل (جاهزة للنسخ واللصق)
   - التكوين وإعداد البيئة
   - أنماط معالجة الأخطاء
   - خطافات المراقبة والتحليلات
   - أمثلة الاختبار (الوحدة، التكامل، E2E)
   - قائمة التحقق من الاختبار اليدوي
   - اعتبارات الأمن والخصوصية
   - مسودة ADR مع الخيارات والإيجابيات/السلبيات
   - تفاصيل القصة مع معايير القبول المقدمة/متى/ثم
   - خطة التراجع والمخاطر
   - قالب هيئة العلاقات العامة
   - Official documentation sources only (cited with title, URL, date)

4. **Output**: يوفر الوكيل المطالبة ككتلة تعليمات برمجية لسهولة النسخ واللصق
5. **Processing**: يقوم المستخدم بلصق نتائج ChatGPT مرة أخرى إلى Claude
6. **Saving**: يقوم الوكيل بحفظ النتائج في `docs/10-research/<YYYYMMDD>-<slug>.md`
7. **Indexing**: يقوم الوكيل بتحديث فهرس البحث
8. **Notification**: يقوم الوكيل بإعلام الوكيل الطالب عبر رسالة الناقل

## مثال

```bash
# Via /babysit
/agileflow:babysit
> "Need to research JWT vs OAuth2 for authentication"
```

سيقوم وكيل الأبحاث بما يلي:
1. البحث عن مصادر موثوقة (JWT RFC، مواصفات OAuth2، مقالات Security.com)
2. جمع النتائج الرئيسية (عديم الحالة مقابل ذو الحالة، رموز التحديث، التفويض، حالات الاستخدام)
3. قم بإنشاء مذكرة بحثية لمقارنة الأساليب
4. حفظ في docs/10-research/20251021-jwt-vs-oauth2.md
5. تحديث فهرس الأبحاث
6. نشر رسالة الناقل: "اكتمل البحث: تم حفظ JWT vs OAuth2 في docs/10-research/20251021-jwt-vs-oauth2.md"

## السلوكيات الرئيسية

- **Official Sources Only**: يستشهد البحث بالوثائق الرسمية، وليس منشورات مدونة عشوائية
- **Comprehensive Prompts**: مطالبات ChatGPT شاملة، وتطلب تفاصيل التنفيذ والاختبارات والاعتبارات الأمنية ومسودات ADR
- **Structured Formatting**: تتبع جميع الأبحاث بنية متسقة (الملخص، النتائج، النهج الموصى به، المفاضلات، المصادر)
- **Stale Detection**: يحدد البحث الذي مضى عليه أكثر من 90 يومًا ويضع علامات للتحديث المحتمل عند تغير التكنولوجيا
- **No Duplication**: التحقق من الأبحاث الموجودة قبل البدء بأبحاث جديدة
- **Agent Coordination**: إخطار الوكلاء الطالبين على الفور عند اكتمال البحث
- **Clear Sourcing**: تستشهد جميع الأبحاث بالمصادر مع العنوان وعنوان URL وتاريخ الاسترجاع

## الأدوات المتاحة

هذا الوكيل لديه حق الوصول إلى:
- **Read**: تصفح ملفات البحث وفهم النتائج الموجودة
- **Write**: إنشاء ملاحظات بحثية جديدة
- **Edit**: تحديث الملاحظات البحثية الموجودة
- **Glob**: البحث عن ملفات البحث حسب النمط
- **Grep**: البحث في محتوى البحث
- **WebSearch**: ابحث في الويب عن المعلومات الفنية
- **WebFetch**: جلب وتحليل صفحات الويب

## قائمة مراجعة الجودة

قبل حفظ الأبحاث:
- التاريخ حالي (تنسيق YYYY-MM-DD)
- الملخص موجز (2-3 جمل)
- تم إدراج ما لا يقل عن 3 نتائج رئيسية
- النهج الموصى به ذكر بوضوح
- المصادر المذكورة مع عناوين URL وتواريخ الاسترجاع
- ADRs / القصص / الملاحم ذات الصلة المرتبطة
- تم تحديث الفهرس بأحدث إدخال في الأعلى
- جميع الروابط الخارجية هي للوثائق الرسمية

## تحديد الفجوات البحثية

يقوم وكيل الأبحاث بتحديد الفجوات البحثية عندما:
- اختيار التكنولوجيا لم يتم بحثه بعد (مطلوب قبل ADR)
- نهج عدم اليقين المذكور في ملاحظات القصة
- تمت مناقشة طرق متعددة دون فائز واضح
- ADR موجود ولكنه يفتقر إلى البحوث الداعمة
- البحث قديم (> 90 يومًا وتغيرت التكنولوجيا)
- يلزم إجراء بحث مكرر (تم تحديث الإصدارات التقنية)

## التنسيق مع الوكلاء الآخرين

تأتي طلبات البحث عادةً من:
- **AG-UI**: أنظمة التصميم، أنماط المكونات، تقنيات الوصول
- **AG-API**: بنيات واجهة برمجة التطبيقات (API)، وتصميمات قواعد البيانات، وأنماط المصادقة
- **AG-CI**: أطر الاختبار ومنصات CI وأدوات جودة التعليمات البرمجية
- **AG-DEVOPS**: استراتيجيات النشر، تنسيق الحاويات، المراقبة
- **ADR-WRITER**: البدائل الفنية قبل اتخاذ القرارات
- **EPIC-PLANNER**: أبحاث كومة التكنولوجيا قبل تخطيط الملاحم

عند طلب البحث:
1. تحقق من docs/09-agents/bus/log.jsonl للحصول على رسالة طلب البحث
2. بحث كامل (الويب أو موجه ChatGPT)
3. حفظ النتائج في المستندات/10-research/
4. تحديث فهرس الأبحاث
5. نشر رسالة الحافلة لإعلام مقدم الطلب

## قالب ChatGPT الفوري

عند إنشاء موجه بحثي، قم بتضمين ما يلي:

```text
Context: [Project overview from context.md]
Current priority: [From status.json]
Related ADRs: [List relevant decisions]

Research topic: [Specific question]

Please provide:
1. TL;DR (2-3 sentences)
2. Step-by-step implementation plan with exact file paths
3. Minimal runnable code snippets (copy-paste ready)
4. Configuration (env vars, config files, CLI flags)
5. Error handling patterns
6. Observability (logging, metrics, tracing)
7. Tests: unit, integration, E2E examples
8. Manual testing checklist
9. Security checklist (auth, input validation, secrets, etc.)
10. Privacy considerations (PII, GDPR, data retention)
11. ADR draft with context, options, recommendation, consequences
12. Story breakdown (3-6 stories with Given/When/Then AC)
13. Rollback plan
14. Risks and gotchas
15. PR body template
16. Sources: official docs/repos only with title, URL, date

Final checklist:
- [ ] Ready to paste back to Claude for implementation
- [ ] All code snippets tested conceptually
- [ ] Risks clearly flagged
```

## الأنماط الشائعة

**Two Research Workflows**
- **Web Research**: البحث المباشر والتوليف (أسرع للمواضيع المباشرة)
- **ChatGPT Prompt**: إنشاء موجه شامل لتحليل الذكاء الاصطناعي (أفضل للقرارات المعقدة)

**Research Before Decisions**
- ابحث دائمًا قبل إنشاء ADRs (يطلب ADR-WRITER البحث)
- ابحث دائمًا قبل التخطيط للملاحم (يطلب EPIC-PLANNER إجراء بحث)
- ضع علامة على عدم اليقين في القصص والأبحاث قبل التنفيذ

**Stale Research Management**
- تدقيق ربع سنوي للمستندات/10-الأبحاث/ للأبحاث القديمة
- ضع علامة "مستبدل" إذا كان البحث الجديد يتعارض مع القديم
- لا تحذف البحث مطلقًا (احتفظ بالسياق التاريخي)
- قم بالتحديث إذا تغيرت الإصدارات التقنية
