# Agente de accesibilidad

El Agente de Accesibilidad (AG-ACCESSIBILITY) es un especialista en accesibilidad que garantiza que todas las personas puedan utilizar las aplicaciones, incluidas las personas con discapacidades. Este agente realiza auditorías WCAG, pruebas con tecnologías de asistencia y garantiza que se sigan los principios de diseño inclusivos.

## Capacidades

- **Auditoría WCAG 2.1**: Verificar el cumplimiento de los niveles AA y AAA
- **Prueba de lector de pantalla**: prueba con NVDA, JAWS, VoiceOver
- **Navegación por teclado**: asegúrese de que todas las funciones sean accesibles mediante el teclado
- **Verificación de contraste de color**: validar una relación de contraste ≥4,5:1
- **Gestión de enfoque**: indicadores de enfoque visibles y orden de tabulación lógico
- **Pruebas de accesibilidad**: pruebas automáticas y manuales
- **Soporte de tecnología de asistencia**: soporte para lectores de pantalla, control por voz, etc.
- **Diseño inclusivo**: patrones de diseño que funcionan para todos los usuarios
- **Implementación de ARIA**: HTML semántico y etiquetas ARIA adecuadas
- **Documentación de accesibilidad**: cree todas las guías y guías de prueba.normas

## Cuándo utilizar

Utilice el Agente de Accesibilidad cuando:

- Diseño de nuevas características o componentes.
- Necesita verificación de cumplimiento de WCAG 2.1
- Problemas de contraste de color o problemas de accesibilidad visual
- La navegación con el teclado no funciona
- Problemas de compatibilidad con lectores de pantalla.
- Necesidad de realizar pruebas con tecnologías de asistencia.
- Crear estándares de accesibilidad
- Lanzamientos de planificación (se requieren pruebas de accesibilidad)
- Los usuarios con discapacidades informan problemas
- Necesita ADR para decisiones de accesibilidad.

## Cómo funciona

1. **Carga de conocimientos**: el agente lee la experiencia y los estándares WCAG
2. **Análisis de funciones**: el agente revisa la historia en busca de implicaciones de accesibilidad
3. **Auditoría automatizada**: el agente ejecuta axe, Lighthouse, WAVE
4. **Prueba manual**: el agente prueba la navegación con el teclado y los lectores de pantalla.
5. **Documentación del problema**: el agente enumera todos los problemas de accesibilidad con su gravedad.
6. **Planificación de remediación**: el agente coordina con los equipos de diseño/UI
7. **Verificación**: el agente vuelve a realizar pruebas después de las correcciones
8. **Documentación**: Actualizaciones del agentestatus.json e informes de resultados

## Comportamientos clave

- **Inclusivo por defecto**: Diseño para todos los usuarios desde el principio
- **Prueba con usuarios reales**: utilice tecnologías de asistencia reales
- **Teclado primero**: todas las funciones accesibles sin mouse
- **No solo color**: la información no se transmite únicamente por el color
- **Enfoque claro**: indicadores de enfoque visibles en cada elemento interactivo
- **HTML semántico**: utilice elementos HTML adecuados para dar significado
- **Sin suposiciones**: no asuma que todos los usuarios tienen teclados o dispositivos señaladores

## Estándares WCAG 2.1

**Niveles WCAG 2.1**:
- **Nivel A**: Accesibilidad mínima
- **Nivel AA**: Accesibilidad mejorada (estándar recomendado)
- **Nivel AAA**: Accesibilidad óptima (apunta a esto)

**Principios WCAG 2.1**:
1. **Perceptible**: la información y los componentes de la interfaz de usuario deben ser perceptibles
2. **Operable**: Los componentes de la interfaz de usuario y la navegación deben estar operativos
3. **Comprensible**: La información y las operaciones deben ser comprensibles
4. **Robusto**: el contenido debe ser sólido para las tecnologías de asistencia.

**Criterios clave WCAG 2.1 AA**:
- **Contraste de color**: texto ≥4,5:1, componentes de la interfaz de usuario ≥3:1
- **Accesible por teclado**: todas las funciones a través del teclado
- **Sin trampas de teclado**: puede salir de cualquier componente
- **Orden de enfoque**: el orden de las pestañas coincide con el orden visual (lógico)
- **Enfoque visible**: indicador de enfoque visible contorno ≥2px
- **Tamaño de destino táctil**: ≥44x44 píxeles CSS
- **HTML semántico**: utilice elementos HTML adecuados
- **Etiquetas ARIA**: los botones/enlaces tienen nombres accesibles
- **Etiquetas de formulario**: todos los campos del formulario están correctamente etiquetados

## Pruebas de accesibilidad

**Pruebas automatizadas** (ejecute primero para obtener comentarios rápidos):
- **axe DevTools**: extensión del navegador, identifica problemas comunes
- **Lighthouse**: auditoría integrada en Chrome DevTools
- **WAVE**: herramienta WebAIM para comentarios visuales
- **Cuadros de contraste de color**: Verifique la relación 4,5:1

**T manualprueba** (esencial para una evaluación real):
- **Navegación solo con teclado**: tabula a través de toda la página (sin mouse)
- **Pruebas de lectores de pantalla**: NVDA (Windows), JAWS (Windows), VoiceOver (macOS)
- **Verificación del orden de pestañas**: comprobar el flujo lógico
- **Visibilidad del indicador de enfoque**: vea el enfoque en cada elemento interactivo
- **Validación de contraste de color**: Verificación de herramienta + inspección visual

**Compatibilidad con lectores de pantalla**:
- NVDA (Windows, gratis)
- JAWS (Windows, pago, más popular)
- VoiceOver (macOS/iOS, integrado)
- TalkBack (Android, integrado)

## Problemas comunes de accesibilidad

| Problema | Problema | Arreglar |
|-------|---------|-----|
| Sin etiquetas en las entradas | El lector de pantalla no sabe qué campo es | Agregar `<label for="id">` |
| Botones de sólo iconos | Sin nombre para lectores de pantalla | Agregar `aria-label="name"` |
| Bajo contraste de color | Difícil de leer para usuarios con baja visión | Aumentar el contraste a 4,5:1 |
| Indicadores de enfoque faltantes | Los usuarios del teclado no pueden ver dónde están | Agregar contorno de enfoque visible |
| Sin texto alternativo en las imágenes | Carolina del SurLos lectores reales no pueden describir la imagen. Agregue descriptivo `alt="..."` |
| Trampas de teclado | Los usuarios no pueden navegar fuera | Arreglar la gestión del enfoque |
| Información sólo en color | Los usuarios daltónicos pierden el significado | Utilice iconos, texto, patrones |

## Lista de verificación de accesibilidad

**Accesibilidad visual**:
- [ ] Contraste de color ≥4,5:1 para texto (WCAG AA)
- [] Contraste de color ≥3:1 para componentes de la interfaz de usuario
- [] No hay información solo de color (use íconos/texto)
- [] Texto redimensionable al 200% sin desplazamiento
- [] Indicador de enfoque visible (contorno ≥2px)
- [] Objetivos táctiles ≥44x44 píxeles CSS
- [] Diseño visual consistente (los botones parecen botones)

**Accesibilidad del teclado**:
- [] Todas las funciones accesibles mediante el teclado
- [] Sin trampas de teclado (la pestaña puede salir de cualquier componente)
- [] Orden de tabulación lógico (coincide con el orden visual)
- [] Enter/Space activa los botones
- [] Teclas de flecha para navegar por listas/menús
- [] Escape cierra modales/popovers

**Accesibilidad del lector de pantalla**:
- [] HTML semántico (botón, enlace, etiquetas de formulario)
- [] Todos los botones/los enlaces tienen un nombre accesible
- [] Las imágenes tienen texto alternativo (u oculto si son decorativas)
- [] Los campos del formulario tienen etiquetas (o aria-label)
- [] Errores de formulario anunciados a los lectores de pantalla
- [] Puntos de referencia presentes (principal, navegación, complementarios)
- [] Saltar enlaces para la navegación con el teclado
- [] Regiones en vivo para contenido dinámico

## Patrones de diseño inclusivos

**Formularios**:
```html
<!-- Good: Label associated with input -->
<label for="email">Email</label>
<input id="email" type="email" required aria-required="true">
<span class="error" role="alert">Email is required</span>

<!-- Bad: Placeholder only, no label -->
<input placeholder="Email" type="email">
```

**Botones**:
```html
<!-- Good: Clear accessible name -->
<button aria-label="Close menu">
  <svg aria-hidden="true"><!-- icon --></svg>
</button>

<!-- Bad: Icon-only with no label -->
<button>
  <svg><!-- icon --></svg>
</button>
```

**Imágenes**:
```html
<!-- Good: Descriptive alt text -->
<img src="profile.jpg" alt="Sarah Chen, product manager">

<!-- Bad: Alt text missing -->
<img src="profile.jpg">

<!-- Good: Decorative image hidden -->
<img src="decoration.jpg" alt="" aria-hidden="true">
```

**Contraste de color**:
```text
Good: White text (#FFF) on blue (#0066CC) = 8.6:1 ✅ AAA
Bad:  White text (#FFF) on gray (#999) = 3.9:1 ❌ AA (need 4.5:1)
Good: White text (#FFF) on black (#000) = 21:1 ✅ AAA
```

## Prioridad de remediación

| Prioridad | Gravedad | Ejemplos |
|----------|----------|----------|
| Crítico | Bloquea la funcionalidad principal | No hay acceso mediante teclado a las funciones, faltan etiquetas de formulario |
| Mayor | Impacta significativamente la usabilidad | Bajo contraste de color, faltan indicadores de enfoque |
| Menor | Impacta a algunos usuarios | Faltan anotaciones ARIA, mensajes de error detallados |

## Movimiento y animación

**Consideraciones de accesibilidad**:
- [] Las animaciones respetan `prefers-reduced-motion`
- [ ] Sin parpadeos que provoquen convulsiones (no >3 parpadeos/segundo)
- [] Autop.poner deshabilitado por defecto
- [] Los usuarios pueden pausar las animaciones.

El Agente de Accesibilidad coordina con:

- **AG-DESIGN**: Contraste visual, indicadores de enfoque, patrones inclusivos
- **AG-UI**: implementación de ARIA, HTML semántico, navegación por teclado
- **AG-TESTING**: Automatización de pruebas de accesibilidad
- **AG-QA**: Accesibilidad en criterios de lanzamiento
- **Equipo de producto**: validar las mejoras de accesibilidad con usuarios reales
