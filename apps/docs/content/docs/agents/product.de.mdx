# Produktagent

Der Produktagent ist darauf spezialisiert, vage Feature-Ideen in klare, umsetzbare Anforderungen umzuwandeln. Es sammelt Anforderungen von Stakeholdern, schreibt User Stories mit Akzeptanzkriterien und validiert Funktionen vor der epischen Planung.

## Fähigkeiten

- Befragen Sie Stakeholder und erfassen Sie Anforderungen mithilfe bewährter Fragetechniken
- Erstellen Sie detaillierte Benutzerpersönlichkeiten mit Zielen und Schwachstellen
- Schreiben Sie User Stories im Standardformat „Als … ich möchte … damit …“.
- Definieren Sie Akzeptanzkriterien im Format „Gegeben/Wann/Dann“.
- Identifizieren Sie Grenzfälle und Fehlerszenarien, die Entwickler möglicherweise übersehen
- Definieren Sie Erfolgsmetriken für Funktionen
- Verwalten Sie den Bereich, um eine Ausweitung des Bereichs zu verhindern
- Klären Sie unscharfe Anforderungen, bevor Sie sie an die epische Planung übergeben

## Wann zu verwenden

Verwenden Sie dieses Mittel, wenn:

- **Vague feature requests** - „Benutzerprofile hinzufügen“ muss vor der Entwicklung klargestellt werden
- **Starting a new epic** - Erfassen und klären Sie zunächst die Anforderungen
- **Complex features** - Randfallanalyse und Akzeptanzkriterien erforderlich
- **Stakeholder interviews needed** - Benutzerbedürfnisse und Geschäftswert verstehen
- **Scope debates** - Bestimmen Sie, was für diese Version in/out ist
- **User story writing** - Anforderungen in überprüfbare Akzeptanzkriterien umwandeln
- **Feature validation** - Die Bestätigungsfunktion löst das eigentliche Benutzerproblem

## Wie es funktioniert

Der Produktagent folgt einem strukturierten Arbeitsablauf:

### Schritt 1: Laden von Wissen

Bevor Sie mit der Anforderungserfassung beginnen:
- Lesen Sie CLAUDE.md für Produktstrategien
- Sehen Sie sich docs/10-research/ für Benutzerforschung und Wettbewerbsanalysen an
- Überprüfen Sie docs/03-decisions/ auf Produkt-ADRs und Präzedenzfälle
- Überprüfen Sie docs/08-project/roadmap.md auf den Kontext

### Schritt 2: Stakeholder-Interviews

Stellen Sie wichtige Fragen, um die Funktion zu verstehen:

1. **"Why do we need this feature?"** → Geschäftswert und Ergebnisse
2. **"Who will use this?"** → Benutzerpersönlichkeiten und -rollen
3. **"What problem does it solve?"** → Problemstellung
4. **"How will success be measured?"** → Erfolgskennzahlen
5. **"What are edge cases?"** → Fehlerszenarien und Randbedingungen
6. **"Are there constraints?"** → Technische, rechtliche oder geschäftliche Einschränkungen
7. **"What's the priority?"** → Must-have vs. nice-to-have

### Schritt 3: Schreiben Sie User Stories

Erstellen Sie Geschichten im Standardformat:

```text
As a [user role],
I want [action/feature],
so that [benefit/value].
```

Beispiel:

```text
As a new user,
I want to sign up with email and password,
so that I can access my account and manage my data.
```

### Schritt 4: Akzeptanzkriterien definieren

Schreiben Sie überprüfbare Kriterien im Format „Gegeben/Wann/Dann“:

```text
Acceptance Criteria:

- Given a new user with valid email and strong password
  When they submit the signup form
  Then their account is created and verification email sent

- Given a user with an already-registered email
  When they try to signup
  Then they see error "Email already in use"

- Given a user with weak password
  When they try to signup
  Then they see error "Password must be 8+ chars with uppercase and number"
```

### Schritt 5: Randfälle identifizieren

Berücksichtigen Sie immer Fehlerszenarien:
- Ungültige Eingabe (leere Felder, falsches Format)
- Randbedingungen (zu lang, zu kurz, Null, negativ)
- Konfliktszenarien (doppelte E-Mail, gleichzeitige Updates)
- Fehlerbehebung (Was passiert, wenn das Speichern fehlschlägt?)
- Berechtigungsszenarien (nicht authentifiziert, falsche Berechtigungen)

### Schritt 6: Erfolgskennzahlen definieren

Antwort: „Woher wissen wir, dass diese Funktion erfolgreich war?“
- Akzeptanzmetriken (Anzahl der Benutzer, die die Funktion nutzen)
- Engagement-Metriken (täglich/wöchentlich aktive Benutzer)
- Geschäftskennzahlen (Umsatzauswirkungen, Kostensenkung)
- Qualitätskennzahlen (Fehlerraten, Benutzerzufriedenheit)

### Schritt 7: Dokumentumfang

Dokumentieren Sie klar und deutlich, was inbegriffen und ausgeschlossen ist:

```text
Scope: User Profile Management

IN SCOPE:
- View profile page
- Edit name, email, avatar
- Save changes to database

OUT OF SCOPE (Future Features):
- Profile followers/following system
- Activity history timeline
- Profile customization (bio, links)

Rationale: Keeping MVP small to launch faster.
Activity history and following system planned for v2.
```

## User Story-Format

### Standardformat

```text
As a [user role],
I want [action/feature],
so that [benefit/value].
```

### Mit Abschnitt „Anforderungen“.

```text
As a new user,
I want to sign up with email and password,
so that I can access my account.

Requirements:
1. User enters email and password
2. System validates email format
3. System checks password strength (min 8 chars, 1 uppercase, 1 number)
4. System creates account and sends verification email
5. User receives confirmation via email
```

### Mit vollständigen Akzeptanzkriterien

```text
Acceptance Criteria:

- Given a new user with valid email and strong password
  When they submit the signup form
  Then their account is created and verification email sent

- Given a user with an already-registered email
  When they try to signup
  Then they see error "Email already in use"

- Given a user with weak password
  When they try to signup
  Then they see error "Password must be 8+ chars with uppercase and number"

- Given a network error during signup
  When the error occurs
  Then user sees "Connection failed, please try again" and can retry
```

## Priorisierungsrahmen

Der Produktagent verwendet **MoSCoW Method** Anforderungen priorisieren:

| Priority | Definition | Example |
|----------|-----------|---------|
| **Must Have** | Critical for feature to work | User can view their profile |
| **Should Have** | Important but can defer | Profile shows last login time |
| **Could Have** | Nice-to-have, low priority | Profile customization (bio, links) |
| **Won't Have** | Out of scope for this release | Profile followers/following system |

## Standards für Akzeptanzkriterien

### Gute vs. schlechte Beispiele

**Bad**: „Benutzer kann sich anmelden“
- ❌ Vage, nicht überprüfbar

**Good**: „Gültige E-Mail-Adresse/Passwort angegeben, wenn das Formular gesendet wurde, dann ist die Authentifizierung erfolgreich und der Benutzer wird zum Dashboard weitergeleitet.“
- ✅ Spezifisch, überprüfbar, beschreibt Verhalten

**Bad**: „System sollte schnell sein“
- ❌ Subjektiv, keine messbaren Kriterien

**Good**: „Angegebener authentifizierter Benutzer, beim Laden des Profils wird die Seite dann innerhalb von 2 Sekunden geladen.“
- ✅ Messbar, prüfbar

**Bad**: „Fehlerbehandlung“
- ❌ Zu vage, fehlende Szenarien

**Good**: „Beim Speichern ist ein Netzwerkfehler aufgetreten. Wenn ein Fehler auftritt, sieht der Benutzer die Schaltfläche „Wiederholen“ und kann die Arbeit ohne Datenverlust fortsetzen.“
- ✅ Spezifisches Fehlerszenario mit Wiederherstellungspfad

## Checkliste für Edge-Cases

Berücksichtigen Sie immer folgende Szenarien:

- **Invalid Input**: Leere Felder, falsches Format, Sonderzeichen
- **Boundary Conditions**: Zu lang, zu kurz, Null, negative Zahlen
- **Conflict Scenarios**: Doppelte Einträge, gleichzeitige Updates, Rennbedingungen
- **Error Recovery**: Netzwerkfehler, Zeitüberschreitung, Datenbankfehler
- **Permission Scenarios**: Nicht authentifiziert, falsche Berechtigungen, abgelaufene Token

## Scope-Management

Verhindern Sie Scope Creep, indem Sie Folgendes klar dokumentieren:

1. **Write scope statement**: „Diese Funktion umfasst X, schließt Y aus“
2. **Document rationale**: Warum sind Dinge ausgeschlossen?
3. **Create future features**: Ausschlüsse in separate Epics umwandeln
4. **Evaluate new requirements**: Passt es zum Originalumfang?

## Koordination mit Epic Planner

**Before AG-EPIC-PLANNER starts**:
- Anforderungen geklärt und dokumentiert
- Akzeptanzkriterien im Format „Gegeben/Wann/Dann“.
- Erfolgskennzahlen definiert
- Randfälle identifiziert
- Umfang klar dokumentiert mit Begründung

**After AG-EPIC-PLANNER breaks into stories**:
- Die Akzeptanzkriterien für die Rezensionsgeschichte entsprechen den epischen Anforderungen
- Stellen Sie sicher, dass keine Anforderung durchgerutscht ist
- Stellen Sie sicher, dass die Kriterien jeder Geschichte klar und überprüfbar sind

## Schlüsselverhalten

- **NEVER accepts vague AC** - „Benutzer kann sich anmelden“ ist nicht akzeptabel
- **ALWAYS includes edge cases and error scenarios** - Benutzer werden ihnen begegnen
- **ALWAYS defines success metrics** - Woher wissen wir, dass es uns gelungen ist?
- **ALWAYS documents scope clearly** - Kriechen verhindern
- **ALWAYS uses Given/When/Then format** - Für überprüfbare Akzeptanzkriterien
- **Maintains user focus** - Warum braucht der Benutzer das? Welches Problem löst es?

## Häufige Fehler, die es zu vermeiden gilt

- Akzeptieren vager Akzeptanzkriterien („Benutzer kann sich anmelden“)
- Überspringen des „Warum“ (Vorteile, Geschäftswert)
- Zu frühe Festlegung von Anforderungen ohne Beteiligung der Stakeholder
- Randfälle und Fehlerszenarien werden ignoriert
- Akzeptanz des Scope Creep ohne Kompromissdiskussion
- Vergessen, Erfolgskennzahlen zu definieren
- Keine Einbeziehung tatsächlicher Benutzer/Stakeholder in Interviews

## Beispiel: Von vage zu klar

**Original Request**: „Benutzerprofile hinzufügen“

**After Product Agent Work**:

```text
Epic: User Profile Management

User Stories:

1. As a user, I want to view my profile with my name, email, and avatar,
   so that I can see my account information.

   AC:
   - Given authenticated user, When they navigate to profile page,
     Then they see their name, email, and avatar
   - Given user without avatar, When viewing profile,
     Then a default avatar is displayed
   - Given unauthenticated user, When accessing profile page,
     Then they are redirected to login

2. As a user, I want to edit my profile name and email,
   so that I can keep my account information current.

   AC:
   - Given user on profile page, When they click edit,
     Then they see editable form with current values pre-filled
   - Given user with new email already in use, When saving,
     Then they see error "Email already in use"
   - Given user with valid changes, When they save,
     Then changes are persisted and page shows success message

Success Metrics:
- 80% of active users view their profile within first week
- 40% of users update their profile within first month
- Profile edit success rate > 95%

Scope:
IN SCOPE: View, edit name/email/avatar
OUT OF SCOPE: Activity history, profile followers, bio section
```
