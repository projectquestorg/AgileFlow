# Agente de IU

O agente UI (AG-UI) é o seu especialista em frontend. Ele implementa componentes responsivos, sistemas de estilo, tokens de design, recursos de acessibilidade (WCAG 2.1 AA) e aplica princípios de psicologia UX para criar interfaces bonitas e utilizáveis.

## Capacidades

- Crie componentes responsivos (320px a 1920px+)
- Implementar sistemas de design com tokens (cores, espaçamento, tipografia)
- Garantir acessibilidade WCAG 2.1 AA (navegação por teclado, leitores de tela, contraste)
- Aplicar leis de UX (Jakob's, Hick's, Fitts's, Gestalt, Von Restorff, Peak-End, Doherty)
- Escrever testes de componentes (unidade, integração, acessibilidade com axe-core)
- Crie com design voltado para dispositivos móveis
- Integrar com endpoints de API
- Crie temas e suporte ao modo escuro
- Mantenha a consistência do design em todo o aplicativo
- Escreva microcópia e texto de ajuda

## Quando usar

Use o agente de UI quando:

- Você precisa implementar um novo componente ou recurso
- Você precisa criar um sistema de design ou tokens de design
- Você quer melhorar a acessibilidadey (navegação por teclado, leitores de tela)
- Você precisa tornar a UI responsiva em todos os pontos de interrupção
- Você deseja aplicar as melhores práticas de UX e princípios psicológicos
- Você precisa integrar componentes com endpoints de API
- Você precisa escrever testes de componentes abrangentes
- Você deseja melhorar o desempenho percebido e o feedback do usuário

## Como funciona

1. **Carregamento de contexto**: o agente lê expertise, CLAUDE.md e documentos do sistema de design
2. **Verificação do sistema de design**: o agente verifica se o sistema de design existe (apenas no primeiro andar)
3. **Verificação de status**: o agente encontra histórias prontas e verifica bloqueadores de API
4. **Definição de Pronto**: Agente valida critérios de aceitação e stubs de teste
5. **Modo de planejamento**: para recursos complexos, o agente projeta a estrutura dos componentes
6. **Implementação**: o agente cria componentes usando tokens de design
7. **Acessibilidade**: o agente garante suporte à navegação por teclado e leitor de tela
8. **Leis de UX**: o agente aplica princípios psicológicos da pesquisa de UX
9. **Teste**: o agente grava a unidade, a integração e umtestes de acessibilidade
10. **Verificação**: o agente executa testes para garantir a aprovação da linha de base

## Exemplo

```bash
# Via /babysit (recommended)
/agileflow:babysit
> "I need to implement a user profile page"
```

O agente de UI irá:
1. Verifique o sistema de design (crie se estiver faltando)
2. Procure a dependência da API (GET /api/users/:id)
3. Componentes de projeto:
   - ProfileCard (exibe informações do usuário)
   - ProfileForm (editar informações do usuário)
   - AvatarUpload (alterar avatar)
4. Implemente com tokens de design:
   - Cores do sistema de design (sem #colors codificadas)
   - Espaçamento da escala do token (sem 16px codificado)
   - Tipografia do sistema de design
5. Garanta a acessibilidade:
   - Navegação pelo teclado (guia pelos campos do formulário)
   - Leitor de tela (rótulos de formulário associados)
   - Contraste de cores (4,5:1 para texto)
6. Aplique as leis de UX:
   - Jakob: layout de formulário familiar
   - Adequado: alvos de toque grandes (44px +)
   - Peak-End: Animação de sucesso ao salvar
7. Escreva testes (unidade + integração + acessibilidade)
8. Marque a revisão quando os testes forem aprovados

## Principais comportamentos

- **Carregar experiência primeiro**: Lê expertise.yaml antes de qualquer trabalho
- **Sistema de Design Proativo**: Verifica o sistema de design no primeiro andar, cria se estiver faltando
- **Refatorar para tokens**: ofertas para migrar estilos codificados para o sistema de design
- **Mobile-First**: estilos básicos para dispositivos móveis (320px) e, em seguida, adiciona pontos de interrupção
- **Acessibilidade padrão**: cada componente inclui navegação por teclado e suporte para leitor de tela
- **Leis UX aplicadas**: utiliza princípios psicológicos (proximidade, contraste, affordances)
- **Uso de token**: todos os estilos usam tokens de design, nunca valores codificados
- **Integração do Session Harness**: verifica o status do teste antes de iniciar, requer aprovação nos testes
- **Comandos autônomos**: invoca diretamente /agileflow:impact-análise, /agileflow:adr-new, etc.
- **Coordenação de barramento**: Envia mensagem bloqueada para AG-API se o endpoint estiver faltando
- **Atualizações README**: atualiza proativamente a documentação após alterações significativas
- **Preservação de contexto**: usa compact_context (prioridade: alta) para manter o foco durante longas conversas, preservandog detecção de sistema de design e rastreamento de dependência de API por meio de compactação de contexto

## Configuração de contexto compacto

O agente de UI usa compact_context de **alta prioridade** para garantir que o sistema de design e as dependências da API permaneçam em foco:

```yaml
compact_context:
  priority: high
  preserve_rules:
    - "LOAD EXPERTISE FIRST: Always read packages/cli/src/core/experts/ui/expertise.yaml"
    - "CHECK DESIGN SYSTEM FIRST: Detect if design tokens exist; create if missing"
    - "VERIFY SESSION HARNESS: Check environment.json and test_status baseline"
    - "ONLY in-review if tests pass: test_status:passing required"
    - "CHECK FOR API DEPENDENCIES: Search status.json for blocked UI stories"
    - "APPLY UX LAWS: Jakob's, Hick's, Fitts's, Gestalt, Von Restorff, Peak-End, Doherty"
    - "ACCESSIBILITY REQUIRED: WCAG 2.1 AA minimum"
  state_fields:
    - current_story
    - design_system_status
    - api_dependencies
    - test_status_baseline
```

Isso garante que as regras críticas da UI (detecção do sistema de design, padrões de acessibilidade, verificação do bloqueador de API) e o estado atual (quais tokens de design existem, quais endpoints de API são necessários) persistam por meio da compactação de contexto.

## Lista de verificação de acessibilidade

Mínimo WCAG 2.1 AA para todos os componentes:

**Navegação pelo teclado**:
- A ordem das guias é lógica e visível
- Todos os elementos interativos acessíveis via Tab
- Enter/Espaço ativa botões
- Escape fecha modais
- As teclas de seta navegam nas listas

**Leitores de tela**:
- HTML semântico (`<button>`, `<nav>`, `<main>`, `<article>`)- Texto alternativo para imagens (descritivo, não decorativo)
- aria-label para botões somente de ícone
- aria-live para atualizações dinâmicas
- Rótulos de formulário associados (for= atributo)

**Cor e contraste**:
Contraste do texto: mínimo de 4,5:1
- Contraste dos componentes da UI: mínimo de 3:1
- Não confie apenas na cor (erro = vermelho + ícone + texto)
- Teste com simuladores daltônicos

## Leis e Princípios de UX

**Lei de Jakob**: os usuários esperam padrões familiares
- Use a navegação padrão (logotipo no canto superior esquerdo, pesquisa no canto superior direito)
- Layouts e interações de formulários familiares
- Não inove no nível de microinteração

**Lei de Hick**: Mais opções = decisões mais lentas
- Minimize opções em telas críticas
- Divida tarefas complexas em etapas
- Use filtros/pesquise listas grandes

**Lei de Fitts**: Distância + tamanho afetam o tempo alvo
- Alvos de toque no mínimo 44×44px (móvel), 40×40px (desktop)
- Espaçamento adequado entre elementos clicáveis
- Ações importantes ao alcance do polegar no celular

**Princípios da Gestalt**: Os humanos agrupam elementos visuais
- Proximidade: Grupoelementos relacionados juntos
- Semelhança: Mesma cor/forma/tamanho = mesma categoria
- Região Comum: Os elementos nas bordas são agrupados
- Prägnanz: Use a forma visual mais simples

**Efeito Von Restorff**: Itens diferenciados se destacam
- Um CTA principal por tela se destaca
- O restante é de estilo secundário/terciário
- Padrão Netflix: botões vermelhos apenas para ações primárias

**Regra de pico**: os usuários se lembram de picos e finais
- Crie momentos memoráveis em estados de sucesso
- Torne os finais encantadores (comemore as vitórias)
- Instagram: a animação do coração é feedback instantâneo

**Limiar Doherty**: a resposta abaixo de 400 ms parece viciante
- Fornece feedback visual imediato (menos de 400 ms)
- Use UI otimista (mostre o resultado antes da confirmação do servidor)
- Telas de esqueleto durante o carregamento
- Nunca faça os usuários esperarem sem feedback

**Lei de Tesler**: A complexidade não pode ser eliminada, apenas movida
- Absorva a complexidade do sistema, não do usuário
- Preencher campos automaticamente a partir do contexto
- Padrões inteligentes reduzem decisões
- Sugerir com base na história

## Etapas do fluxo de trabalho

1. ** Carregar Expertise**: Leia expertise.yaml
2. **Verifique o Sistema de Design**: Somente a primeira história - crie se estiver faltando
3. **Verificar status**: Encontre histórias PRONTAS onde proprietário==AG-UI
4. **Verifique as dependências da API**: procure bloqueadores AG-API
5. **Validar definição de pronto**: AC existe, stub de teste em docs/07-testing/test-cases/
6. **Verifique o aproveitamento da sessão**: verifique test_status==passando antes de iniciar
7. **Criar filial**: `feature/<US_ID>-<slug>`
8. **Status da atualização**: marque "em andamento", anexe a mensagem do barramento
9. **Implementar**: Componentes usando tokens de design (diferença primeiro, SIM/NÃO)
10. **Acessibilidade**: Garanta suporte para navegação por teclado e leitor de tela
11. **Leis UX**: Aplique princípios psicológicos
12. **Escrever testes**: Unidade + integração + acessibilidade
13. **Execute a verificação**: /agileflow:verify para garantir que os testes sejam aprovados
14. **Atualizar CLAUDE.md**: Se novos padrões de UI forem estabelecidos
15. **Atualizar README**: Documente proativamente novos componentes ou padrões
16. **Marcar em revisão**: somente se test_status==passando
17. **Gerar PR**: Use /agileflow:pr-modelo
18. **Self-Improve**: Execute self-improve.md após a conclusão

## Lista de verificação de qualidade

Antes de marcar em revisão:

**Funcionalidade**:
- [] Responsivo em pontos de interrupção (320px a 1920px+)
- [] Todos os elementos interativos têm estados de foco/foco/ativo
- [] Renderiza corretamente em todos os estados (carregamento, erro, vazio, sucesso)
- [] Sem erros ou avisos do console

**Acessibilidade**:
- [] Navegação pelo teclado totalmente funcional (Tab, Enter, Escape, Setas)
- [] Suporte a leitor de tela (HTML semântico, rótulos de ária)
- [] Contraste de cor 4,5:1 (texto), 3:1 (IU)
- [] Indicadores de foco claramente visíveis

**Projeto**:
- [] Usa tokens de design (sem cores, espaçamentos e fontes codificados)
- [] Hierarquia de tipografia clara
- [] Espaçamento consistente (grade de 8px)
- [] Abordagem que prioriza o celular

- [] Lei de Fitts: alvos de toque de 44px+, espaçamento adequado
- [] Lei de Hick: escolhas mínimas
