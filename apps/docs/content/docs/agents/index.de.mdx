# Agenten

AgileFlow umfasst **30 spezialisierte Agenten**, jeder ein Experte auf seinem Gebiet. Agenten können vom `/babysit`-Mentor erzeugt oder direkt mit dem Task-Tool aufgerufen werden.

## Wie Agenten arbeiten

1. **Expertise-First**: Jeder Agent lädt seine Expertise-Datei, um Ihre Codebasismuster zu verstehen
2. **Kontextbewusst**: Agenten lesen Projektdokumente, status.json und aktuelle Änderungen
3. **Kontexterhaltung**: Erweiterte Agenten verwenden die Konfiguration „compact_context“, um wichtiges Wissen während der Konversationskomprimierung zu bewahren
4. **Kollaborativ**: Agenten kommunizieren über den Nachrichtenbus und aktualisieren den Story-Status
5. **Selbstverbesserung**: Agenten aktualisieren ihr Fachwissen nach Abschluss der Arbeit

## Kompakte Kontextfunktion

Um den Fokus bei langen Gesprächen aufrechtzuerhalten, verwenden AgileFlow-Agenten die Konfiguration **compact_context**, um wichtige Informationen beizubehalten, wenn der Kontext automatisch komprimiert wird. Dadurch wird sichergestellt, dass Agenten während des Gesprächs keine wichtigen Regeln, Zustände und Prioritäten verlieren.

### Wie es funktioniert

JederDer Agent hat eine `compact_context`-Konfiguration mit drei Komponenten:

**Prioritätsstufe**: Bestimmt, wie aggressiv der Kontext des Agenten erhalten bleibt
- `critical` - Immer beibehalten (Agenten mit der höchsten Priorität wie Mentor, API, UI)
- `high` - In den meisten Verdichtungen erhalten (wichtige Agenten)
– `medium` – Wird nach Möglichkeit beibehalten (Standardagenten)
– `low` – Zuerst entfernt (selten, für Beratungsagenten verwendet)

**Regeln bewahren**: Grundlegende Verhaltensregeln, die niemals verloren gehen dürfen
- Identität und Rolle des Agenten
- Kritische Einschränkungen und Sicherheitsregeln
- Verfahren, die unbedingt befolgt werden müssen (z. B. „Zuerst Fachwissen laden“)
- Anforderungen an die staatliche Verwaltung
- Koordinierungsprotokolle

**Zustandsfelder**: Dynamischer Zustand, der sich während der Arbeit ändert
- Aktuelle Story wird umgesetzt (US-#### ID)
- Aufgabenfortschritt und Abschlussprozentsatz
- Bekannte Blocker und Abhängigkeiten
- Nächste priorisierte Aktionen
- Test-/Build-Status

### Beispielkonfiguration

```yaml
compact_context:
  priority: critical
  preserve_rules:
    - "LOAD EXPERTISE FIRST: Read packages/cli/src/core/experts/api/expertise.yaml"
    - "VERIFY TEST BASELINE: Check test_status before starting"
    - "PRIORITIZE AG-UI BLOCKERS: Unblock UI stories waiting on API endpoints"
    - "DIFF-FIRST APPROACH: Show edits with confirmation before applying"
  state_fields:
    - current_story
    - endpoints_implemented
    - blocked_ui_stories
    - test_status_baseline
```

Wenn Claude Code con komprimiertText werden diese Regeln und Statusfelder für alle Agenten beibehalten und so die Kontinuität auch nach mehreren Konversationskomprimierungsrunden sichergestellt.

### Agentenprioritätsverteilung

- **Kritisch** (2 Agenten): Mentor, API
- **Hoch** (9 Agenten): UI, Datenbank, CI, Entwickler, Tests, Sicherheit, Dokumentation, Leistung, Codebasisabfrage
- **Mittel** (15 Agenten): Datenmigration, Qualitätssicherung, Zugänglichkeit, Überwachung, Forschung, Werbetexter, Analyse, Integrationen, Mobil, Design, Compliance, Refactor, Produkt, Readme-Updater
- **Niedrig** (2 Agenten): epischer Planer, Orchestrator

Dieser abgestufte Ansatz stellt sicher, dass die kritischsten Agenten immer den Fokus behalten, während andere den Kontext bei Bedarf sanft herabsetzen.

## Spawning Agents

### Über /babysit (empfohlen)

Der Mentor `/babysit` erkennt automatisch, welcher Agent benötigt wird:

```bash
/agileflow:babysit
# Then describe your task - babysit will spawn the right agent
```

### Direct Spawning

Sie können auch spawn-Agenten direkt mit dem Task-Tool:

```text
Task(
  description: "Design user sessions schema",
  prompt: "Create a sessions table to track user logins with proper indexes",
  subagent_type: "agileflow-database"
)
```

### Parallele Agenten

Für Full-Stack-Funktionen erzeugen Sie mehrere Agenten gleichzeitig:

```text
# API work
Task(subagent_type: "agileflow-api", run_in_background: true, ...)

# UI work (runs in parallel)
Task(subagent_type: "agileflow-ui", run_in_background: true, ...)
```

## Agentenkommunikation

Agenten koordinieren über `docs/09-agents/`:

<FileTree>
  <Folder name="docs">
    <Folder name="09-agents">
      <File name="status.json" comment="Story statuses and assignments" />
      <Folder name="bus">
        <File name="log.jsonl" comment="Agent messages and handoffs" />
      </Folder>
    </Folder>
  </Folder>
</FileTree>

### Statusaktualisierungen

Agenten aktualisieren `status.json`, während sie arbeiten:

```json
{
  "stories": {
    "US-0042": {
      "status": "in-progress",
      "owner": "AG-DATABASE",
      "summary": "Created sessions table with indexes"
    }
  }
}
```

### Nachrichtenbus

Agenten protokollieren Koordinationsnachrichten:

```json
{"timestamp": "2025-12-24T10:00:00Z", "type": "assign", "story": "US-0042", "owner": "AG-DATABASE"}
{"timestamp": "2025-12-24T11:00:00Z", "type": "handoff", "story": "US-0042", "from": "AG-DATABASE", "to": "AG-API"}
```

## Best Practices

1. **Lassen Sie den Babysitter wählen**: Der Mentor weiß, welcher Agent am besten passt
2. **Kontext bereitstellen**: Informieren Sie Agenten über vorhandene Muster in Ihrer Codebasis
3. **Updates überprüfen**: Überprüfen Sie die von den Agenten vorgenommenen status.json-Änderungen
