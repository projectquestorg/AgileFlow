# Dokumentationsagent

Der Documentation Agent (AG-DOCUMENTATION) ist Ihr Spezialist für technisches Schreiben, API-Dokumentation, Benutzerhandbücher und die Aktualisierung der Dokumentation mit Ihrer Codebasis. Dieser Agent stellt sicher, dass die gesamte Dokumentation sowohl für neue als auch für erfahrene Entwickler aktuell, klar und hilfreich bleibt.

## Fähigkeiten

- **API Documentation**: OpenAPI/Swagger-Spezifikationen, Endpunktbeschreibungen, Anfrage-/Antwortbeispiele und Fehlercodes
- **User Guides & Tutorials**: Erste Schritte, Schritt-für-Schritt-Anleitungen, Best Practices und Fehlerbehebung
- **Developer Guides**: Projekteinrichtung, Entwicklungsworkflow, Architekturübersicht und Erweiterungsmuster
- **README Management**: Root-, modulspezifische und funktionsspezifische README-Dateien
- **Changelog Maintenance**: Versionshinweise und benutzerseitige Änderungsverfolgung
- **Documentation Audits**: Identifizieren Sie veraltete Inhalte, defekte Links und fehlende Beispiele
- **Auto-Generated Docs**: Unterstützung für OpenAPI Generator, TypeDoc/JSDoc und Architekturdiagramme

## Wann zu verwenden

Verwenden Sie diesen Agenten, wenn Sie Folgendes benötigen:

- Schreiben oder aktualisieren Sie die API-Dokumentation mit Arbeitsbeispielen
- Erstellen Sie Benutzerhandbücher für neue Funktionen (Schritt-für-Schritt-Anleitungen)
- Aktualisieren Sie die README-Dateien, um den aktuellen Projektstatus widerzuspiegeln
- Schreiben Sie Onboarding-Leitfäden für Entwickler
- Erstellen Sie Anleitungen und FAQs zur Fehlerbehebung für häufige Probleme
- Dokumentieren Sie Architekturentscheidungen und Systemdesign
- Pflegen Sie das Änderungsprotokoll mit Versionshinweisen
- Beheben Sie defekte Dokumentationslinks
- Prüfen Sie die Dokumentation auf Aktualität und Genauigkeit
- Halten Sie die Dokumentation mit Codeänderungen synchron

## Wie es funktioniert

1. **Context Loading**: Agent liest Fachwissensdatei (Dokumentationsmuster, Dateispeicherorte, Konventionen) und analysiert die Projektstruktur (CLAUDE.md, status.json, aktuelle Versionen)

2. **Documentation Audit**: Der Agent identifiziert Dokumentationslücken, veraltete Abschnitte, defekte Links und fehlende Beispiele, indem er Dokumente/Strukturen scannt und mit dem Code vergleicht

3. **Planning**: Der Agent schlägt Verbesserungen der Dokumentation vor, schlägt neue Leitfäden vor, identifiziert veraltete Inhalte und priorisiert die Arbeit

4. **Writing**: Der Agent erstellt/aktualisiert die Dokumentation so, dass Klarheit Vorrang vor Kürze hat, enthält Arbeitsbeispiele (Kopieren und Einfügen möglich) und behält eine konsistente Formatierung bei

5. **Quality Review**: Der Agent überprüft, ob die Dokumentation mit dem Code auf dem neuesten Stand ist, alle Schritte erklärt werden, Beispiele funktionieren und die Navigation zwischen den Dokumenten klar ist

6. **Coordination**: Der Agent aktualisiert status.json, hängt Abschlussmeldungen an und sendet Aktualisierungen an den Agentenbus

## Beispiel

```bash
# Via /babysit
/agileflow:babysit
> "We just released v2.17.0. Need to update the changelog and API docs for the new endpoints"
```

Der Dokumentationsagent würde:
1. Überprüfen Sie die neuen API-Endpunkte, die in Version 2.17.0 hinzugefügt wurden
2. Aktualisieren Sie die API-Dokumentation mit Anfrage-/Antwortbeispielen
3. Fügen Sie Versionshinweise zum Änderungsprotokoll hinzu, um benutzerbezogene Änderungen hervorzuheben
4. Aktualisieren Sie die README-Datei bei Bedarf
5. Überprüfen Sie, ob alle Links funktionieren

## Schlüsselverhalten

- **Documentation Priority**: Klarheit statt Kürze – erklärt alles für neue Benutzer
- **Always Include Examples**: Dokumentation ohne Beispiele ist nutzlos; Alle Beispiele können kopiert und eingefügt werden
- **Keep Current**: Die Dokumentation muss mit dem Code aktuell bleiben; Veraltete Dokumente sind schlimmer als keine Dokumente
- **Include Troubleshooting**: Beheben Sie häufige Probleme, auf die Benutzer stoßen
- **Document Breaking Changes**: Kritisch für Benutzer, die Versionen aktualisieren
- **Link Checking**: Behebt proaktiv defekte Dokumentationslinks
- **Consistency**: Behält einheitliche Formatierung, Ton und Struktur in allen Dokumenten bei
- **Context Preservation**: Verwendet „compact_context“ (Priorität: hoch), um den Fokus bei langen Gesprächen beizubehalten und die Verfolgung der Dokumentationsabdeckung und Linkvalidierung durch Kontextkomprimierung beizubehalten

## Kompakte Kontextkonfiguration

Der Dokumentationsagent verwendet **high priority** kompakter_Kontext, um sicherzustellen, dass die Dokumentationsqualität im Fokus bleibt:

```yaml
compact_context:
  priority: high
  preserve_rules:
    - "LOAD EXPERTISE FIRST: Always read packages/cli/src/core/experts/documentation/expertise.yaml"
    - "NEVER LET DOCS LAG BEHIND CODE: Stale docs are worse than no docs (misinformation)"
    - "ALWAYS INCLUDE EXAMPLES: Documentation without examples is useless"
    - "KEEP LINKS CURRENT: Run link checker, fix broken references"
    - "DOCUMENT BREAKING CHANGES: Critical for users upgrading versions"
    - "CLARITY OVER BREVITY: Explain everything for new users"
  state_fields:
    - current_story
    - documentation_coverage
    - broken_links_count
    - outdated_sections
```

Dadurch wird sichergestellt, dass dokumentationskritische Regeln (Aktualität der Dokumente, einschließlich Beispiele, Vollständigkeit der Dokumentation) und der aktuelle Status (welche Dokumente veraltet sind, welche Links defekt sind) durch die Kontextkomprimierung erhalten bleiben.

## Verfügbare Werkzeuge

Dieser Agent hat Zugriff auf:
- **Read**: Dokumentationsdateien durchsuchen und Struktur verstehen
- **Write**: Create new documentation files
- **Edit**: Vorhandene Dokumentation aktualisieren
- **Bash**: Link-Prüfer ausführen, Dokumentationsstruktur validieren
- **Glob**: Dokumentationsdateien nach Muster suchen
- **Grep**: Dokumentationsinhalte durchsuchen und defekte Links identifizieren

## Dokumentationstypen

### API-Dokumentation
- OpenAPI/Swagger-Spezifikationen
- Endpunktbeschreibungen mit Pfaden, Methoden, Parametern
- Beispiele für Anfragen/Antworten
- Fehlercodes und ihre Bedeutung
- Authentifizierungs- und Autorisierungsdetails
- Informationen zur Ratenbegrenzung
- Codebeispiele in mehreren Sprachen

### Benutzerhandbücher
- Anleitungen für den Einstieg
- Häufige Aufgaben mit Schritt-für-Schritt-Anleitungen
- Best Practices und Tipps & Tricks
- Abschnitte zur Fehlerbehebung
- FAQ-Seiten
- Links zu Video-Tutorials

### Entwicklerhandbücher
- Projekteinrichtung und Installation
- Entwicklungsworkflow
- Architekturübersicht
- Organisation und Struktur des Codes
- So erweitern/anpassen
- Test- und Debugging-Ansätze
- Gemeinsame Muster und Anti-Muster

### README-Dateien
- Projektübersicht und Zweck
- Installations-/Einrichtungsanweisungen
- Schnellstart mit Arbeitsbeispielen
- Funktionsliste
- Dokumentationslinks
- Mitwirkende Richtlinien
- Lizenz und Namensnennung

## Qualitätscheckliste

Vor der Dokumentation:
- Die Dokumentation ist mit dem Code auf dem neuesten Stand
- Alle neuen Funktionen mit Beispielen dokumentiert
- Die API-Dokumentation enthält Arbeitsbeispiele
- Links sind nicht defekt
- Die Formatierung ist in allen Dateien konsistent
- Beispiele sind funktionsfähig und zum Kopieren und Einfügen bereit
- Im Abschnitt „Fehlerbehebung“ werden häufig auftretende Probleme behandelt
- Die Navigation zwischen den Dokumenten ist klar
- README ist genau und hilfreich
- Es bleiben keine veralteten Informationen übrig

## Gemeinsame Muster

**Documentation-as-Code**
- Die gesamte Dokumentation wurde mit Code in Git versioniert
- Die Quelle der Wahrheit ist die lebendige Codebasis, nicht separate Dokumente
- Änderungen am Code lösen Dokumentationsaktualisierungen aus

**Numbered Directory Structure**
- Dokumentation organisiert in Dokumenten/mit nummerierten Verzeichnissen (00-Meta bis 10-Recherche)
- Konsistente Struktur über alle Projekte hinweg

**Documentation Maintenance**
- Planen Sie vierteljährliche Audits, um veraltete Informationen zu finden
- Aktualisieren Sie nach jeder Veröffentlichung die Dokumentation
- Aktualisieren Sie nach jeder API-Änderung die API-Dokumente
- Beheben Sie defekte Links proaktiv

## Koordinationsnachrichten

Der Dokumentationsagent kommuniziert mit anderen Agenten über Busnachrichten:

```jsonl
{"ts":"2025-10-21T10:00:00Z","from":"AG-DOCUMENTATION","type":"status","text":"API docs updated for v2.17.0"}
{"ts":"2025-10-21T10:05:00Z","from":"AG-DOCUMENTATION","type":"question","text":"UI component API changed - need updated documentation for props"}
```

Wenn andere Agenten ihre Arbeit abschließen:
- AG-API completes endpoint → Documentation Agent updates API docs
- AG-UI gibt Komponenten frei → Der Dokumentationsagent dokumentiert Komponenten-Requisiten und -Nutzung
- Die Veröffentlichung erfolgt → Der Dokumentationsagent aktualisiert das Änderungsprotokoll und die Versionshinweise
- Architekturentscheidung getroffen → Dokumentationsagent aktualisiert ADR-Dokumentation
