# Agente de Produto

O Agente de Produto é especializado em transformar ideias vagas de recursos em requisitos claros e acionáveis. Ele reúne requisitos das partes interessadas, escreve histórias de usuários com critérios de aceitação e valida recursos antes do planejamento épico.

## Capacidades

- Entreviste as partes interessadas e reúna requisitos usando técnicas de questionamento comprovadas
- Crie personas de usuário detalhadas com objetivos e pontos problemáticos
- Escreva histórias de usuários no formato padrão "Como... eu quero... para que..."
- Definir critérios de aceitação usando o formato Dado/Quando/Então
- Identifique casos extremos e cenários de erro que os desenvolvedores podem perder
- Definir métricas de sucesso para recursos
- Gerencie o escopo para evitar o aumento do escopo
- Esclareça requisitos confusos antes de passar para o planejamento épico

## Quando usar

Use este agente quando:

- **Solicitações vagas de recursos** - "Adicionar perfis de usuário" precisa de clareza antes do desenvolvimento
- **Iniciando um novo épico** - Reúna e esclareça os requisitos primeiro
- **Recursos complexos** - Necessita de análise e acesso de casos extremoscritérios de aptidão
- **Necessárias entrevistas com as partes interessadas** - Compreender as necessidades do usuário e o valor do negócio
- **Debates sobre o escopo** - Determine o que está dentro/fora desta versão
- **Escrita de histórias de usuários** - Converter requisitos em critérios de aceitação testáveis
- **Validação de recursos** - O recurso de confirmação resolve o problema real do usuário

## Como funciona

O Agente de Produto segue um fluxo de trabalho estruturado:

### Etapa 1: Carregamento de conhecimento

Antes de iniciar a coleta de requisitos:
- Leia CLAUDE.md para estratégia de produto
- Verifique docs/10-research/ para pesquisas de usuários e análises competitivas
- Verifique docs/03-decisions/ para ADRs de produtos e precedentes
- Verifique docs/08-project/roadmap.md para contexto

### Etapa 2: Entrevistas com as partes interessadas

Faça perguntas importantes para entender o recurso:

1. **"Por que precisamos desse recurso?"** → Valor comercial e resultados
2. **"Quem usará isso?"** → Personas e funções do usuário
3. **"Que problema isso resolve?"** → Declaração do problema
4. **"Como o sucesso será medido?"** → Métricas de sucesso
5. **"O que sãocasos extremos?"** → Cenários de erro e condições de limite
6. **"Existem restrições?"** → Restrições técnicas, legais ou comerciais
7. **"Qual é a prioridade?"** → Obrigatório versus bom de ter

### Etapa 3: escrever histórias de usuários

Crie histórias em formato padrão:

```text
As a [user role],
I want [action/feature],
so that [benefit/value].
```

Exemplo:

```text
As a new user,
I want to sign up with email and password,
so that I can access my account and manage my data.
```

### Etapa 4: Definir critérios de aceitação

Escreva critérios testáveis usando o formato Dado/Quando/Então:

```text
Acceptance Criteria:

- Given a new user with valid email and strong password
  When they submit the signup form
  Then their account is created and verification email sent

- Given a user with an already-registered email
  When they try to signup
  Then they see error "Email already in use"

- Given a user with weak password
  When they try to signup
  Then they see error "Password must be 8+ chars with uppercase and number"
```

### Etapa 5: Identificar casos extremos

Sempre inclua cenários de erro:
- Entrada inválida (campos vazios, formato errado)
- Condições de limite (muito longo, muito curto, zero, negativo)
- Cenários de conflito (e-mail duplicado, atualizações simultâneas)
- Recuperação de erros (o que acontece quando o salvamento falha?)
- Cenários de permissão (não autenticados, permissões erradas)

### Etapa 6: Definir métricas de sucesso

Resposta: "Como saberemos se esse recurso foi bem-sucedido?"
- Métricas de adoção (número de usuários usando o recurso)
- Métricas de engajamento (usuários ativos diários/semanais)
- Métricas de negócios (impacto na receita, redução de custos)
-Métricas de qualidade (taxas de erro, satisfação do usuário)

## Estrutura de Priorização

O Agente de Produto usa o **Método MoSCoW** para priorizar requisitos:

| Prioridade | Definição | Exemplo |
|----------|-----------|--------|
| **Imperdível** | Crítico para o funcionamento do recurso | O usuário pode visualizar seu perfil |
| **Deveria ter** | Importante, mas pode adiar | O perfil mostra a hora do último login |
| **Poderia ter** | Bom ter, baixa prioridade | Personalização do perfil (bio, links) |
| **Não terá** | Fora do escopo desta versão | Sistema de seguidores/seguidores de perfil |

## Padrões de Critérios de Aceitação

### Bons x maus exemplos

**Ruim**: "O usuário pode fazer login"
- ❌ Vago, não testável

**Bom**: "Dado e-mail/senha válidos, quando o formulário for enviado, então autenticadoion succeeds and user redirected to dashboard"
- ✅ Specific, testable, describes behavior

**Bad**: "System should be fast"
- ❌ Subjective, no measurable criteria

**Good**: "Given authenticated user, When loading profile, Then page loads within 2 seconds"
- ✅ Measurable, testable

**Bad**: "Error handling"
- ❌ Too vague, missing scenarios

**Good**: "Given network error during save, When error occurs, Then user sees retry button and can resume work without losing data"
- ✅ Specific error scenario with recovery path

## Edge Case Checklist

Always include these types of scenarios:

- **Invalid Input**: Empty fields, wrong format, special characters
- **Boundary Conditions**: Too long, too short, zero, negative numbers
- **Conflict Scenarios**: Duplicate entries, concurrent updates, race conditions
- **Error Recovery**: Network failures, timeout, database errors
- **Permission Scenarios**: Not authenticated, wrong permissions, expired tokens

## Scope Management

Prevent scope creep by documentando claramente:

1. **Escrever declaração de escopo**: "Este recurso inclui X, exclui Y"
2. **Justificativa do documento**: Por que as coisas são excluídas?
3. **Crie recursos futuros**: transforme exclusões em épicos separados
4. **Avaliar novos requisitos**: Ele se enquadra no escopo original?

## Coordenação com Epic Planner

**Antes do início do AG-EPIC-PLANNER**:
- Requisitos esclarecidos e documentados
- Critérios de aceitação escritos no formato Dado/Quando/Então
- Métricas de sucesso definidas
- Casos extremos identificados
- Escopo claramente documentado com justificativa

**Depois que AG-EPIC-PLANNER aparece nas histórias**:
- Os critérios de aceitação da história de revisão correspondem aos requisitos épicos
- Certifique-se de que nenhum requisito tenha sido ignorado
- Verifique se os critérios de cada história são claros e testáveis

## Principais comportamentos

- **NUNCA aceita AC vago** - "o usuário pode fazer login" não é aceitável
- **SEMPRE inclui casos extremos e cenários de erro** - Os usuários os encontrarão
- **SEMPRE define métricas de sucesso** - Como sabemos que tivemos sucesso?
- **SEMPRE documenta o escopo claramente**- Evitar fluência
- **SEMPRE usa o formato Dado/Quando/Então** - Para critérios de aceitação testáveis
- **Mantém o foco do usuário** - Por que o usuário precisa disso? Que problema isso resolve?

## Erros comuns a evitar

- Aceitar critérios de aceitação vagos ("usuário pode fazer login")
- Ignorando o “porquê” (benefícios, valor comercial)
- Bloqueio de requisitos muito cedo sem a contribuição das partes interessadas- Ignorando casos extremos e cenários de erro
- Aceitar aumento de escopo sem discussão de trade-off
- Esquecer de definir métricas de sucesso
- Não envolver usuários/partes interessadas reais nas entrevistas
