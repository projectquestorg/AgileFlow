# Agente de Acessibilidade

O Agente de Acessibilidade (AG-ACCESSIBILITY) é um especialista em acessibilidade que garante que os aplicativos sejam utilizáveis por todos, incluindo pessoas com deficiências. Este agente realiza auditorias WCAG, testa com tecnologias assistivas e garante que os princípios de design inclusivo sejam seguidos.

## Capacidades

- **Auditoria WCAG 2.1**: Verificar conformidade nos níveis AA e AAA
- **Teste com Leitor de Tela**: Testar com NVDA, JAWS, VoiceOver
- **Navegação por Teclado**: Garantir que toda funcionalidade seja acessível via teclado
- **Verificação de Contraste de Cor**: Validar proporção de contraste ≥4.5:1
- **Gerenciamento de Foco**: Indicadores de foco visíveis e ordem de tabulação lógica
- **Testes de Acessibilidade**: Testes automatizados e manuais de a11y
- **Suporte a Tecnologia Assistiva**: Suporte para leitores de tela, controle de voz, etc.
- **Design Inclusivo**: Padrões de design que funcionam para todos os usuários
- **Implementação ARIA**: HTML semântico e labels ARIA apropriados
- **Documentação de Acessibilidade**: Criar guias de testes a11y e padrões

## Quando Usar

Use o Agente de Acessibilidade quando:

- Projetando novos recursos ou componentes
- Precisar de verificação de conformidade WCAG 2.1
- Problemas de contraste de cor ou preocupações de acessibilidade visual
- Navegação por teclado não funcionando
- Problemas de compatibilidade com leitor de tela
- Precisar testar com tecnologias assistivas
- Criando padrões de acessibilidade
- Planejando releases (teste de acessibilidade necessário)
- Usuários com deficiências relatarem problemas
- Precisar de ADR para decisões de acessibilidade

## Como Funciona

1. **Carregamento de Conhecimento**: Agente lê expertise e padrões WCAG
2. **Análise de Recurso**: Agente revisa story para implicações de acessibilidade
3. **Auditoria Automatizada**: Agente executa axe, Lighthouse, WAVE
4. **Teste Manual**: Agente testa navegação por teclado, leitores de tela
5. **Documentação de Problemas**: Agente lista todos os problemas de acessibilidade com severidade
6. **Planejamento de Remediação**: Agente coordena com equipes de design/UI
7. **Verificação**: Agente re-testa após correções
8. **Documentação**: Agente atualiza status.json e relatórios de descobertas

```bash
> "Construímos um novo componente de formulário. Ele é acessível?"

# Saída do agente:
# Auditoria de Acessibilidade: Novo Componente de Formulário
#
# Problemas Encontrados:
# CRÍTICO (bloqueia funcionalidade):
# ├── Campos de formulário sem labels (faltam elementos <label>)
# ├── Botão enviar sem nome acessível
# └── Mensagens de erro não anunciadas a leitores de tela
#

# ├── Contraste de cor: 3.8:1 (precisa 4.5:1)
# └── Indicadores de foco não visíveis
#
# MENOR (impacta alguns usuários):
# ├── Instruções do formulário poderiam ser mais claras
# └── aria-required faltando em campos obrigatórios
#
# Conformidade WCAG: 25% AA (precisa 100%)
#
# Passos de Remediação:
# 1. Adicionar <label> para cada campo de formulário
# 2. Adicionar nomes acessíveis aos botões (aria-label)

# Checklist de Acessibilidade:
# ✅ Testar com leitor de tela (NVDA)
# ✅ Navegação somente com teclado
# ✅ Verificação de ordem de tabulação
# ✅ Validação de contraste de cor
# ✅ Verificação de indicador de foco
```

## Comportamentos-Chave

- **Inclusivo por Design**: Projetar para todos os usuários desde o início
- **Testar com Usuários Reais**: Usar tecnologias assistivas reais
- **Teclado Primeiro**: Toda funcionalidade acessível sem mouse
- **Cor Não é Única**: Informação não transmitida apenas por cor
- **Foco Claro**: Indicadores de foco visíveis em todo elemento interativo
- **HTML Semântico**: Usar elementos HTML apropriados para significado
- **Sem Suposições**: Não assumir que todos os usuários têm teclados ou dispositivos apontadores

## Ferramentas Disponíveis

- Read, Write, Edit (operações de arquivo)
- Bash (executar testes a11y)
- Glob (encontrar arquivos de teste a11y)
- Grep (procurar por problemas de a11y)

## Padrões WCAG 2.1

**Níveis WCAG 2.1**:
- **Nível A**: Acessibilidade mínima
- **Nível AA**: Acessibilidade aprimorada (padrão recomendado)
- **Nível AAA**: Acessibilidade ideal (aim for this)

**Princípios WCAG 2.1**:
1. **Perceptível**: Informação e componentes UI devem ser perceptíveis
2. **Operável**: Componentes UI e navegação devem ser operáveis
3. **Compreensível**: Informação e operações devem ser compreensíveis
4. **Robusto**: Conteúdo deve ser robusto para tecnologias assistivas

**Critérios-Chave WCAG 2.1 AA**:
- **Contraste de Cor**: Texto ≥4.5:1, componentes UI ≥3:1
- **Acessível por Teclado**: Toda funcionalidade via teclado
- **Sem Armadilhas de Teclado**: Pode Tab sair de qualquer componente
- **Ordem de Foco**: Ordem de tabulação corresponde ordem visual (lógica)
- **Foco Visível**: Indicador de foco visível ≥2px outline
- **Tamanho de Alvo de Toque**: ≥44x44 CSS pixels
- **HTML Semântico**: Usar elementos HTML apropriados
- **Labels ARIA**: Botões/links têm nomes acessíveis
- **Labels de Formulário**: Todos campos devidamente rotulados

## Testes de Acessibilidade

**Teste Automatizado** (executar primeiro para feedback rápido):
- **axe DevTools**: Extensão de navegador, identifica problemas comuns
- **Lighthouse**: Auditoria integrada em Chrome DevTools
- **WAVE**: Ferramenta WebAIM para feedback visual
- **Verificadores de contraste de cor**: Verificar proporção 4.5:1

**Teste Manual** (essencial para avaliação real):
- **Navegação somente com teclado**: Tab através de toda página (sem mouse)
- **Teste com leitor de tela**: NVDA (Windows), JAWS (Windows), VoiceOver (macOS)
- **Verificação de ordem de tabulação**: Verificar fluxo lógico
- **Visibilidade de indicador de foco**: Ver foco em todo elemento interativo
- **Validação de contraste de cor**: Verificação com ferramenta + inspeção visual

**Suporte a Leitor de Tela**:
- NVDA (Windows, gratuito)
- JAWS (Windows, pago, mais popular)
- VoiceOver (macOS/iOS, integrado)
- TalkBack (Android, integrado)

## Problemas de Acessibilidade Comuns

| Problema | Problema | Solução |
|----------|----------|---------|
| Sem labels em inputs | Leitor de tela não sabe qual campo é | Adicionar `<label for="id">` |
| Botões somente ícone | Sem nome para leitores de tela | Adicionar `aria-label="name"` |
| Contraste de cor baixo | Difícil ler para usuários com baixa visão | Aumentar contraste para 4.5:1 |
| Indicadores de foco faltando | Usuários de teclado não conseguem ver onde estão | Adicionar outline de foco visível |
| Sem alt text em imagens | Leitores de tela não conseguem descrever imagem | Adicionar `alt="..."` descritivo |
| Armadilhas de teclado | Usuários não conseguem navegar para fora | Corrigir gerenciamento de foco |
| Informação somente cor | Usuários daltônicos perdem significado | Usar ícones, texto, padrões |

## Checklist de Acessibilidade

**Acessibilidade Visual**:
- [ ] Contraste de cor ≥4.5:1 para texto (WCAG AA)
- [ ] Contraste de cor ≥3:1 para componentes UI
- [ ] Sem informação somente cor (usar ícones/texto)
- [ ] Texto redimensionável para 200% sem scrolling
- [ ] Indicador de foco visível (≥2px outline)
- [ ] Alvos de toque ≥44x44 CSS pixels
- [ ] Design visual consistente (botões parecem botões)

**Acessibilidade de Teclado**:
- [ ] Toda funcionalidade acessível via teclado
- [ ] Sem armadilhas de teclado (Tab pode sair de qualquer componente)
- [ ] Ordem de tabulação lógica (corresponde ordem visual)
- [ ] Enter/Space ativa botões
- [ ] Setas navegam listas/menus
- [ ] Escape fecha modais/popovers

**Acessibilidade de Leitor de Tela**:
- [ ] HTML semântico (button, link, labels de formulário)
- [ ] Todos botões/links têm nome acessível
- [ ] Imagens têm alt text (ou hidden se decorativas)
- [ ] Campos de formulário têm labels (ou aria-label)
- [ ] Erros de formulário anunciados a leitores de tela
- [ ] Landmarks presentes (main, nav, complementary)
- [ ] Skip links para navegação por teclado
- [ ] Live regions para conteúdo dinâmico

## Padrões de Design Inclusivo

**Formulários**:
```html
<!-- Bom: Label associado com input -->
<label for="email">Email</label>
<input id="email" type="email" required aria-required="true">
<span class="error" role="alert">Email é obrigatório</span>

<!-- Ruim: Placeholder apenas, sem label -->
<input placeholder="Email" type="email">
```

**Botões**:
```html
<!-- Bom: Nome acessível claro -->
<button aria-label="Fechar menu">
  <svg aria-hidden="true"><!-- ícone --></svg>
</button>

<!-- Ruim: Somente ícone sem label -->
<button>
  <svg><!-- ícone --></svg>
</button>
```

**Imagens**:
```html
<!-- Bom: Alt text descritivo -->
<img src="profile.jpg" alt="Sarah Chen, gerente de produto">

<!-- Ruim: Alt text faltando -->
<img src="profile.jpg">

<!-- Bom: Imagem decorativa hidden -->
<img src="decoration.jpg" alt="" aria-hidden="true">
```

**Contraste de Cor**:
```text
Bom: Texto branco (#FFF) em azul (#0066CC) = 8.6:1 ✅ AAA
Ruim:  Texto branco (#FFF) em cinza (#999) = 3.9:1 ❌ AA (precisa 4.5:1)
Bom: Texto branco (#FFF) em preto (#000) = 21:1 ✅ AAA
```

## Prioridade de Remediação

| Prioridade | Severidade | Exemplos |
|-----------|-----------|----------|
| Crítica | Bloqueia funcionalidade central | Sem acesso via teclado a recursos, labels de formulário faltando |
| Importante | Impacta significativamente usabilidade | Contraste de cor baixo, indicadores de foco faltando |
| Menor | Impacta alguns usuários | Anotações ARIA faltando, mensagens de erro verbosas |

## Movimento & Animação

**Considerações de acessibilidade**:
- [ ] Animações respeitam `prefers-reduced-motion`
- [ ] Sem flashes indutoras de convulsão (sem >3 flashes/segundo)
- [ ] Autoplay desabilitado por padrão
- [ ] Usuários conseguem pausar animações

## Princípios de Acessibilidade

- **Inclusivo por design**: Considerar todos os usuários desde o início
- **Testar com usuários reais**: Usar tecnologias assistivas reais
- **Sem atalhos**: Conformidade WCAG é obrigatória, não opcional
- **Camadas de acessibilidade**: Visual, teclado, leitor de tela, cognitiva
- **Melhoria contínua**: Corrigir problemas conforme descobertos
- **Educação**: Ajudar equipe a entender por que acessibilidade importa
