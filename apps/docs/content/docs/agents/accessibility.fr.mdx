# Agent d'accessibilité

L'agent d'accessibilité (AG-ACCESSIBILITY) est un spécialiste de l'accessibilité qui assure que les applications sont utilisables par tous, y compris les personnes en situation de handicap. Cet agent mène des audits WCAG, teste avec les technologies d'assistance et assure que les principes de conception inclusive sont respectés.

## Capacités

- **Audit WCAG 2.1**: Vérifier la conformité des niveaux AA et AAA
- **Test des lecteurs d'écran**: Tester avec NVDA, JAWS, VoiceOver
- **Navigation au clavier**: Assurer que toute la fonctionnalité est accessible via le clavier
- **Vérification du contraste des couleurs**: Valider le rapport de contraste ≥4.5:1
- **Gestion du focus**: Indicateurs de focus visibles et ordre de tabulation logique
- **Tests d'accessibilité**: Tests automatisés et manuels d'accessibilité
- **Support des technologies d'assistance**: Support pour lecteurs d'écran, commande vocale, etc.
- **Conception inclusive**: Modèles de conception qui fonctionnent pour tous les utilisateurs
- **Implémentation ARIA**: HTML sémantique et étiquettes ARIA appropriées
- **Documentation d'accessibilité**: Créer des guides de test d'accessibilité et des normes

## Quand l'utiliser

Utiliser l'agent d'accessibilité quand:

- Concevoir de nouvelles fonctionnalités ou composants
- Besoin de vérification de conformité WCAG 2.1
- Problèmes de contraste des couleurs ou préoccupations concernant l'accessibilité visuelle
- La navigation au clavier ne fonctionne pas
- Problèmes de compatibilité avec le lecteur d'écran
- Besoin de tester avec les technologies d'assistance
- Créer des normes d'accessibilité
- Planifier les versions (test d'accessibilité requis)
- Les utilisateurs en situation de handicap signalent des problèmes
- Besoin d'ADR pour les décisions d'accessibilité

## Comment ça marche

1. **Chargement des connaissances**: L'agent lit l'expertise et les normes WCAG
2. **Analyse des fonctionnalités**: L'agent examine la story pour les implications d'accessibilité
3. **Audit automatisé**: L'agent exécute axe, Lighthouse, WAVE
4. **Test manuel**: L'agent teste la navigation au clavier, les lecteurs d'écran
5. **Documentation des problèmes**: L'agent liste tous les problèmes d'accessibilité avec gravité
6. **Planification de la correction**: L'agent coordonne avec les équipes de conception/UI
7. **Vérification**: L'agent teste à nouveau après les corrections
8. **Documentation**: L'agent met à jour status.json et rapporte les conclusions

## Exemple

```bash
# Via babysit - audit du nouveau composant de formulaire pour l'accessibilité
/agileflow:babysit
> "We built a new form component. Is it accessible?"

# Résultat de l'agent:
# Audit d'accessibilité: Nouveau composant de formulaire
#
# Problèmes trouvés:
# CRITIQUE (bloque la fonctionnalité):
# ├── Les champs du formulaire n'ont pas d'étiquettes (éléments <label> manquants)
# ├── Le bouton d'envoi n'a pas de nom accessible
# └── Les messages d'erreur ne sont pas annoncés aux lecteurs d'écran
#
# MAJEUR (impacte significativement l'utilisabilité):
# ├── Contraste des couleurs: 3.8:1 (besoin de 4.5:1)
# └── Les indicateurs de focus ne sont pas visibles
#
# MINEUR (impacte certains utilisateurs):
# ├── Les instructions du formulaire pourraient être plus claires
# └── aria-required manquant sur les champs obligatoires
#
# Conformité WCAG: 25% AA (besoin de 100%)
#
# Étapes de correction:
# 1. Ajouter <label> pour chaque champ du formulaire
# 2. Ajouter des noms accessibles aux boutons (aria-label)
# 3. Ajouter aria-live aux messages d'erreur
# 4. Augmenter le contraste des couleurs à 4.5:1
# 5. Ajouter des indicateurs de focus visibles (contour 2px)
#
# Liste de contrôle d'accessibilité:
# ✅ Tester avec le lecteur d'écran (NVDA)
# ✅ Navigation au clavier uniquement
# ✅ Vérification de l'ordre de tabulation
# ✅ Validation du contraste des couleurs
# ✅ Vérification de l'indicateur de focus
```

## Comportements clés

- **Inclusif par défaut**: Concevoir pour tous les utilisateurs dès le départ
- **Tester avec les utilisateurs réels**: Utiliser les technologies d'assistance réelles
- **Clavier d'abord**: Toute la fonctionnalité accessible sans souris
- **Pas juste la couleur**: L'information n'est pas transmise uniquement par la couleur
- **Focus clair**: Indicateurs de focus visibles sur chaque élément interactif
- **HTML sémantique**: Utiliser des éléments HTML appropriés pour le sens
- **Pas de suppositions**: Ne supposez pas que tous les utilisateurs ont des claviers ou des dispositifs de pointage

## Normes WCAG 2.1

**Niveaux WCAG 2.1**:
- **Niveau A**: Accessibilité minimale
- **Niveau AA**: Accessibilité améliorée (norme recommandée)
- **Niveau AAA**: Accessibilité optimale (viser ceci)

**Principes WCAG 2.1**:
1. **Perceptible**: Les informations et les composants UI doivent être perceptibles
2. **Opérable**: Les composants UI et la navigation doivent être opérables
3. **Compréhensible**: Les informations et les opérations doivent être compréhensibles
4. **Robuste**: Le contenu doit être robuste pour les technologies d'assistance

**Critères clés WCAG 2.1 AA**:
- **Contraste des couleurs**: Texte ≥4.5:1, composants UI ≥3:1
- **Accessible au clavier**: Toute la fonctionnalité via le clavier
- **Pas de pièges au clavier**: Peut Tabulation hors de tout composant
- **Ordre de focus**: L'ordre de tabulation correspond à l'ordre visuel (logique)
- **Focus visible**: Indicateur de focus visible ≥2px contour
- **Taille de la cible tactile**: ≥44x44 pixels CSS
- **HTML sémantique**: Utiliser les éléments HTML appropriés
- **Étiquettes ARIA**: Les boutons/liens ont des noms accessibles
- **Étiquettes de formulaire**: Tous les champs du formulaire sont correctement étiquetés

## Tests d'accessibilité

**Tests automatisés** (exécuter d'abord pour retours rapides):
- **axe DevTools**: Extension de navigateur, identifie les problèmes courants
- **Lighthouse**: Audit intégré dans Chrome DevTools
- **WAVE**: Outil WebAIM pour retours visuels
- **Contraste des couleurs**: Vérifier le rapport 4.5:1

**Tests manuels** (essentiels pour évaluation réelle):
- **Navigation au clavier uniquement**: Tabulation sur la page entière (pas de souris)
- **Test du lecteur d'écran**: NVDA (Windows), JAWS (Windows), VoiceOver (macOS)
- **Vérification de l'ordre de tabulation**: Vérifier le flux logique
- **Visibilité de l'indicateur de focus**: Voir le focus sur chaque élément interactif
- **Validation du contraste des couleurs**: Vérification des outils + inspection visuelle

**Support des lecteurs d'écran**:
- NVDA (Windows, gratuit)
- JAWS (Windows, payant, le plus populaire)
- VoiceOver (macOS/iOS, intégré)
- TalkBack (Android, intégré)

## Problèmes d'accessibilité courants

| Problème | Problématique | Correction |
|----------|-------------|-----------|
| Pas d'étiquettes sur les entrées | Le lecteur d'écran ne sait pas quel champ c'est | Ajouter `<label for="id">` |
| Boutons avec icône uniquement | Pas de nom pour les lecteurs d'écran | Ajouter `aria-label="name"` |
| Faible contraste des couleurs | Difficile à lire pour les utilisateurs malvoyants | Augmenter le contraste à 4.5:1 |
| Indicateurs de focus manquants | Les utilisateurs au clavier ne peuvent pas voir où ils sont | Ajouter un contour de focus visible |
| Pas de texte alt sur les images | Les lecteurs d'écran ne peuvent pas décrire l'image | Ajouter descriptif `alt="..."` |
| Pièges au clavier | Les utilisateurs ne peuvent pas naviguer | Corriger la gestion du focus |
| Information par couleur uniquement | Les utilisateurs daltoniens manquent le sens | Utiliser des icônes, du texte, des modèles |

## Liste de contrôle d'accessibilité

**Accessibilité visuelle**:
- [ ] Contraste des couleurs ≥4.5:1 pour le texte (WCAG AA)
- [ ] Contraste des couleurs ≥3:1 pour les composants UI
- [ ] Pas d'information par couleur uniquement (utiliser des icônes/texte)
- [ ] Texte redimensionnable à 200% sans défilement
- [ ] Indicateur de focus visible (≥2px contour)
- [ ] Cibles tactiles ≥44x44 pixels CSS
- [ ] Design visuel cohérent (les boutons ressemblent à des boutons)

**Accessibilité du clavier**:
- [ ] Toute la fonctionnalité accessible via le clavier
- [ ] Pas de pièges au clavier (Tabulation peut quitter tout composant)
- [ ] Ordre de tabulation logique (correspond à l'ordre visuel)
- [ ] Entrée/Espace active les boutons
- [ ] Les touches fléchées parcourent les listes/menus
- [ ] Échap ferme les modales/popovers

**Accessibilité du lecteur d'écran**:
- [ ] HTML sémantique (button, link, étiquettes de formulaire)
- [ ] Tous les boutons/liens ont un nom accessible
- [ ] Images ont un texte alt (ou caché si décoratif)
- [ ] Les champs du formulaire ont des étiquettes (ou aria-label)
- [ ] Les erreurs du formulaire sont annoncées au lecteur d'écran
- [ ] Points de repère présents (main, nav, complementary)
- [ ] Liens de saut pour la navigation au clavier
- [ ] Régions en direct pour le contenu dynamique

## Modèles de conception inclusive

**Formulaires**:
```html
<!-- Bon: Étiquette associée à l'entrée -->
<label for="email">Email</label>
<input id="email" type="email" required aria-required="true">
<span class="error" role="alert">Email is required</span>

<!-- Mauvais: Placeholder uniquement, pas d'étiquette -->
<input placeholder="Email" type="email">
```

**Boutons**:
```html
<!-- Bon: Nom accessible clair -->
<button aria-label="Close menu">
  <svg aria-hidden="true"><!-- icon --></svg>
</button>

<!-- Mauvais: Icône uniquement avec pas d'étiquette -->
<button>
  <svg><!-- icon --></svg>
</button>
```

**Images**:
```html
<!-- Bon: Texte alt descriptif -->
<img src="profile.jpg" alt="Sarah Chen, product manager">

<!-- Mauvais: Texte alt manquant -->
<img src="profile.jpg">

<!-- Bon: Image décorative cachée -->
<img src="decoration.jpg" alt="" aria-hidden="true">
```

**Contraste des couleurs**:
```text
Bon: Texte blanc (#FFF) sur bleu (#0066CC) = 8.6:1 ✅ AAA
Mauvais:  Texte blanc (#FFF) sur gris (#999) = 3.9:1 ❌ AA (besoin 4.5:1)
Bon: Texte blanc (#FFF) sur noir (#000) = 21:1 ✅ AAA
```

## Priorité de correction

| Priorité | Gravité | Exemples |
|----------|---------|----------|
| Critique | Bloque la fonctionnalité principale | Pas d'accès au clavier aux fonctionnalités, étiquettes de formulaire manquantes |
| Majeur | Impacte significativement l'utilisabilité | Faible contraste des couleurs, indicateurs de focus manquants |
| Mineur | Impacte certains utilisateurs | Annotations ARIA manquantes, messages d'erreur verbeux |

## Mouvement et animation

**Considérations d'accessibilité**:
- [ ] Les animations respectent `prefers-reduced-motion`
- [ ] Pas de clignotement induisant la saisie (pas >3 clignotements/seconde)
- [ ] Autoplay désactivé par défaut
- [ ] Les utilisateurs peuvent mettre en pause les animations

## Coordination

L'agent d'accessibilité coordonne avec:

- **AG-DESIGN**: Contraste visuel, indicateurs de focus, modèles inclusifs
- **AG-UI**: Implémentation ARIA, HTML sémantique, navigation au clavier
- **AG-TESTING**: Automatisation des tests d'accessibilité
- **AG-QA**: Accessibilité dans les critères de version
- **Équipe produit**: Valider les améliorations d'accessibilité avec les vrais utilisateurs

## Commandes slash

- `/agileflow:research:ask TOPIC=...` - Rechercher les meilleures pratiques d'accessibilité
- `/agileflow:ai-code-review` - Examiner le code pour les problèmes d'accessibilité
- `/agileflow:adr-new` - Documenter les décisions d'accessibilité
- `/agileflow:status STORY=... STATUS=...` - Mettre à jour le statut de la story

## Principes d'accessibilité

- **Inclusif par conception**: Considérer tous les utilisateurs dès le départ
- **Tester avec les vrais utilisateurs**: Utiliser les technologies d'assistance réelles
- **Pas de raccourcis**: La conformité WCAG est obligatoire, pas facultative
- **Couches d'accessibilité**: Visuelle, clavier, lecteur d'écran, cognitive
- **Amélioration continue**: Corriger les problèmes dès qu'ils sont découverts
- **Éducation**: Aider l'équipe à comprendre pourquoi l'accessibilité compte
